## 設計上の判断

### 1. Store パターン（集約状態管理）

**選択理由**: Single Source of Truth を確保し、状態変更を一元化

```javascript
Store = {
    habits: [],      // 習慣リスト
    records: {}      // { habitId: [YYYY-MM-DD, ...] }
}
```

**利点**:
- LocalStorage との同期が明確
- テスト容易性が高い
- 複数機能から同じデータにアクセス可能

### 2. 日付正規化（YYYY-MM-DD 形式）

**選択理由**: タイムゾーン問題を回避し、日付比較を単純化

```javascript
const dateStr = `${year}-${month}-${day}`;
// ISO 8601 ではなくローカルタイム基準
```

**利点**:
- 夜中のチェックインでもユーザーのローカルタイムで正確に「今日」と認識
- 文字列比較で日付判定が高速
- LocalStorage で簡潔に記録可能

### 3. 連続日数計算（後方走査）

**アルゴリズム**:
1. 本日から過去へ遡る
2. 毎日の記録有無をチェック
3. 記録がない日で終了 → 連続日数確定

**利点**:
- シンプルで理解しやすい
- エッジケース（2月末など）に強い
- 最大 366 日分の走査で計算完了

### 4. UI 層の分離

**責務分割**:
- **Store**: 状態管理と業務ロジック
- **UI**: DOM 操作と画面表示
- **Controller**: イベント処理と制御フロー

**利点**:
- 各層が独立してテスト可能
- 状態変更と画面更新が分離
- 保守性が向上

### 5. XSS 対策（escapeHtml）

```javascript
// ユーザー入力は textContent で安全に挿入
const div = document.createElement('div');
div.textContent = userInput;
return div.innerHTML;
```

**利点**:
- innerHTML の直接使用を避け、XSS を防止
- 特殊文字を正しく表示

### 環境・ブラウザ制限

- **プライベート/シークレットモード**: LocalStorage が機能しない（ブラウザ制限）
- **IE11 以下**: サポート対象外（ES6+ 使用）
- **ローカルストレージ 5MB 制限**: 習慣数が極端に多い場合のみ影響

### エッジケース対応済み

- ✓ 空文字・スペースのみ入力
- ✓ 習慣の重複登録
- ✓ 最大10個超過
- ✓ 同一日の複数チェックイン
- ✓ LocalStorage エラー（QuotaExceededError）
- ✓ 不正な JSON（parse エラー）
