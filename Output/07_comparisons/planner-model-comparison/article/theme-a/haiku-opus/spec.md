## 1. 概要

Claude Code のマルチエージェント機能を活用し、コードレビュープロセスを自動化する実践的アプローチを解説する記事。セキュリティ監査、命名規則チェック、パフォーマンス検査、設計原則審査など、複数の観点を担当するサブエージェント群を設計・運用する具体的なステップを提供する。読者はこの記事を通じて、自社プロジェクトへの適用可能性を判断できるようになることを目指す。

### セクション 5: 実装例（600～800文字）
**目的**: 実際のコード / 設定で動作をイメージさせる

**実装例 1: エージェント定義**
```
# CLAUDE.md の構造（簡略版）
- エージェント名: SecurityReviewAgent
  - 責務: セキュリティ脆弱性の検出
  - スキル: pattern-matching, credential-scanner
  - 出力形式: JSON レポート
  - 次エージェント: PerformanceAgent へ自動パス
```

**実装例 2: hooks の設定例**
```json
# settings.json の hooks セクション
{
  "hooks": [
    {
      "event": "on_pr_created",
      "agents": ["SecurityReviewAgent"],
      "parallel": false
    },
    {
      "event": "security_check_done",
      "agents": ["PerformanceAgent", "ConventionAgent"],
      "parallel": true
    }
  ]
}
```

**実装例 3: エージェント間通信の例**
```
SecurityAgent 出力形式:
{
  "status": "completed",
  "findings": [...],
  "next_context": {...}
}

PerformanceAgent の入力:
- 前エージェントの findings を context として受け取り
- 自身の検査を実施
- 統合レポート作成へ
```

**含めるもの**:
- 実際に動作する設定ファイルのスニペット（3～4 ブロック）
- エージェント定義のメタ情報（何を入力、何を出力するか）

## 5. 各セクションの執筆ポイント（詳細）

### 導入
- 冒頭: 「コードレビューは重要だが、工数がかかる」という一般的認識から開始
- 中盤: 「その原因は何か」を3～4項目で分析
- 後半: 「マルチエージェント化はその解決の一つである」と提示
- トーン: 共感的かつ希望的

### 問題背景
- 表形式で「現状の手法」と「提案するマルチエージェント方式」の比較
- 具体的な数値例があれば理想的（例: 「手動レビューは PR あたり平均 2 時間、自動化後は...」）
- 「完全自動化は目指さない」という釈明も重要

### ソリューション
- エージェント群のアーキテクチャ図をテキストで表現
- 各エージェントの責務を 1 行で簡潔に書く
- パラレル実行可能な観点の明示（これにより速度向上をビジュアル化）

### 実装方法
- ステップバイステップの手順を番号付きリストで
- 初心者向け説明と上級者向け応用のバランス
- 設定ファイルはコピペ可能なサンプルにする

### 実装例
- 最低2つのコード例は実装可能 / 動作確認済みレベルの品質
- コメント多めで各セクションの意図を明記
- 「このコードでできること / できないこと」を明記

### 運用のコツ
- 実運用フェーズで起こりうる問題を具体的に列挙
- 各問題に対する対処法を明示
-「チーム全体で運用する」という視点を忘れずに

### まとめ
- このアプローチの三大メリットを強調
- 今後の発展可能性（他の自動化との組み合わせなど）を示唆
- 読者へのコールトゥアクション（「まずは セキュリティチェック1つから」など）

## 8. エッジケース・注意点

### 実装時の落とし穴
1. **エージェント間のコンテキスト喪失**
   - 問題: 各エージェントが独立して実行され、結果が集約されない
   - 対応: 明示的な「コンテキストパス」の設計、上流エージェントの出力を下流エージェントへ入力

2. **実行時間の爆発**
   - 問題: エージェント数が増えると、総実行時間が増加
   - 対応: パラレル実行可能な観点を分離、不要なチェック項目の削除

3. **誤検知・見落とし**
   - 問題: 自動化エージェントは完全ではなく、人間判定が必須な箇所がある
   - 対応: エージェントの責務を明確に、「自動化の範囲」と「手動確認の範囲」を図示

### 記事執筆時の注意
- **架空機能を書かない**: Claude Code の実際の hooks 機構、スキル定義をベースに
- **一社/一プロジェクトの例に限定しない**: 一般化した参考例として提示
- **限界を明記**: 「このアプローチは PR 単位のコードレビューに向いている」など、適用範囲を示す
- **プライバシー**: 実在企業名やプロジェクト名は記載しない（一般化）

### 読者への事前期待値設定
- 記事冒頭に「この記事は設計と実装例を提供し、完全な自動化システムを構築することは目指さない」と明記
- 「チーム固有の課題に対して、このフレームワークをカスタマイズする責任は読者にある」と伝える
