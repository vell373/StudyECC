# Claude Code のサブエージェントを使ってコードレビューを自動化する方法

## 1. 導入

チームのコードレビュー、工数がかかりすぎていませんか？一般的に、PR 1 件あたりのレビュー時間は平均 30～60 分とされており、人的リソースの制約の中で品質を保つことは難しい課題です。

Claude Code のマルチエージェント機能を活用することで、この課題に実践的に対処できます。セキュリティリスク、パフォーマンス問題、スタイル違反といった「観点が決まった作業」を複数の専門エージェントに分担させることで、レビュー品質を一貫させながら工数を大幅に削減できるのです。

本記事では、コードレビュープロセスを体系的に自動化するアプローチを実践的に解説します。自分のチーム・プロジェクトで実装できる具体的な設定例とエージェント設計を示すことで、読了後 3 日以内に実装を開始できるレベルの情報をお届けします。

**この記事の対象者**：Git / CI-CD の基本知識がある 1～8 年目のソフトウェアエンジニア、AI ツールを初めて本格的に活用したいエンジニア

---

## 2. 既存手法の課題

### 手動コードレビューの具体的な限界

**時間コスト**：PR 1 件あたり平均 30～60 分のレビュー時間がかかり、チーム全体では相当な生産性低下につながります。

**レビュー観点のばらつき**：レビュアーの経験値や知識によって指摘内容が変わります。セキュリティリスクを見落とすエンジニア、スタイル違反に厳しいエンジニアなど、一貫性が保ちにくいのです。

**属人性の問題**：ベテランエンジニアが持つレビュー知見（アーキテクチャのチェックポイント、よくあるセキュリティ脆弱性など）が形式化されず、チーム全体に共有されません。新入社員のレビュー精度が低いのはこのためです。

**スケーラビリティの欠如**：チームメンバーが増えるとレビュー待ち時間が増加し、開発スピードが低下します。

### 既存の自動化ツールの限界

現在の静的解析ツールや CI チェック（ESLint、Prettier、SonarQube など）は、スタイルチェックや軽微なバグ検出には優れています。しかし、**論理構造の妥当性**、**セキュリティリスクの微妙な判断**、**アーキテクチャへの適合性**といった観点は、人間の判断が必要とされます。

つまり、「ルール化できた観点」は自動化済みですが、「判断が必要な観点」はいまだに人手に依存しているのが現状です。

## 3. マルチエージェント設計の基本概念

### マルチエージェント方式とは

Claude Code では、1 つの主エージェント（Orchestrator）と複数の専門サブエージェントを組み合わせる設計が可能です。各エージェント が明確な責務を持ち、役割を分担することで、複雑なレビュープロセスを体系的に実行できます。

たとえば、以下のように分業します：

- **Security Reviewer**：セキュリティリスク検出専門
- **Style Checker**：コード規約・命名規則のチェック専門
- **Performance Analyzer**：パフォーマンスボトルネック検出専門

### このアプローチのメリット

**再現性**：同じチェックリストで毎回同じ観点から検査できるため、レビュー品質が安定します。

**スケール性**：新しい観点を追加したい場合、新しいエージェント を追加するだけ。既存のエージェントには影響しません。

**継続性**：チーム内の知見を spec.md / hook.md という「型」として蓄積できるため、新メンバーのオンボーディング時の知識共有がスムーズです。

**優先度管理**：セキュリティは最優先で検査し、スタイルは低優先といった制御が可能です。

## 4. コードレビューエージェント群の設計

複数のエージェントを効果的に組織するには、各エージェントの責務を明確にし、連携フローを定めることが重要です。

### エージェント設計表

| エージェント名 | 責務 | 入力 | 出力 | 優先度 |
|---|---|---|---|---|
| **Security Reviewer** | セキュリティ脆弱性のスキャン | PR のコード内容 | セキュリティ指摘リスト | 1（高） |
| **Style Checker** | コード規約・フォーマット チェック | PR の変更コード | 規約違反リスト | 3（低） |
| **Performance Analyzer** | パフォーマンス問題検出 | アルゴリズム・ループ構造 | ボトルネック指摘 | 2（中） |
| **Architecture Checker** | 設計・アーキテクチャ適合性 | クラス構造・依存関係 | 設計上の指摘 | 2（中） |
| **Test Coverage Auditor** | テストカバレッジの十分性 | テストコード + 本体コード | テスト不足箇所 | 2（中） |

### エージェント間の連携フロー

```
PR 作成・更新
    ↓
[Security Reviewer] 実行 → セキュリティ指摘が多数の場合、他エージェント実行をスキップ
    ↓
[Style Checker] 実行 → 軽微な指摘を集約
    ↓
[Performance / Architecture Checker] 並列実行 → 中優先度の指摘を収集
    ↓
[Test Coverage Auditor] 実行 → テストの完全性確認
    ↓
[統合レポート生成] → PR コメント として投稿
```

このフローによって、セキュリティリスクは必ず検査される一方、スタイル違反がない場合は不要なチェックをスキップできます。

## 5. 実装のポイント

### spec.md の作成方法

各エージェントの「入力」「出力」を明確に定義することが鍵です。たとえば Security Reviewer の spec.md では、「SQLインジェクションリスク」「認証情報の硬コード」「外部入力の検証不足」といった検査項目を、数値化できる形で列挙します。

「きれいなコード」ではなく「SQL が安全か」のように具体化することで、エージェント の指摘がぶれなくなります。

### Hook の活用

PR がオープンされたとき、または新しいコミットが推された時点で、エージェント群を自動実行する Hook を設定します。重要なのは、初回実行（PR オープン時）では高優先度のエージェントのみ、更新時には全エージェントを実行といった **段階的なトリガー設定** です。

### コンテキスト共有

プロジェクトの CLAUDE.md に「レビュー基準の統一仕様」を記述し、全エージェント間で共有します。これにより「このプロジェクトではログ出力は禁止」といった共通ルールが、全エージェント に自動的に反映されます。

### 段階的導入アプローチ

- **Phase 1（1～2 週間）**：Security Checker + Style Checker のみ導入。運用の課題を把握
- **Phase 2（2～4 週間）**：Performance / Architecture Checker を追加。チーム内フィードバック収集
- **Phase 3（1 ヶ月以降）**：Test Coverage Auditor 追加。継続的改善

最初から全エージェント を有効にするのではなく、小さく始めることが成功の鍵です。

## 6. 実装例（コードサンプル）

### 例 1：CLAUDE.md でのマルチエージェント設定

プロジェクトの CLAUDE.md に、以下のようにエージェント群を定義します。各エージェント は model 属性で使用するモデルを指定できます。セキュリティは高精度の Claude Opus を、スタイルチェックは軽量な Claude Haiku を使用することで、コストと品質のバランスを取ります。

```yaml
# CLAUDE.md 内のエージェント定義例
---
agents:
  security_reviewer:
    description: "セキュリティ脆弱性のスキャン"
    model: "claude-opus"
    checklist:
      - SQLインジェクションリスク
      - 認証情報の硬コード
      - エスケープ不足
      - 外部入力の検証不足
      - 暗号化通信の使用確認
  
  style_checker:
    description: "コード規約・命名規則チェック"
    model: "claude-haiku"
    checklist:
      - 命名規則（camelCase / PascalCase 統一）
      - ファイル行数（300行超で警告）
      - 関数行数（50行超でリファクタリング提案）
      - コメント記述の「なぜ」の有無
      - インデント統一（スペース/タブ）

  performance_analyzer:
    description: "パフォーマンスボトルネック検出"
    model: "claude-opus"
    checklist:
      - ネストの深いループ構造
      - 不要なメモリ割り当て
      - 外部 API の同期呼び出し
```

### 例 2：Hook 定義（自動実行トリガー）

GitHub / GitLab の Webhook または Claude Code 組み込みの Hook 機構により、以下のような定義で自動レビューを実行します。

```json
{
  "hooks": {
    "on_pr_opened": {
      "name": "初期レビュー",
      "agents": [
        "security_reviewer",
        "style_checker"
      ],
      "execution": "sequential",
      "timeout_minutes": 10,
      "report_to": "pr_comment"
    },
    "on_pr_updated": {
      "name": "更新時レビュー",
      "agents": [
        "security_reviewer",
        "style_checker",
        "performance_analyzer",
        "architecture_checker"
      ],
      "execution": "parallel",
      "timeout_minutes": 15,
      "report_to": "pr_comment",
      "skip_if_no_code_changes": true
    }
  }
}
```

**説明**：`on_pr_opened` では最小限のエージェント（セキュリティ + スタイル）を同期実行することで、即座にフィードバックが得られます。`on_pr_updated` では全エージェント を並列実行し、タイムアウトは 15 分に設定。「コード変更がない場合（ドキュメント修正のみ）はスキップ」というフィルタにより、無駄な実行を防止します。

## 7. 運用のコツ

### 段階的な導入と継続的改善

Phase 1 で 1～2 週間運用すると、「誤検知が多い」「重要な指摘を見落とす」といった課題が見えてきます。そこで spec.md の検査基準を調整してから Phase 2 へ進みます。完璧を目指さず、「今のチームに合ったエージェント群」を作ることが重要です。

### チーム内での共有方法

毎月 1 回、「このエージェント の指摘が最も役立った」「誤検知が多い項目」をチーム内で共有し、spec.md をアップデートします。変更があれば CLAUDE.md の変更履歴に記載することで、チーム全体が「現在のルール」を常に把握できます。

### トラブル対応例

- **「誤検知が多い」**：Spec.md の検査基準を厳しすぎる部分を緩和、「除外パターン」を追加（例：テストコードのみで使用される SQLはスキップ）
- **「レビュー時間がかえって増加」**：エージェント数を 2～3 個に限定し、高優先度のもののみ導入
- **「エージェント応答がない」**：タイムアウト設定を緩く（15 分以上）し、リトライロジックを追加

## 8. まとめと今後の展望

複数のエージェントを組み合わせることで、スケーラブルで再現性あるコードレビューが実現できます。セキュリティ、スタイル、パフォーマンスといった異なる観点から、一貫性を保ったレビューが毎回実行されるのです。

**読者への次のアクション**：
まずは Phase 1（Security + Style チェッカー）から始めてみてください。1～2 週間で成果を感じられるはずです。成功事例や困ったことがあれば、プロジェクト内で共有し、継続的に改善していくことが重要です。

本記事では扱いませんでしたが、今後は **自動修正機能の追加**（Prettier や ESLint の自動適用）や **メトリクス収集**（「このエージェント がいなくなると品質がどう変わるか」を測定）も視野に入れられます。生成 AI による開発プロセス自動化は、急速に進化しています。
