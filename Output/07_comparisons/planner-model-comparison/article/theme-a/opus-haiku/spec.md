# 記事執筆タスク設計仕様書（spec.md）

## テーマ
「Claude Code のサブエージェントを使ってコードレビューを自動化する方法」

---

## 記事概要

本記事は、Claude Code のマルチエージェント機能を活用して、コードレビュープロセスを体系的に自動化する方法を実践的に解説する技術記事です。従来の手動コードレビューの課題（時間コスト、レビュー観点のばらつき、継続性の難しさ）に対して、複数の専門エージェント（スタイルチェック、セキュリティ検査、パフォーマンス分析など）を組み合わせることで、品質と効率を両立させるアプローチを提示します。記事を読み終えた読者が、自分のチーム・プロジェクトでマルチエージェントコードレビューを導入し、具体的に実装できる段階まで到達することを目指します。

## 各セクションの執筆ポイント

### 1. 導入（Introduction）
**目標字数**: 250～300字

**含めるべき要素**:
- オープニングで問題提起：「チームのコードレビュー、工数がかかりすぎていませんか？」
- 記事の価値提案：「Claudeの複数エージェント機能を使えば、主要な指摘を自動化できます」
- 読者がこれから学べることの明示（期待設定）
- この記事の対象者（上記ペルソナ）の確認

**注意点**:
- 架空の機能に触れない（Claude Code の実際の機能に基づく）
- 根拠のない断定を避ける（「完全自動化できます」でなく「大幅に効率化できます」）

---

### 2. 既存手法の課題（Pain Points）
**目標字数**: 300～400字

**含めるべき要素**:
- 手動コードレビューの具体的な課題
  - **時間コスト**: PR 1 件あたりのレビュー時間（例：平均 30～60 分）
  - **レビュー観点のばらつき**: レビュアーによって指摘内容が異なる
  - **属人性**: ベテランエンジニアのレビュー知見が形式化されていない
  - **スケーラビリティ**: メンバー数が増えるとレビュー待ち時間が増加
- 現在の自動化ツール（静的解析、CI チェック）の限界
  - 「スタイルチェックは自動化されているが、論理構造やセキュリティの観点は人手」

**注意点**:
- 業界統計やケーススタディを参照する場合、確信のない数字は「おそらく」「一般的に」と前置き
- 手動レビューそのものを否定しない（人間にしかできない観点も存在）

---

### 3. マルチエージェント設計の基本概念
**目標字数**: 400～500字

**含めるべき要素**:
- 「マルチエージェント」とは何か（Claude Code の機能説明）
  - 1 つの主エージェント + 複数の専門サブエージェント
  - 各エージェント は役割を明確に分担（責務の分離）
  - エージェント間のデータ・結果の受け渡し
- このアプローチのメリット
  - **再現性**: 同じチェックリストで毎回同じ観点から検査できる
  - **スケール性**: エージェントを追加すれば新しい観点を加えられる
  - **継続性**: チーム内の知見を型(spec.md / hook.md)として蓄積できる
- コードレビューに適用する理由
  - 「観点が決まった作業」（セキュリティ、パフォーマンス、スタイル等）向き

**注意点**:
- Claude Code が実際に持つ「サブエージェント」機能の説明に限定
- 実装の詳細は次セクション以降で説明する旨、予告する

---

### 4. コードレビューエージェント群の設計
**目標字数**: 350～450字

**含めるべき要素**:
- **エージェント設計図テーブル**（5～6種類のエージェント例）

| エージェント名 | 責務 | 入力 | 出力 | 優先度 |
|---|---|---|---|---|
| Style Checker | コード規約・フォーマット チェック | PR の変更コード | 規約違反リスト | 3（低） |
| Security Reviewer | セキュリティリスク検出 | PR のコード内容 | セキュリティ指摘 | 1（高） |
| Performance Analyzer | パフォーマンス問題の検出 | PR のアルゴリズム・ループ等 | ボトルネック指摘 | 2（中） |
| Architecture Checker | 設計・アーキテクチャ適合性 | クラス構造・依存関係 | 設計上の指摘 | 2（中） |
| Documentation Checker | コメント・ドキュメント充実度 | コード + コメント | ドキュメント改善提案 | 3（低） |
| Test Coverage Auditor | テストカバレッジの十分性 | テストコード + 本体コード | テスト不足箇所 | 2（中） |

- **エージェント間の連携フロー**（テキスト形式）
  ```
  PR 作成
    ↓
  [Style Checker] → 指摘が多数の場合、他エージェントは実行スキップ
    ↓
  [Security Reviewer] → 重大リスク検出時、即座に報告
    ↓
  [Performance / Architecture Checker] → 並列実行可能
    ↓
  [Documentation / Test Coverage] → 最後の確認
    ↓
  統合レポート生成 → PR コメント へ投稿
  ```

**注意点**:
- エージェント数は無闇に増やさない（5～6個が実運用向き）
- 優先度の設定の理由を説明（セキュリティ > パフォーマンス > スタイル）

### 5. 実装のポイント
**目標字数**: 300～400字

**含めるべき要素**:
- **spec.md の作成方法**
  - 各エージェントの「入力」「出力」を明確に定義
  - 検査項目の優先順位付け
  - 検査基準の数値化（「何が引っかかるのか」を具体的に）

- **Hook の活用**
  - PR オープン時 / コミット時のトリガー設定
  - エージェント群の自動実行スケジュール
  - エラー時の再試行ロジック

- **コンテキスト共有**
  - 各エージェント間で「プロジェクトのコーディング規約」等の共通情報を共有
  - CLAUDE.md で チーム共通の「レビュー基準」を記述

- **段階的導入**
  - Phase 1: セキュリティ・スタイル チェッカーのみ導入
  - Phase 2: パフォーマンス・設計チェッカーを追加
  - Phase 3: ドキュメント・テストカバレッジ検査まで拡張

**注意点**:
- Hook.md・spec.md の実装例は次セクション（コードサンプル）で示す
- 「完璧を目指さない」ことを強調（Phase 1 で十分な成果が得られることを指摘）

### 6. 実装例（コードサンプル）
**目標字数**: 400～600字

**含めるべき要素**:
この部分が最も重要です。実装可能な 2～3 個のコード例を含めます。

#### 例1：CLAUDE.md での マルチエージェント設定
```markdown
【コード例1の説明】
プロジェクトの CLAUDE.md に、コードレビュー用エージェント群を定義する例。
各エージェントの責務、入力形式、出力形式を記載。
```

**コードブロック内容の例**:
```yaml
# CLAUDE.md 内の設定例
---
agents:
  security_reviewer:
    description: "セキュリティ脆弱性のスキャン"
    model: "claude-opus"
    checklist:
      - SQLインジェクションリスク
      - 認証情報の硬コード
      - エスケープ不足
      - 外部入力の検証不足
  
  style_checker:
    description: "コード規約・命名規則チェック"
    model: "claude-haiku"
    checklist:
      - 命名規則（camelCase / PascalCase）
      - ファイル行数（300行超で警告）
      - 関数行数（50行超でリファクタリング提案）
      - コメント記述の「なぜ」の有無
```

**説明**:
- このように CLAUDE.md に「エージェント定義」を集約することで、チーム全体が同じ基準で動作
- モデル選択（Opus vs Haiku）について、セキュリティは高精度（Opus）、スタイルはスピード優先（Haiku）など、役割に応じた使い分け

#### 例2：Hook 定義（自動実行トリガー）
```markdown
【コード例2の説明】
PR がオープン・更新されたときに、自動的にコードレビュー エージェント群を実行する Hook の定義例。
```

**コードブロック内容の例**:
```json
{
  "hooks": {
    "on_pr_opened": {
      "agents": [
        "security_reviewer",
        "style_checker"
      ],
      "parallel": false,
      "report_to": "pr_comment",
      "timeout_minutes": 10
    },
    "on_pr_updated": {
      "agents": [
        "security_reviewer",
        "style_checker",
        "performance_analyzer"
      ],
      "parallel": true,
      "report_to": "pr_comment",
      "skip_if_no_code_changes": true
    }
  }
}
```

**説明**:
- `on_pr_opened` では最小限（セキュリティ + スタイル）を同期実行
- `on_pr_updated` では全エージェントを並列実行（タイムアウトは 10 分）
- 「コード変更がない場合はスキップ」で無駄な実行を防止

#### 例3（オプション）：エージェント間の連携 spec.md の抜粋
```markdown
【コード例3の説明（推奨）】
レビュー結果を統合し、優先度順に PR コメント へ整形・出力する集約エージェント。
```

**コードブロック内容の例**:
```markdown
# PR Review Aggregator - spec.md

## 責務
複数のエージェント結果を受け取り、1 つの統合レポートに変換して PR コメントへ投稿する。

## 入力形式
```json
{
  "pr_number": 123,
  "results": [
    { "agent": "security_reviewer", "issues": [...] },
    { "agent": "style_checker", "issues": [...] }
  ]
}
```

## 出力形式
PR へコメント投稿：
```
🔍 **Code Review Results**
### 🚨 Critical Issues (Security)
- [xxx]
### ⚠️ Warning (Style)
- [xxx]
```
```

**注意点**:
- コード例は「実行可能」であること（JSON/YAML 記法の正確性）
- 説明文で「なぜこの設定にしたか」を補足
- 実装例は 3 つ程度が最適（多すぎるとセクション肥大化）

### 7. 運用のコツ
**目標字数**: 250～350字

**含めるべき要素**:
- **導入フェーズの段階化**
  - Phase 1（1～2週間）: セキュリティ + スタイル チェッカーのみ運用、改善点の把握
  - Phase 2（2～4週間）: パフォーマンス・設計チェッカー追加、チーム内フィードバック収集
  - Phase 3（1ヶ月以降）: ドキュメント・テスト検査拡張、継続的改善

- **チーム内での共有方法**
  - 定期的なレビュー基準のアップデート（月 1 回など）
  - エージェント追加・修正時の共有（変更ログを CLAUDE.md に記載）
  - 月次ふりかえり：「どのエージェントが最も役立ったか」をチーム内で共有

- **トラブル対応と改善**
  - 「誤検知が多い」→ エージェントの Spec.md を調整、「除外パターン」を追加
  - 「レビュー時間がかえって増加」→ エージェント数を絞る、優先度を再設定
  - 「エージェント応答がない」→ タイムアウト設定、リトライロジックの調整

**注意点**:
- 「完璧を目指さない」という姿勢（エージェントが 100% 完璧なことはない）
- 継続的改善のサイクルを重視

## エッジケース・注意点

### 実装時に起こりやすい問題と対応

| 問題 | 原因 | 対応方法 |
|------|------|---------|
| 「エージェント数が増えすぎて、レビュー時間が増加した」 | 欲張りすぎて Phase 3 まで一気に導入 | Phase 1（セキュリティ + スタイル）に限定、段階的に拡張することを記事に強調 |
| 「誤検知が多く、レビュアーの負担が増える」 | Spec.md の検査基準が厳しすぎる、または曖昧 | 記事で「検査基準は段階的に調整する」ことを明記、チューニング方法を記載 |
| 「エージェント間の連携が失敗する」 | 各エージェントの出力形式が異なっている | 記事で「統合エージェント」の存在と、出力形式の統一の重要性を説明 |
| 「チームメンバーがエージェント定義を理解できない」 | CLAUDE.md / Spec.md が複雑すぎる | 記事で「ドキュメント例」を示し、可読性を重視することを強調 |

### 記事では明記すべき留意点

- **100% 自動化はできない**: 最終判断は人間が行うこと
- **最初は小さく**: 「完璧なシステム」を目指さず、Phase 1 で十分な成果を感じることが重要
- **継続的改善**: エージェントは「セット＆フォーゲット」でなく、定期的にチューニングが必要
- **チームのルール化**: 個人差なく「レビュー基準」を適用することが効果の鍵

## Generator への引き渡し指示

   - 各コード例に「何をしているか」「なぜそうするのか」の説明を付ける
   - 実装例だけでなく「段階的な導入例」も示す（Phase 1 → Phase 2）

3. **視覚的な構造化**
   - テーブル・リスト・コードブロックのバランスを取る
   - 長い段落は避け、読みやすさを重視
   - 見出しの階層構造を適切に（H2 / H3 / H4）

4. **論理展開**
   - 「問題 → 解決策 → 実装方法 → 実装例 → 継続方法」という流れを厳守
   - セクション間の接続を明確に（「ここまでで○○について説明しました。次に～の実装方法を見ていきます」）

5. **根拠のある記述**
   - 架空の機能は含めない（Claude Code の実際の機能に限定）
   - 数字や統計が不確かな場合は「おそらく」「一般的に」と前置き

Generator が記事作成時に参考にすべきもの（言及する必要はないが、設計の背景として）：

## まとめ

**spec.md の使命**:
この設計仕様書は、Generator エージェントが「何を、どのように、何文字で、どの構成で」記事を執筆するかを完全に指示します。Generator はこの spec.md に従い、2000～3000字の実践的で説得力のある技術記事を生成することを目標とします。

完成後の記事は、読者が「自分たちのチームでもマルチエージェントコードレビューを実装してみたい」と思える、具体的で実行可能な内容となるはずです。
