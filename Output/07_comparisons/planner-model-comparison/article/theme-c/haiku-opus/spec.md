#### セクション 2: 従来のマルチエージェントパターン（500～700行）

**目標**: 3 つのパターンを「同じ粒度」で比較  
**カバーするパターン**:
1. **Pipeline パターン** — エージェント A → B → C と直列接続
2. **Collaborative パターン** — 複数エージェントが並列に協調
3. **Tree パターン** — 階層構造で分岐・マージ

**構成**:
- **2.1 Pipeline パターン**（150～200行）
  - 定義：「エージェント間の出力が次のエージェントの入力になる直列フロー」
  - メリット：実装が単純、実行順序が明確、デバッグが容易
  - デメリット：並列化不可、1 つのボトルネックが全体を遅延、柔軟性に欠ける
  - ユースケース：データクリーニング → 分析 → レポート生成
  - テキスト図：簡単な矢印フロー

- **2.2 Collaborative パターン**（150～200行）
  - 定義：「複数エージェントが同じ入力に対して並列に処理し、結果を統合」
  - メリット：並列実行で速度向上、各エージェントが独立、柔軟性高
  - デメリット：調整が複雑、結果の統合ロジック必要、オーバーヘッド増加
  - ユースケース：複数視点のレビュー、並列翻訳、複数検索エンジン併用
  - テキスト図：複数のエージェントが並列に動く図

- **2.3 Tree パターン**（150～200行）
  - 定義：「階層的な分岐・マージにより、複雑なタスク分解を実現」
  - メリット：スケーラビリティ高、段階的な処理が自然、中間結果のキャッシュ可能
  - デメリット：構造設計が難しい、分岐の最適化が課題、デバッグが複雑
  - ユースケース：大規模マインドマップ生成、意思決定ツリー、ドキュメント階層化
  - テキスト図：木構造の図

- **2.4 パターン比較表**（80～120行）
  - 表形式で「実行方式、速度、実装難易度、柔軟性、適用シーン」を一覧化

**執筆ポイント**:
- 各パターンについて「定義 → メリット → デメリット → ユースケース」の順序を統一
- テキスト図は 1 つのパターンに付き 1 つ以上
- メリット・デメリットは「対比」で書く（例：「速いが複雑」）

#### セクション 3: GAN スタイルの特徴（400～600行）

**目標**: GAN スタイルの構造と動作フローを詳しく説明  
**構成**:

- **3.1 GAN スタイルの概要**（80～120行）
  - 定義：「生成（Generator）と評価（Evaluator）のループで品質を段階的に向上させるパターン」
  - 背景：「従来パターンでは 1 回の処理で高品質を達成困難 → ループで改善」
  - 適用シーン：複雑な技術記事・コード生成・デザイン案件等

- **3.2 エージェントの役割分担**（200～300行）
  - **Planner エージェント**（60～80行）
    - 役割：「タスク分析 → 仕様策定 → Generator への指示作成」
    - 入力：ユーザー要件・前回の Evaluator フィードバック
    - 出力：設計仕様書（spec.md）
    - 特徴：推論能力を重視（複雑な要件分析に向く）

  - **Generator エージェント**（60～80行）
    - 役割：「Planner の仕様に基づいて案を生成」
    - 入力：Planner の spec.md
    - 出力：記事・コード・デザイン案等
    - 特徴：効率重視（大量生成に向く）

  - **Evaluator エージェント**（60～80行）
    - 役割：「Generator の出力を評価 → スコア算出 → フィードバック作成」
    - 入力：Generator の出力・評価基準
    - 出力：スコア + 改善フィードバック
    - 特徴：厳密性重視（品質チェック専門）

- **3.3 動作フロー（ループ構造）**（100～150行）
  - ステップバイステップで「Planner → Generator → Evaluator → 判定 → ループor終了」を図示
  - テキスト図：流れ図（開始 → 各エージェント処理 → スコア判定 → ループ/終了の分岐）
  - スコア基準の設定方法を明示（例：「60点以上で合格」）
  - 停止条件：「スコア基準達成」「最大ループ回数」「タイムアウト」等

**執筆ポイント**:
- 各エージェントの「得意分野」を強調（Planner は複雑な分析、Generator は効率、Evaluator は厳密性）
- ループ構造を「フィードバック + 改善」という観点で説明
- 「なぜこの 3 つのエージェントなのか」という設計理由を示唆

#### セクション 4: なぜ GAN スタイルが有効か（300～400行）

**目標**: メリット・デメリット・適用条件を公正に説く  
**構成**:

- **4.1 GAN スタイルのメリット**（100～150行）
  - **品質向上ループ**：複数回の改善で高品質を実現
  - **自動改善**：Evaluator のフィードバック × Planner の最適化で、人手を最小化
  - **柔軟な仕様策定**：Planner が状況に応じて spec を調整可能
  - **失敗時のリカバリ**：「うまくいかなければ再ループ」という設計

- **4.2 GAN スタイルのデメリット・限界**（100～150行）
  - **計算コスト**: 複数回のエージェント実行 → API 呼び出し数増加
  - **レイテンシ**: ループ回数分の待ち時間
  - **スコア設定の難しさ**：「何点なら合格か」を判断するのが非自明
  - **過度な改善への陥穽**：ループが無限化しないよう設計が必要
  - **シンプルなタスクには過剰**：「1 回で完成するタスク」には向かない

- **4.3 適用条件・判断基準**（100～150行）
  - ✓ 向く：「複雑性が高い」「品質重視」「複数回の改善が許容される」タスク
  - ✗ 向かない：「低遅延必須」「単純なタスク」「スコア基準の設定が困難」
  - 判断フロー：「品質要件 → 複雑性 → 予算」の 3 軸で判断

**執筆ポイント**:
- メリット・デメリットは「同じ粒度」で書く（バランスが重要）
- 「GAN スタイルは最強」ではなく、「適切に選べば強力」というニュアンスを出す
- 読者が「自分たちに適用できるか」判断できるチェックリストを提示

## 5. 各セクションの執筆ポイント（全体統一）

### 共通原則
1. **1 段落 = 1 アイデア**：複数の概念を混在させない
2. **見出しの階層化**：セクション → サブセクション → 項目の 3 レベルを活用
3. **テキスト図の活用**：複雑な構造は図で説明（ASCII アート、Mermaid 互換）
4. **具体例の提示**：抽象的な説明の直後に「例えば…」で具体化
5. **比較・対比**：複数のパターンを述べる際は「〜に対して」「一方」で対比を明示
6. **終末への導き**：各セクション最後に「次のセクションへの橋渡し」を 1 文入れる

### 用語の統一
- **Planner / Generator / Evaluator**: 必ず英語（カタカナ化しない）
- **GAN スタイル**：「GAN スタイル」で統一（「GAN 型」は避ける）
- **スコア**：「スコア」で統一（「点数」「得点」は混用しない）
- **ループ**：「ループ」で統一（「繰り返し」は避ける）

### 日本語表記ルール
- **数字**：記事本文では「1 つ」「3 パターン」と算用数字 + 助数詞
- **括弧**：概念説明は「（例：…）」、補足は「～など」
- **句点**：1 文は 80 字以内を心がける
- **敬語**：「ですます」調で統一（技術記事の標準）

### レビュー観点

**構成の論理性**

## 8. エッジケース・注意点

### 実装レベルでの注意

1. **無限ループの防止**
   - 「スコアが延びず、ループが終わらない」ケースに対する対策を記事で言及
   - 推奨：「最大ループ回数 5」「タイムアウト 3600 秒」といった具体値の例示

2. **スコア基準の設定難度**
   - 「60 点で合格とは何か」を定義するのが非自明という課題を認める
   - 実装例で「スコア基準の決め方」を 1 段落で説明

3. **エージェントの役割混同**
   - Planner と Generator の責務境界が曖昧になりがちという注意喚起
   - 「Planner は『何を作るか』、Generator は『どう作るか』」という区別を明示

4. **ファイル構成の柔軟性**
   - セクション 5.1 の「推奨ファイル構成」は「最小限の例」であり、プロジェクトに応じて変更可能であることを明記

5. **従来パターンとの組み合わせ**
   - 「GAN スタイル内部に Pipeline を含む」など、複合的な設計も可能であることを示唆（Nice-to-Have）

### 執筆時の注意

1. **ECC 実装への依存度**
   - ECC の実装が変わっても「パターン論」としては成立するよう、本質的な説明を心がける

2. **言語・フレームワーク中立性**
   - 「Python で実装する」など特定の言語に限定しない
   - 疑似コードや概念図で言語依存を避ける

3. **読者の前提知識**
   - 「エージェント」「API 呼び出し」といった基本概念は説明なしで使える（想定読者参照）
   - 「スコアリング」「フィードバック」などは簡潔に定義

4. **記事の鮮度**
   - ECC の実装例の「バージョン依存」に気をつける
   - 可能なら「2026 年 5 月時点の ECC では…」という時間軸を入れる
