# マルチエージェントハーネスの設計パターン：GAN スタイルで品質を最大化する

## 1. 導入

複雑なタスクを 1 つのエージェントで処理しようとすると、品質や処理速度の面で限界が生じます。特に技術記事の執筆やコード生成など、複数の視点から検討が必要なタスクでは、単一エージェントでは十分な品質を実現困難です。マルチエージェント設計は、タスクを複数のエージェントに分割し、各エージェントが専門分野に特化することで、この問題を解決します。

本記事では、Claude Code で実装可能な 3 つのマルチエージェントパターンを比較し、特に「GAN スタイル」（Generator-Adversarial-Network）の実装方法と有効性について詳しく解説します。記事を読むことで、自分たちのプロジェクトに適切なパターンを選択できるようになるはずです。

---

## 2. 従来のマルチエージェントパターン

マルチエージェント設計には、いくつかの確立されたパターンがあります。各パターンの特性を理解することが、最適な選択につながります。

### 2.1 Pipeline パターン

**定義**：エージェント A → B → C と直列に接続し、前のエージェントの出力が次のエージェントの入力となるパターンです。

**メリット**：
- 実装が単純で、各エージェントの責務が明確
- 実行順序が一意に決まるため、デバッグが容易
- 中間結果をキャッシュして再利用可能

**デメリット**：
- 並列化できず、処理時間が各エージェントの処理時間の合計
- 1 つのエージェントがボトルネックになると全体が遅延
- 後段のエージェントが改善しても前段の誤りは取り戻せない

**テキスト図**：
```
┌─────────┐    ┌─────────┐    ┌─────────┐
│ Agent A │───→│ Agent B │───→│ Agent C │
└─────────┘    └─────────┘    └─────────┘
入力           中間出力        最終出力
```

**ユースケース**：データクリーニング → 分析 → レポート生成

---

### 2.2 Collaborative パターン

**定義**：複数のエージェントが同じ入力に対して並列に処理し、その結果を統合するパターンです。

**メリット**：
- 複数エージェントを並列実行して全体の処理時間を短縮
- 各エージェントが独立しており、追加・削除が容易
- 複数の視点から同時にアプローチ可能

**デメリット**：
- 結果の統合ロジックが複雑になりやすい
- 各エージェント実行のオーバーヘッド増加
- 結果の矛盾を解決するための追加処理が必要

**テキスト図**：
```
          ┌─────────┐
          │ 共有入力  │
          └────┬────┘
         ┌─────┼─────┐
         ↓     ↓     ↓
      ┌──┐ ┌──┐ ┌──┐
      │A1│ │A2│ │A3│（並列実行）
      └──┘ └──┘ └──┘
         ↓     ↓     ↓
         └─────┼─────┘
          ┌────▼────┐
          │ 結果統合  │
          └─────────┘
```

**ユースケース**：複数視点のコードレビュー、並列翻訳、複数検索エンジン併用

---

### 2.3 Tree パターン

**定義**：階層的な分岐・マージにより、複雑なタスクを段階的に分解・処理するパターンです。

**メリット**：
- スケーラビリティが高く、大規模なタスク分解が可能
- 段階的な処理が自然で、中間結果をキャッシュ可能
- 分岐・マージのバランスを調整して処理効率を最適化可能

**デメリット**：
- 木構造の設計が複雑で、分岐の最適化が非自明
- デバッグが複雑で、特定のパスを追跡しづらい
- 中間ノードの失敗が部分木全体に波及

**ユースケース**：大規模マインドマップ生成、意思決定ツリー、ドキュメント階層化

---

### 2.4 パターン比較表

| 項目 | Pipeline | Collaborative | Tree |
|------|----------|------------------|------|
| **実行方式** | 直列 | 並列 | 階層的分岐・マージ |
| **処理速度** | 遅い（合計時間） | 速い（最大時間） | 中程度（構造依存） |
| **実装難易度** | 低 | 中 | 高 |
| **柔軟性** | 低 | 中 | 高 |
| **適用シーン** | 単純な流れ | 多視点判定 | 複雑な構造化タスク |

## 3. GAN スタイルの特徴

従来の 3 つのパターンは「1 回の処理」を想定していますが、GAN スタイルは異なるアプローチです。品質向上を目指し、複数回のループで段階的に改善します。

### 3.1 GAN スタイルの概要

**定義**：生成（Generator）と評価（Evaluator）のループで品質を段階的に向上させるマルチエージェントパターン。

**背景**：従来のパターンでは、1 回の処理で十分な品質を達成するのが困難です。特に複雑な要件では、専門分野ごとに役割を分け、フィードバックループで改善する方が効果的です。

**適用シーン**：複雑な技術記事・高品質なコード生成・デザイン案の反復改善

---

### 3.2 エージェントの役割分担

GAN スタイルは 3 つのエージェントで構成されます。

#### Planner エージェント
- **役割**：タスクを分析し、仕様書（spec.md）を作成
- **入力**：ユーザー要件、Evaluator からのフィードバック
- **出力**：Generator への指示となる詳細仕様書
- **特徴**：複雑な要件分析に向く（Opus 推奨）

#### Generator エージェント
- **役割**：Planner の仕様に基づいて具体的な案を生成
- **入力**：Planner の spec.md
- **出力**：記事、コード、デザイン案など
- **特徴**：大量生成に向く（Haiku でも効果的）

#### Evaluator エージェント
- **役割**：Generator の出力を評価し、スコアとフィードバックを作成
- **入力**：Generator の出力、評価基準
- **出力**：スコア（0～100）+ 改善フィードバック
- **特徴**：厳密な品質チェック専門

---

### 3.3 GAN ループの動作フロー

```
┌──────────────────────┐
│  ユーザー要件        │
└──────────┬───────────┘
           ↓
      ┌─────────────┐
      │  Planner    │← Evaluator フィードバック
      │ (分析→spec) │
      └─────┬───────┘
           ↓
      ┌─────────────┐
      │ Generator   │
      │ (生成)      │
      └─────┬───────┘
           ↓
      ┌─────────────┐
      │ Evaluator   │
      │ (評価)      │
      └─────┬───────┘
           ↓
        スコア判定
       /        \
   ✓ ≥ 80点     < 80点
     |          |
    終了    ループ更新
```

**停止条件**：
- スコアが 80 点以上（または設定値以上）
- 最大ループ回数に到達（無限ループ防止）
- タイムアウト時間を超過

## 4. なぜ GAN スタイルが有効か

### 4.1 GAN スタイルのメリット

**品質向上ループ**：複数回の改善サイクルにより、1 回の処理より高品質を実現します。

**自動改善**：Evaluator のフィードバックと Planner の最適化により、人手による修正を最小化できます。

**柔軟な仕様策定**：Planner が状況に応じて spec を動的に調整できるため、不確実な要件でも対応可能です。

**失敗時のリカバリ**：うまくいかなかった場合、フィードバック反映による再ループで改善できます。

---

### 4.2 GAN スタイルのデメリット・限界

**計算コスト**：複数回のエージェント実行により、API 呼び出し数が増加します。コスト予算との兼ね合いが重要です。

**レイテンシ**：ループ 1 回に複数エージェント実行分の時間が必要です。低遅延が必須のタスクには向きません。

**スコア設定の難しさ**：「何点なら合格か」を判断するのは非自明であり、タスクごとに試行錯誤が必要です。

**過度な改善への陥穽**：スコア基準が低すぎるとループが無限化しないよう、停止条件の設計が重要です。

**シンプルなタスクには過剰**：1 回で完成するタスクに GAN スタイルを適用すると、コストだけが増加します。

---

### 4.3 適用条件・判断基準

**GAN スタイルが向くケース**：
- 品質要件が高い（80 点以上の出力が必須）
- タスク複雑性が高い（要件が多く、視点が複数必要）
- 複数回の改善が許容される（時間に余裕がある）
- コスト制約が緩い

**向かないケース**：
- 低レイテンシが必須（リアルタイム処理）
- タスクが単純（1 回で完成可能）
- スコア基準の設定が困難（定量的な評価基準がない）
- API 呼び出しのコスト上限が厳しい

**判断フロー**：「品質要件 → 複雑性 → 予算」の 3 軸で総合判断してください。

## 5. 実装のポイント

### 5.1 ファイル構成

推奨される最小限のファイル構成：

```yaml
project/
├── agents/
│   ├── planner.md          # Planner エージェント定義
│   ├── generator.md        # Generator エージェント定義
│   └── evaluator.md        # Evaluator エージェント定義
├── commands/
│   └── gan-build.md        # GAN ループ実行コマンド
├── specs/
│   └── {task}_spec.md      # タスク仕様書（Planner 生成）
├── outputs/
│   └── {task}_output.md    # 最終出力ファイル
└── config.yaml             # ループ設定（停止条件等）
```

各ファイルの役割：
- **agents/** → エージェント定義（システムプロンプト、能力等）
- **commands/** → エージェント実行順序とループ制御
- **specs/** → 仕様書（Planner が作成、Generator が参照）
- **outputs/** → 最終的な成果物
- **config.yaml** → ループ停止条件やスコア閾値

---

### 5.2 エージェント間のデータ受け渡し

Planner → Generator への spec.md フォーマット例：

```yaml
# task_spec.md

**タスク名**: 技術記事執筆

**要件**:
- 2000～3000文字
- 初級～中級向け
- コード例 2 個以上

**セクション構成**:
1. 導入（100字）
2. 基礎知識（400字）
3. 実装例（600字）
4. まとめ（200字）

**スタイル**: 「ですます」調、漢字・ひらがなのバランス

**評価基準**: 要件充足度、論理構成、実装可能性
```

Generator → Evaluator への出力受け渡し：生成ファイル（article.md）をそのまま Evaluator に渡します。

Evaluator → Planner へのフィードバック（JSON フォーマット）：

```json
{
  "score": 72,
  "evaluation": {
    "文字数": "2100字（OK）",
    "セクション構成": "4つ完成（要5つ）",
    "コード例": "1個（要2個以上）"
  },
  "feedback": "コード例を1個追加し、セクション5を新規作成してください"
}
```

---

### 5.3 停止条件の設定

ループ停止条件の設定例（YAML）：

```yaml
stop_conditions:
  score_threshold: 80         # スコア 80 点以上で完了
  max_iterations: 5           # 最大 5 ループまで
  timeout_seconds: 3600       # 1 時間でタイムアウト
  score_plateau_threshold: 2  # 2 ループ連続で改善なし → 終了
```

**説明**：
- `score_threshold`: スコアがこの値以上なら合格
- `max_iterations`: 無限ループ防止
- `timeout_seconds`: 総実行時間の上限
- `score_plateau_threshold`: スコア改善が停滞したら終了（リソース節約）

---

### 5.4 エラーハンドリング・デバッグ

**エージェント実行失敗時**：
- Planner の失敗 → spec.md 内容を簡略化して再実行
- Generator の失敗 → spec.md の要件を見直し、次ループでリトライ
- Evaluator の失敗 → 評価基準を確認し、ログを出力

**スコアが伸びない場合**：
- Planner の spec.md が不十分でないか確認
- Generator に「改善ポイント」を明示的に指示
- 評価基準が厳しすぎないか見直し

**ログ出力の重要性**：
各ループで以下をログに記録してください：
```
[Loop 1] Planner: spec.md 作成完了
[Loop 1] Generator: article.md 生成完了
[Loop 1] Evaluator: スコア 65 点
[Loop 1] 判定: ループ継続（フィードバック反映）
```

## 6. 実装例

ECC（EverythingClaudeCode）では、GAN スタイルの実装が gan-build.md で定義されています。

### 6.1 実行フロー

ECC の gan-build.md に記載されるコマンド構成は以下のようです：

```markdown
# gan-build.md - GAN ループ実行コマンド

## ステップ 1: Planner 実行
claude-code /agents/planner --input "user_requirement.txt" --output "specs/gen_spec.md"

## ステップ 2: Generator 実行
claude-code /agents/generator --input "specs/gen_spec.md" --output "outputs/draft.md"

## ステップ 3: Evaluator 実行
claude-code /agents/evaluator --input "outputs/draft.md" --rubric "rubrics/article-rubric.md" --output "evaluation.json"

## ステップ 4: スコア判定
IF evaluation.json.score >= 80 THEN
  DONE
ELSE
  UPDATE specs/gen_spec.md WITH evaluation.json.feedback
  GO TO ステップ 2
END IF
```

このフローにより、Planner が要件を分析 → Generator が生成 → Evaluator が評価という 3 段階が自動実行され、スコア基準達成までループします。

### 6.2 ECC の Planner / Generator / Evaluator エージェント定義

ECC での実装では、各エージェントが明確に定義されています：

**Planner の特徴**（ECC/agents/planner.md）：
- 複雑な要件分析に適した（モデル Opus 推奨）
- spec.md を生成
- Evaluator フィードバックを反映して spec を改善

**Generator の特徴**（ECC/agents/generator.md）：
- Planner の spec に基づいて高速に生成（Haiku でも有効）
- 大量のコンテンツ生成に特化
- spec に従うことを最優先

**Evaluator の特徴**（ECC/agents/evaluator.md）：
- ルーブリック（採点基準）に基づいて厳密に評価
- スコアと改善フィードバックを JSON で出力
- Generator の出力を客観的に審査

## 7. まとめ

本記事では、マルチエージェント設計の 3 つのパターン（Pipeline、Collaborative、Tree）を比較し、特に GAN スタイルの特徴・メリット・デメリット・実装方法を詳しく解説しました。

**重要なポイント**：
- マルチエージェント設計には複数のパターンがあり、タスク特性に応じて選択が必要
- GAN スタイルは品質向上ループにより高品質を実現し、複雑で品質重視のタスクに向く
- 反面、コスト・レイテンシ・スコア設定に課題があり、適用条件の判断が重要
- ファイル構成、データフロー、停止条件を正しく設計すれば、自動改善が実現可能

**応用可能性**：GAN スタイルは、技術記事・コード生成・デザイン案だけでなく、翻訳、カスタマーサポート対応、データ分析レポートなど、品質が重視される多くの分野に応用できます。

**次のステップ**：ECC の gan-build.md や agents/gan-*.md を参照し、自分たちのプロジェクトに適応させたカスタムハーネスを構築してみましょう。各エージェントをカスタマイズして、特定のドメイン要件に合わせることで、さらに強力な品質管理システムが実現できます。
