### 2. 従来のマルチエージェント設計パターン（700～800 文字）
詳細度: 各パターン 200～250 文字程度

#### 2.1 Pipeline パターン（直列接続）
- **図**: A → B → C の流れ
- **説明**: エージェント A の出力がエージェント B の入力になる、の繰り返し
- **ユースケース**: テンプレート出力 → レビュー → 修正、など
- **長所**: シンプル、実装が容易、デバッグしやすい
- **短所**: 途中のエージェントが失敗すると全体が止まる、柔軟性が低い
- **実装例**: 簡潔な疑似コード（構成ファイル例）

#### 2.2 Collaborative パターン（並列協調）
- **図**: 複数エージェントが並列で動作し、最後に統合する図
- **説明**: A、B、C が同時に異なるアプローチで解く → 結果を統合
- **ユースケース**: 複数の観点からの分析、複合的な判断が必要な場面
- **長所**: 効率的（並列化）、多角的視点
- **短所**: 結果の統合が複雑、実装と管理が難しい
- **実装例**: 並列実行の概略（スケジューリング、同期メカニズム）

#### 2.3 Tree パターン（階層構造）
- **図**: トップダウンツリー構造（親 → 複数の子エージェント）
- **説明**: 親エージェントが判断し、異なる子エージェントに分岐
- **ユースケース**: カテゴリ分類後に異なる処理、複雑な意思決定ツリー
- **長所**: 柔軟性、複雑な分岐処理に対応
- **短所**: 深さが増すと管理が複雑化、エージェント数が増加
- **実装例**: 分岐ロジックと子エージェント管理

### 3. GAN スタイルの特徴（750～850 文字）

#### 3.1 GAN スタイルとは（概略）
- **定義**: Generative-Adversarial-Network にインスパイアされた連携パターン
- **4 つの役割**:
  - **Planner**: 仕様・要件を明確化（本記事なら、記事の骨組みを決める）
  - **Generator**: 案を生成（複数パターンを試す）
  - **Evaluator**: 品質評価（明確な基準でスコア付け）
  - **Loop**: スコア達成まで Generator-Evaluator を繰り返す

#### 3.2 各エージェントの責務
**Planner エージェント:**
- 入力を分析し、タスク仕様書（spec）を生成
- 「何を、どう、いつまでに」を定義
- 品質基準を明文化

**Generator エージェント:**
- spec に沿って複数バリエーションを生成
- 同じ spec からでも異なるアプローチを試す

**Evaluator エージェント:**
- spec の品質基準に照らして評価
- スコア化（数値または定性評価）
- 改善点をフィードバック

**Loop:**
- Evaluator のスコアが合格ライン以上になるまで、Generator を呼び出し続ける
- 最大試行回数などの停止条件

#### 3.3 実装の流れ
```
(Theme Input)
  ↓
[Planner] → spec.md
  ↓
[Generator] → article_v1.md
  ↓
[Evaluator] → score + feedback
  ↓
score >= threshold ?
  YES → Output article
  NO → [Generator] (feedback 込み) → article_v2.md → ...
```

### 4. なぜ GAN スタイルが有効か（500～600 文字）

#### 4.1 品質と効率のバランス
- Evaluator による自動チェック → 人間の目を減らせる
- 明確な基準があることで、Generator も改善方向が分かる
- 複数試行により、単一回の生成より品質が高い傾向

#### 4.2 自動改善ループ
- Feedback loop が明示的
- 改善プロセスが自動化される（人手がかからない）
- スコア基準を変えることで、簡単に品質設定を変更可能

#### 4.3 スケーラビリティ
- エージェント数が少ない（Planner, Generator, Evaluator）
- 新しいタスクでも同じ 3 つのロールを適用可能
- Pipeline や Tree より実装が簡潔

#### 4.4 人間の介入を最小化
- 仕様が明確 → 人間が「何をすべきか」を後付けで説明する必要がない
- 評価基準が明確 → 「OK/NG」の判断が機械化できる

### 5. メリット・デメリット（300～350 文字）

#### メリット
- 品質が高い（複数試行による改善）
- スケーラブル（新しいタスクへの適用が容易）
- 自動化度が高い（人間の関与が少ない）
- デバッグが容易（各ステップが独立）

#### デメリット
- コスト増（複数試行による API 呼び出し増加）
- 初期設計に時間（spec と評価基準の定義）
- すべてのタスクに適さない（単純なタスクには過剰設計）
- Loop 制御が複雑（停止条件の設定）

### 6. 実装のポイント（600～700 文字）

#### 6.1 ファイル構成
```
project/
├── planner.py         # Planner エージェント
├── generator.py       # Generator エージェント
├── evaluator.py       # Evaluator エージェント
```

#### 6.2 エージェント間データ受け渡し

- Generator → Evaluator: 生成物（article.md など）

#### 6.3 停止条件の設定
- スコア閾値（`score >= 80/100` など）
- 最大試行回数（無限ループ防止）
- タイムアウト（API コスト管理）

#### 6.4 失敗時の対応
- Evaluator が「改善不可能」と判定した場合の処理
- Generator が spec を満たせない場合のログ出力
- 人間への エスカレーション

- ECC/commands/gan-build.md の構成

### 7. 実装例：記事執筆への適用（400～500 文字）

#### ユースケース
「マルチエージェント設計の技術記事を執筆する」タスク

#### 各エージェントの役割
**Planner:**
- theme.md を入力として、記事の仕様書（構成、文字数、盛り込むべき要素）を生成
- 例: spec.md（本ドキュメント）

**Generator:**
- spec.md に従い、記事本体を執筆
- 複数バージョンを試す（例: 学術的トーン vs. 実践的トーン）

**Evaluator:**
- 以下を基準に評価：
  - 文字数が要件を満たしているか
  - すべての必須セクションが含まれているか
  - パターン比較が公正か
  - コード例は実装可能か
  - スコア化（例: 1～100）

**Loop:**
- スコア 80 以上で完了
- 3 回の試行で改善できない場合は、Planner に戻ってフィードバック

#### 実装コード（疑似コード）
```python
# Loop 制御
spec = planner_agent(theme)
max_attempts = 5
threshold = 80

for attempt in range(max_attempts):
    article = generator_agent(spec)
    score, feedback = evaluator_agent(article, spec)
    
    if score >= threshold:
        return article
    else:
        spec['feedback'] = feedback
        continue

# 改善できない場合
report_failure(spec, feedback)
```

## エッジケース・注意点

### 1. GAN スタイルの過信
- **注意**: すべてのタスクに適さない（単純なタスクは Pipeline で十分）
- **記事での対策**: 「GAN スタイルを選ぶべき場面」を明確に

### 2. コスト増大
- **注意**: 複数試行による API コスト増加
- **記事での対策**: Loop の「最大試行回数」「threshold 設定」の重要性を強調

### 3. Loop の無限化
- **注意**: Evaluator の基準が甘すぎると、永遠に終わらない
- **記事での対策**: 「threshold は適切に設定する」「timeout を入れる」

### 4. Spec の品質
- **注意**: Planner が不良 spec を生成すると、Generator が何もできない
- **記事での対策**: 「Planner の出力品質が全体の品質を左右する」という重要性

### 5. Feedback の有効性
- **注意**: Evaluator のフィードバックが曖昧だと、Generator が改善できない
- **記事での対策**: 「フィードバックは構造化すべき」という推奨

### 6. 他のパターンとの組み合わせ
- **注意**: 実際には GAN の中に Pipeline や Tree が隠れているかもしれない
- **記事での対策**: 「パターンはピュアではなく、実装では混在する」という柔軟性

### 7. 言語・モデルの違い
- **注意**: 日本語での実装例が十分か？Claude Opus / Haiku での違いは？
- **記事での対策**: 本記事は日本語を想定。多言語対応は触れない（スコープ外）

### 8. 評価基準の客観性
- **注意**: Evaluator が「良い悪い」を判断する基準が客観的でない場合
- **記事での対策**: 「スコア化は可能な限り定量的に」という指針
