# マルチエージェントハーネスの設計パターン：GANスタイルで品質を最大化する

## 1. 導入：複数エージェントをどう組み合わせるか

単一のAIエージェントでは対応できないタスクに直面したことはないでしょうか。複雑な記事執筆、コード生成、デザイン案出しなど、品質が重視される作業では「ひとつのエージェントに任せる」では不十分です。

複数エージェントを組み合わせれば、それぞれの強みを活かして高品質な成果物を生み出せます。しかし、どのように連携させるか—これが重要な設計判断です。Pipeline（直列接続）、Collaborative（並列協調）、Tree（階層構造）、そして**GAN スタイル**（Planner-Generator-Evaluator-Loop）の4つのパターンがあります。

本記事では、これらのパターンを比較しながら、なぜ**GAN スタイル**が品質を最大化するのか、そして実装方法を具体例とともに説明します。記事を読み終わるころには、自分のプロジェクトに最適なパターンを選択できるようになるでしょう。

---

## 2. 従来のマルチエージェント設計パターン

### 2.1 Pipeline パターン（直列接続）

最もシンプルなパターンです。エージェント A の出力がエージェント B の入力になり、その出力がエージェント C に渡される—という流れです。

```
Input → [Agent A] → Output_A → [Agent B] → Output_B → [Agent C] → Final Output
```

**ユースケース例**：テンプレート生成 → スタイル調整 → 最終校正

**長所**：
- 実装がシンプル、デバッグしやすい
- 各エージェントの責務が明確

**短所**：
- 途中で失敗すると全体が止まる
- 改善の柔軟性が低い
- エージェント数が増えると管理が複雑化

### 2.2 Collaborative パターン（並列協調）

複数のエージェントが異なるアプローチで同時に問題を解き、最後に結果を統合するパターンです。

```
Input → ┬→ [Agent A] → Output_A ─┐
         ├→ [Agent B] → Output_B ─┼→ [Integrator] → Final Output
         └→ [Agent C] → Output_C ─┘
```

**ユースケース例**：複数の視点からコードレビューを実施し、指摘を統合

**長所**：
- 複数の観点から問題を分析できる
- 処理時間が短い（並列化）
- 単一視点の盲点を補える

**短所**：
- 結果の統合ロジックが複雑
- エージェント間の重複や矛盾の処理が困難
- 実装と管理のコストが高い

### 2.3 Tree パターン（階層構造）

親エージェントが判断し、異なる子エージェントに処理を分岐させるパターンです。

```
[Parent Agent]
     ↓
判定結果
  ├→ [Child Agent A]
  ├→ [Child Agent B]
  └→ [Child Agent C]
```

**ユースケース例**：ドキュメント分類後、カテゴリ別に異なる処理を実行

**長所**：
- 複雑な分岐処理に柔軟に対応
- 各枝での処理を独立して管理可能

**短所**：
- ツリーの深さが増すと管理が指数関数的に複雑化
- エージェント数が増加しやすい
- デバッグが困難

## 3. GAN スタイルの特徴：自動改善ループで品質を徹底

GAN（Generative Adversarial Network）にインスパイアされたマルチエージェント連携パターンです。

### 3.1 4つの役割

**Planner エージェント**
- 入力を分析して、詳細なタスク仕様書（spec）を生成
- 「何を、どのレベルまで、どのようなフォーマットで」を明文化
- 品質評価の基準も定義

**Generator エージェント**
- spec に従って成果物を生成
- 複数バージョン・複数アプローチを試す

**Evaluator エージェント**
- spec の基準に照らして、成果物を自動評価
- スコア化と改善フィードバックを提供
- 「合格か不合格か」を判定

**Loop**
- Evaluator のスコアが合格ラインを超えるまで、Generator を繰り返し呼び出す
- 最大試行回数やタイムアウトで無限ループを防止

### 3.2 実装フロー

```
(Theme Input)
  ↓
[Planner] → spec.md
  ↓
[Generator] → article_v1.md
  ↓
[Evaluator] → score: 75/100, feedback: "セクション不足"
  ↓
score >= 80 ?
  NO → spec に feedback を統合
       ↓
  [Generator] → article_v2.md
       ↓
  [Evaluator] → score: 88/100 ✓
  ↓
  YES → Output article_v2.md
```

### 3.3 エージェント間のデータ受け渡し

| 受け渡し | フォーマット | 内容 |
|---------|------------|------|
| Planner → Generator | spec.md（構造化 Markdown） | 要件、文字数、盛り込むべき要素、品質基準 |
| Generator → Evaluator | article.md（生成物） | 実装した成果物 |
| Evaluator → Generator | feedback.md（JSON/Markdown） | スコア、改善点、次の試行への指示 |

## 4. なぜ GAN スタイルが有効か

### 4.1 品質と効率のバランス

従来のパターンでは、品質を高めようとすると人手による確認が増え、効率が落ちます。GAN スタイルは異なります。

Evaluator による**自動チェック**があるため、人間の目を減らせます。同時に、明確な基準があることで Generator も「どう改善すべきか」が分かり、試行錯誤の方向性が明確になります。

### 4.2 自動改善ループ

改善プロセスが明示的で自動化されている点が強力です。Evaluator のフィードバックをそのまま Generator に渡すことで、人間を介さずに品質を高められます。

また、評価基準を変えるだけで、簡単に品質要求を調整可能—これはスケーラビリティの観点で重要です。

### 4.3 スケーラビリティと再利用性

エージェント数が少ない（3つの役割のみ）ため、新しいタスクに対しても同じ構造を適用できます。

Pipeline や Tree は、新しいタスクごとに「エージェントの数と配置」を再設計する必要があります。GAN スタイルは Planner、Generator、Evaluator の 3 つの役割さえあれば、ほぼすべてのタスクに対応できるのです。

### 4.4 人間の介入を最小化

仕様が明確→「何をすべきか」を後付けで説明する必要がない  
評価基準が明確→「OK / NG」の判断が機械化できる

これにより、人間は「仕様定義」と「最終判断」のみに専念でき、反復的な作業負荷が大幅に減ります。

## 5. メリット・デメリット

| 観点 | 説明 |
|------|------|
| **メリット：品質** | 複数試行による改善で、単一回の生成より品質が高い |
| **メリット：自動化** | フィードバック ループが自動化され、人手がかからない |
| **メリット：スケーラビリティ** | 新しいタスクへの適用が容易（3 つの役割で再利用） |
| **メリット：デバッグ** | 各ステップが独立しており、問題箇所を特定しやすい |
| **デメリット：コスト** | 複数試行による API 呼び出しが増加し、実行コストが高い |
| **デメリット：初期設計** | spec 定義と評価基準の設定に時間と工数がかかる |
| **デメリット：適用範囲** | 単純なタスク（yes/no 判定など）には過剰設計 |
| **デメリット：Loop 制御** | 停止条件（threshold、max_attempts）の設定が難しい |

## 6. 実装のポイント

### 6.1 ファイル構成

```
project/
├── planner.py           # Planner エージェント
├── generator.py         # Generator エージェント
├── evaluator.py         # Evaluator エージェント
├── loop.py              # Loop 制御（メイン）
├── specs/               # 生成された仕様書（spec.md）
│   └── task_spec.md
├── outputs/             # 最終成果物（合格版）
│   └── final_article.md
└── logs/                # 試行ログ、フィードバック履歴
    └── attempt_history.json
```

### 6.2 Planner 出力の例

```yaml
# spec.md
## タスク要件

### 概要
マルチエージェント設計パターンの技術記事を執筆する。

### 構成要件
- セクション数：6 個以上
- 文字数：2000～3000 字
- コード例：3 個以上
- テーブル・リスト・ダイアグラム：必須

### 品質基準
- パターン説明の正確性
- 図表の有用性（読者が理解しやすいか）
- 実装可能性（読者が自プロで適用できるか）

### 評価方法
スコア = （構成点 30%）+ （正確性 30%）+ （実装可能性 40%）
```

### 6.3 Evaluator フォーマットの例

```json
{
  "score": 78,
  "threshold": 80,
  "verdict": "RETRY",
  "feedback": {
    "structure": "セクション数は十分だが、流れが若干不自然",
    "accuracy": "パターン説明は正確。GAN スタイルの詳細は優秀",
    "practicality": "コード例は2個のみ。3個以上が要件",
    "next_action": "コード例を1個追加し、構成を改善してください"
  },
  "attempt": 1,
  "max_attempts": 5
}
```

### 6.4 Loop メイン処理（疑似コード）

```python
def gan_loop(theme_input, max_attempts=5, threshold=80):
    # Step 1: Planner が spec を生成
    spec = planner_agent(theme_input)
    print(f"[Planner] spec generated: {len(spec)} chars")
    
    # Step 2: 改善ループ
    for attempt in range(1, max_attempts + 1):
        # Generator が成果物を生成
        artifact = generator_agent(spec)
        print(f"[Generator] Attempt {attempt}: {len(artifact)} chars")
        
        # Evaluator が評価
        evaluation = evaluator_agent(artifact, spec)
        score = evaluation["score"]
        feedback = evaluation["feedback"]
        
        print(f"[Evaluator] Score: {score}/{threshold}")
        
        # 合格判定
        if score >= threshold:
            print(f"[Loop] ✓ PASSED at attempt {attempt}")
            return artifact, evaluation
        
        # 不合格の場合、フィードバックを spec に統合
        print(f"[Loop] Feedback: {feedback['next_action']}")
        spec["feedback"] = feedback
    
    # 最大試行回数に達した場合
    print(f"[Loop] ✗ FAILED after {max_attempts} attempts")
    return None, evaluation
```

### 6.5 停止条件の設定

- **スコア閾値**: `score >= 80` など明確な数値基準
- **最大試行回数**: `max_attempts = 5` で無限ループを防止
- **タイムアウト**: 実行時間の上限を設定してコスト管理
- **改善不可判定**: Evaluator が「spec が不良」と判断した場合は Planner に戻す

## 7. 実装例：記事執筆への適用

### 7.1 ユースケース

本記事執筆のプロセスそのものが GAN スタイルの実例です。

**タスク**: マルチエージェント設計パターンの技術記事を執筆
  - 要件：2000～3000 字、4 パターン比較、コード例 3 個以上
  - 品質基準：正確性・実装可能性・わかりやすさ

**各エージェントの役割**：

| エージェント | 責務 |
|-----------|------|
| **Planner** | theme.md を入力として spec.md を生成（構成、文字数、評価基準） |
| **Generator** | spec.md に従い、article.md を複数バージョン執筆 |
| **Evaluator** | 文字数、構成、正確性、実装可能性をスコア化 |
| **Loop** | スコア 80 以上で完了。3 回改善できなければ Planner に戻す |

### 7.2 実際の流れ

```
1回目試行:
  Generator → article_v1.md (2100 字)
  Evaluator → score: 75 ("コード例が2個のみ")
  
2回目試行:
  Generator → article_v2.md (2450 字、コード例3個)
  Evaluator → score: 82 ✓ PASSED
  
出力: article_v2.md
```

### 7.3 実装コード（Python 疑似コード）

前節の Loop メイン処理が、そのまま本タスクに適用できます。

重要なのは、**Generator と Evaluator が何度でも呼び出せる設計**にすること。そしてフィードバックが**構造化されている**こと（単なる「悪い」ではなく「何が足りないか」を明記）。

## 8. パターン比較表

| パターン | ユースケース | 実装複度 | 品質保証 | スケーラビリティ | 推奨用途 |
|---------|-----------|--------|--------|------------|--------|
| **Pipeline** | 単純な処理フロー（テンプレ生成 → 整形 → 出力） | 低 | 各段階での検証 | 低 | シンプルなタスク |
| **Collaborative** | 複数の観点から分析（コードレビュー、複合判定） | 中 | 並列化で多角的 | 中 | 多面的分析が必要 |
| **Tree** | 複雑な分岐処理（分類後、異なる処理） | 中〜高 | 階層構造で明確 | 中 | 条件分岐が多い |
| **GAN** | 品質が最重要なタスク（記事、デザイン、コード） | 中 | 自動改善ループ | **高** | 高品質・反復改善 |

## 9. GAN スタイルが不向きな場面

すべてのタスクに GAN スタイルが適切とは限りません。

- **単純な判定**: yes/no だけなら Pipeline で十分
- **リアルタイム性重視**: 複数試行は時間がかかるため、即座の応答が必要なタスクには不向き
- **質より速度**: 初版を素早く出力したい場合は Generator 1 回で十分
- **低コスト優先**: API 呼び出し回数が増えるため、コスト制約がきつい場合は避けるべき

## 10. まとめ

**4 つのパターンの特性**：

- **Pipeline**: シンプルだが、柔軟性と品質が限定的
- **Collaborative**: 多角的視点を得られるが、統合が複雑
- **Tree**: 複雑な分岐に対応するが、管理負荷が大きい
- **GAN**: 品質と自動化のバランスに優れ、スケーラブル

**重要なメッセージ**：「タスクの性質に応じて、最適なパターンを選ぶことが、エンジニアとしての判断力である」

**GAN スタイルの位置付け**：品質が重要で、反復改善が必要なタスク（記事執筆、コード生成、デザイン案出し、分析報告書など）では、**第 1 選択肢**となります。

**次のステップ**：

1. 自分のプロジェクトで反復改善が必要なタスクを洗い出す
2. Planner、Generator、Evaluator のエージェントを実装
3. 小さなスコープで試し、評価基準を調整しながら精緻化
4. 複数のタスクに適用し、パターンを確立

GAN スタイルは「自動化と品質の両立」を実現する、Claude Code や AI エージェントの時代だからこそ活躍するパターンです。ぜひ自分のハーネスに組み込んでみてください。
