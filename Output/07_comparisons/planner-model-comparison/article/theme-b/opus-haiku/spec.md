### 4.2 各セクションの詳細設計

#### セクション1: 導入（250～350文字）
**目的**: 読者の「プロンプトだけで十分では？」という疑問を引き出す

**執筆ポイント**:
- プロンプトの成功例を1～2つ挙げる（「良いプロンプト = 良い結果」のイメージを醸成）
- しかし「一度だけ成功」「毎回説明が必要」という現実的な課題を指摘
- 「実は大事な別の層がある」という期待感を与える

**フック**: 「AI エンジニアリングには『見えない階層』がある」

---

#### セクション2: プロンプトエンジニアリングの限界（400～500文字）
**目的**: なぜプロンプトだけでは不十分なのかを理論的に説明

**執筆ポイント**:
- **単一セッション内の最適化**: プロンプトは「このセッション内で何をするか」の指示に過ぎない
- **文脈喪失の問題**: セッション終了後、同じコンテキストを再度入力する必要がある
- **スケーラビリティの限界**: 100 回目のセッションでも「この プロジェクトについてのルール」を説明し直す
- **具体例**: Claude Code ユーザーが同じ設定を毎回手で入力する苦労

**図表素材**:
```
プロンプト（セッション内）
  ↓
  └─ セッション終了 → コンテキスト消滅 → 次回セッションで再入力
     (1回目、2回目、3回目...)
```

#### セクション3: コンテキストエンジニアリングの定義と範囲（500～650文字）
**目的**: 「コンテキストエンジニアリングとは何か」を明確に定義

**執筆ポイント**:
- **定義**: AI システムに対する「長期的な指針・背景・ルール」をシステムレベルで組み込む技術
- **範囲**: プロンプト（一度限り）を超えて、ルール・スキル・フック・設定を総合的に設計すること
- **3つの要素**:
  1. **ルール（Rules）**: 「AI がどう振る舞うべきか」の原則（道徳規範的）
  2. **スキル（Skills）**: 「AI が何ができるか」の拡張機能（道具的）
  3. **フック（Hooks）**: 「どのタイミングで何を実行するか」の自動化（時間軸的）
- **エージェント間のコンテキスト伝播**: 複数 AI エージェント間でコンテキストを共有
- **プロンプトとの違い**: プロンプトは「短期の指示」、コンテキストは「長期の型枠」

**比較表（挿入位置）**: セクション3の最後に「プロンプト vs コンテキスト比較表」を配置

## 7. エッジケース・注意点

### 7.1 記事執筆時の注意点

1. **ECC からの実装例は「読み取り専用」で**
   - ECC のコードやファイルをそのままコピー＆ペーストするのは OK
   - ただし「実在する」ことを確認してから引用する
   - 架空の例示は絶対に避ける

2. **「初心者向け」と「説得力」のバランス**
   - 過度に簡潔に説明すると説得力が失われる
   - 逆に技術詳細に走ると初心者がついてこれない
   - 「基本概念 + 1つの具体例」のセットを複数用意することで両立

3. **プロンプトエンジニアリングの否定にならないように**
   - 「プロンプトは不要」ではなく「プロンプト + コンテキスト」という立場を貫く
   - プロンプトの価値も認めつつ、限界を指摘する

4. **ECC 以外の Claude Code の例について**
   - ECC は理想形だが、読者は自分のプロジェクトで実装することが多い
   - 「Claude Code の一般的な機能」（.claude/rules, .claude/skills など）の説明も含める
   - 「ECC はこの機能をこう実装している」という構成がベスト

5. **セクション4（実装例）の構成**
   - ルール → スキル → フック → エージェント設定の順で説明すると理解しやすい
   - 各例は「何をするための機能か」を 1～2 文で明確にしてから実装を示す

### 7.2 文字数調整時の優先度

- **削るべき（優先度低）**: 図表、注釈、補足情報
- **残すべき（優先度高）**: セクション1～3（導入・限界・定義）、セクション4 の実装例
- **文字数超過時の対応**: セクション5（ポイント）や 6（まとめ）を簡潔に調整

### 7.3 構成を変更した場合の注意

- テーマ要件に「導入 → 限界 → 定義 → 実例 → まとめ」という構成が示唆されているため、大幅な入れ替えは避ける
- セクション内の順序は変更してもよい（例：実例を先に示してから限界を説明するなど）
- ただし「限界を説明せずに実例だけ示す」のはコンテキストが不足するため避ける
