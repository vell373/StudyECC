# コンテキストエンジニアリング入門：プロンプトより強力な設計技術

## 1. 導入

「良いプロンプトを書けば、AI は期待通りに動く」—— これは半分正しく、半分間違っています。確かに1回のセッション内なら工夫したプロンプトが大きな力を発揮します。しかし現実は異なります。同じプロジェクトに100回アクセスするたびに「命名規則はこれです」と説明し直す。セッション終了時にプロジェクト固有のルールが消える。こうした繰り返しが開発効率を蝕みます。

AI エンジニアリングには「見えない階層」があります。それが **コンテキストエンジニアリング** —— システム全体でAI に対する指針を組み込む技術です。

## 2. プロンプトエンジニアリングの限界

プロンプトエンジニアリングは、**単一セッション内** の最適化に特化しています。しかし3つの大きな限界があります。

**1) 文脈喪失：** セッション終了後に背景知識が消滅
**2) スケーラビリティ欠如：** 規模が大きくなると説明コストが増大
**3) 一貫性保証の困難さ：** 複数セッションで同じ原則を維持するのが難しい

## 3. コンテキストエンジニアリングの定義と範囲

コンテキストエンジニアリングとは、**AI システムに対する「長期的な指針・背景・ルール」をシステムレベルで組み込む技術** です。ルール・スキル・フック・設定を総合的に設計することで、セッション間の一貫性を保ち、スケーラブルな AI 運用を実現します。

主要な3つの要素：

- **ルール（Rules）** - AI がどう振る舞うべきかの原則（コーディング規約、セキュリティ方針）
- **スキル（Skills）** - AI が何ができるかの拡張機能（自動分析、パターン抽出）
- **フック（Hooks）** - 自動化されるアクション（コミット前のチェック、ファイル保存時の処理）

さらに、複数の AI エージェント（Planner、Generator、Reviewer など）が同じ指針で協働できます。

プロンプトとコンテキストは「対立」ではなく「補完」の関係です。

| 観点 | プロンプト | コンテキスト |
|------|-----------|-----------|
| **スコープ** | 単一セッション内 | セッションを超えた全体 |
| **寿命** | セッション終了時に消滅 | システムに永続的に組み込まれ |
| **定義者** | ユーザー（毎回） | システム設計者（一度だけ） |
| **再利用性** | なし（毎回説明が必要） | あり（何度でも機能） |
| **スケーラビリティ** | 低い | 高い |

## 4. ECC / Claude Code での実装例

### 例1: ルール（Rules）による一貫性確保

ECC の `.claude/rules/` フォルダには、コーディング規約やセキュリティ方針がテキストファイルとして整理されています。例えば `coding-standards.md` には「変数は camelCase、定数は UPPER_SNAKE_CASE」といった規則が記載されます。Claude Code がこのプロジェクトで動作するたび、このルールが自動的に AI に伝わります。

### 例2: フック（Hooks）による自動化と一貫性

`settings.json` の `hooks` セクションでは、自動実行されるアクションを定義できます。

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "node scripts/hooks/pre-write-validation.js",
            "description": "Write 実行前のコンテキスト検証"
          }
        ]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "node scripts/hooks/post-bash-auto-commit.js",
            "async": true,
            "description": "Bash 実行後の自動コミット"
          }
        ]
      }
    ]
  }
}
```

フックにより、ファイル保存時に自動バリデーション、コマンド実行後に自動コミットなど、指示なしに AI が「このシステムではこうするべき」と判断して動作します。

### 例3: エージェント間コンテキスト伝播

ECC では、複数の AI エージェント（Planner、Generator、Reviewer）の役割が CLAUDE.md で一元管理されます。すべてのエージェントが同じルール・スキル・コンテキストを参照することで、分割されたタスク間でも一貫性が保たれます。これにより「Planner が設計した方針を Generator が正確に実装でき、Reviewer が同じ基準でレビューできる」という状況が実現されるのです。

## 5. 実装のポイント

**ポイント1: システムの「型」を先に設計する**
ルール・スキル・フックを先に定義し、その上でプロンプトを書く。この順序が重要です。

**ポイント2: 初心者は Rules から始める**
Rules は最も実装が簡単です。CLAUDE.md に `.md` ファイルを追加するだけで機能し始めます。

**ポイント3: 複数セッション / 複数エージェントで同じ説明が繰り返されたら、コンテキスト化する**
この状況がコンテキストエンジニアリングへの移行サインです。

## 6. まとめ

プロンプトエンジニアリングとコンテキストエンジニアリングは対立ではなく補完関係です。プロンプトは「今このセッションで」の最適化、コンテキストは「このシステムではいつも」の一貫性を実現します。

ルール・スキル・フック・エージェント設定を体系的に整理することで、複数セッション・複数タスク間での一貫性と効率が大幅に改善されます。

まずは自分のプロジェクトで1つのルール、1つのスキルを作ってみてください。「毎回説明する手間が減った」という体験が、AI エンジニアリングへの理解を深めるはずです。
