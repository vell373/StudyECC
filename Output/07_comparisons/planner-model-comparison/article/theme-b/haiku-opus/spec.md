## 各セクションの執筆ポイント

### 1. 【導入】 良いプロンプトだけでは足りない理由

**目的**: 読者の「プロンプト疲れ」に寄り添い、新しい視点の存在を予感させる

**ポイント**:
- 「プロンプトを何度も修正しているのに、結果が不安定」という読者の実感を名言する
- 例：「同じプロンプトで複数回実行すると、ある時は完璧だが、別の時は外れている」
- 「それは実はプロンプトの問題ではなく、[別の層の問題]かもしれません」という問題提起
- 「コンテキスト」という言葉の予告（まだ定義しない）

**避けるべき**: プロンプトエンジニアリングの全否定。あくまで「補完」という立場

---

### 2. 【プロンプトエンジニアリングの限界】

**目的**: 「なぜコンテキスト設計が必要か」の理論的背景を説明

**ポイント**:
- **限界1**: 単一セッション内での最適化に留まる
  - 例：チャットウィンドウを閉じたら工夫が無駄になる
- **限界2**: ユーザーごと・用途ごとに異なるプロンプトが必要
  - 例：営業向け AI とエンジニア向け AI で全く異なるプロンプトを書かねばならない
- **限界3**: メンテナンスが難しい
  - 例：プロンプト記述の「ベストプラクティス」が散在し、統一できない
- **限界4**: スケーラビリティの欠如
  - 例：エージェント数が増えると、管理が爆発的に複雑化する

**避けるべき**: 学術的な難しい言葉。「何が困るか」を具体的に

---

### 3. 【コンテキストエンジニアリングとは】

**目的**: 概念の定義を与え、プロンプトとの違いを明確化

**ポイント**:
- **定義**: 「セッション前・セッション外で、AI の行動・判断の枠組みを設計する技術」
- **範囲**: 以下を統合的に設計する
  - ルール・ガイドライン（何をやるべきか、やってはいけないか）
  - スキル・ツール（AI が使える能力）
  - フック（特定のイベント時に自動実行される処理）
  - エージェント構成（複数 AI の役割分担）
  - メモリ・状態管理（セッション間での学習）

- **「コンテキスト」という名前の理由**: LLM の入力（context）を、セッション時点で最適化するのではなく、[事前に・再利用可能な形で]設計すること

**ここに比較表を挿入**:

| 観点 | プロンプトエンジニアリング | コンテキストエンジニアリング |
|------|---------------------------|---------------------------|
| **適用時期** | セッション内・リアルタイム | セッション前・事前設計 |
| **保持期間** | 単一セッション内に限定 | 複数セッション・長期維持 |
| **スコープ** | プロンプト文字列の最適化 | システム全体の行動設計 |
| **ツール** | テキスト編集・試行錯誤 | ルール・スキル・フック・エージェント定義 |
| **メンテナンス** | その都度修正（属人的） | ドキュメント化・共有化 |
| **スケーラビリティ** | ユースケース数に比例 | 設計の再利用で軽減 |
| **例** | 「〇〇というトーンで答えて」 | ルール定義ファイル・スキル実装・フック設定 |
| **適用場面** | 単発の質問・チューニング | 継続的なシステム運用 |

---

### 4. 【ECC / Claude Code での実例】

**目的**: 抽象的な「コンテキスト設計」を、実装という形で具体化

**4-1. ルール（rules/）の役割と例**

- **役割**: 「AI に守らせる行動ルール・思考ルール」をテキスト化し、セッション前から固定
- **ECC の例**:
  - `/Users/yasuvel/ECC/rules/` 配下に複数のルールファイルが存在
  - 例：`coding-standards.md`（コーディング規約）、`output-writing.md`（出力形式の規約）
  - これらは「毎回プロンプトで説明する」のではなく、システムが読み込んで自動適用する
- **コード例**:
  ```markdown
  # 例：rules/coding-standards.md から抜粋
  ## 命名規則（クイックリファレンス）
  - 変数・関数: camelCase
  - クラス・型: PascalCase
  - 定数: UPPER_SNAKE_CASE
  - ファイル名（モジュール）: kebab-case
  ```
  - このファイル 1 つを指定するだけで、AI がコーディング規約を全て遵守する
  - 同じ規約をプロンプトで毎回説明する無駄がない

- **ポイント**: ルール = 「変わらない（変わりづらい）設計決定を、コード化して再利用する」

**4-2. スキル（skills/）の設計思想**

- **役割**: AI が「自分で行動できる能力」を設計・定義する
- **ECC の例**:
  - `/Users/yasuvel/ECC/skills/` に複数の skill が存在
  - 例：`/analyze <ECCのパス>`（指定パスを解析して出力に保存）
  - 例：`/question <疑問>`（疑問を 06_questions に追加）
  - これらは「毎回 AI が考える」のではなく、あらかじめ「この操作ができる」と定義
- **設計のポイント**: 
  - AI が「何をできるのか」を明確に定義 → セッション内での試行錯誤が減少
  - ユーザー側も「こういう操作が使える」と予測可能
  - 複数セッション・複数ユーザーで同じスキルが使える

- **コード例**（疑似コード）:
  ```javascript
  // skill定義イメージ
  skill: {
    name: "question",
    description: "疑問を06_questions/open/に追加",
    input: { type: "string", description: "疑問のテキスト" },
    handler: (text) => appendToFile("06_questions/open/...", text)
  }
  ```

- **ポイント**: スキル = 「AI ができる操作を、あらかじめ実装して提供する」

**4-3. フック（hooks/）による自動化**

- **役割**: 「特定のイベントが起こったとき、自動で実行される処理」を定義
- **ECC の例**:
  - `hooks/` 配下に複数の hook ファイルが存在
  - 例：`pre-commit.js`（コミット前に自動実行 → 形式チェック）
  - 例：`post-generate.js`（エージェント実行後に自動実行 → 結果ログ記録）
- **プロンプトとの違い**:
  - プロンプト：「〇〇のときは、こう考えて」（毎回 AI に指示）
  - フック：「〇〇のときは、[自動で]こう処理する」（システムが勝手に実行）
- **コード例**（イメージ）:
  ```javascript
  // pre-commit hook: コミット前に自動チェック
  module.exports = async () => {
    const status = await exec("git status");
    if (status.includes("sensitive-file")) {
      throw new Error("Sensitive file detected");
    }
  }
  ```
- **ポイント**: フック = 「ルーチン化できる処理を、AI ではなくシステムに任せる」

**4-4. エージェント間のコンテキスト伝播（Optional）**

- **役割**: 複数の AI エージェントが協力するとき、共通のコンテキストを維持する
- **ECC の例**:
  - 複数エージェント（Planner → Generator → Reviewer）が存在
  - 共通の「rules/」「CLAUDE.md」を参照
  - 結果として、エージェント A が作成した出力が、エージェント B で同じルールで検証される
- **効果**: 「AI 間の会話がスムーズ」「設計の一貫性が保証される」

## エッジケース・注意点

### 1. **実装例の引用が古くなる可能性**
- **対策**: ECC の現在の実装を参照日時付きで記載する。将来、ECC が変わっても記事が陳腐化しないよう、「2026年5月時点の」という限定を入れる

### 2. **「コンテキストエンジニアリング」という言葉の曖昧性**
- **対策**: 記事冒頭で「このテクストでいう『コンテキストエンジニアリング』とは…」と定義を明記。他の定義との混同を防ぐ

### 3. **プロンプトエンジニアリングを全否定に見える恐れ**
- **対策**: 「プロンプトとコンテキストは補完関係。どちらが上位ではなく、両方が必要」という立場を明記

### 4. **初心者と上級者の知識格差**
- **対策**: 
  - 初心者向け：「ルール = ファイルに書いた行動ルール」くらいの理解で十分
  - 上級者向け：セクション 4-4（エージェント伝播）で深掘りのヒント

### 5. **「実装のポイント」セクションが説教臭くなる恐れ**
- **対策**: 「これが正解」ではなく、「ECC の事例から見える共通パターン」として記述

### 6. **コード例が長すぎるとスクロール疲れ**
- **対策**: 1 つのコード例は 10～15 行程度で抑える。全部を示さず、「核となる部分」のみ
