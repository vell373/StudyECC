# コンテキストエンジニアリング入門：プロンプトより強力な設計技術

## 1. 導入：プロンプト疲れから脱出する方法

「同じプロンプトで複数回実行しても、結果がばらつく」「プロンプトを工夫してもスケールしない」。こうした悩みを持つ開発者は多くいます。

実は、その問題は**プロンプト最適化では解決できない別の層**の設計が欠けているからかもしれません。プロンプトエンジニアリングだけでは、システム全体の AI 活用を安定・スケールさせることはできません。そこで登場するのが「**コンテキストエンジニアリング**」という考え方です。

---

## 2. プロンプトエンジニアリングの限界

プロンプトエンジニアリングには、４つの大きな限界があります。

| 限界 | 説明 |
|------|------|
| **単一セッション内に留まる** | チャットウィンドウを閉じたら、その工夫は無効になります |
| **ユースケースごとにプロンプトが増殖** | 営業向け・エンジニア向けなど、異なるペルソナごとに管理が必要 |
| **メンテナンスが困難** | プロンプト記述の「ベストプラクティス」が散在し、統一できません |
| **スケーラビリティの欠如** | エージェント数が増えると、管理が爆発的に複雑化します |

新規開発者が加わったとき「どのプロンプトを使えばいいのか」わかりません。同じ内容を何度も説明しなければなりません。これは長期的には持続不可能です。

---

## 3. コンテキストエンジニアリングとは

### 定義

**「セッション前・セッション外で、AI の行動・判断の枠組みを事前に設計する技術」**です。

ここでいう「コンテキスト」は、再利用可能な、セッション独立の設計資産です。ルール・スキル・フック・エージェント構成など、複数の層を統合的に設計することで、プロンプト最適化の負担を大きく軽減します。

### プロンプト vs コンテキスト比較

| 観点 | プロンプト | コンテキスト |
|------|----------|-----------|
| **適用時期** | セッション内・リアルタイム | セッション前・事前設計 |
| **保持期間** | 単一セッションのみ | 複数セッション・長期維持 |
| **スコープ** | テキスト文字列の最適化 | システム全体の行動設計 |
| **ツール** | テキスト編集・試行錯誤 | ルール・スキル・フック・エージェント |
| **スケーラビリティ** | ユースケース数に比例 | 設計の再利用で軽減 |
| **適用場面** | 単発の質問・チューニング | 継続的なシステム運用 |

## 4. ECC・Claude Code での実例

### 4-1. ルール（rules/）：再利用可能な行動指針

**役割**：セッション前から固定された「AI に守らせる行動ルール」です。

ECC では `/Users/yasuvel/ECC/rules/` に複数のルールファイルが存在します。例えば `coding-style.md` は、コーディング規約を次のように定義しています：

```markdown
# rules/common/coding-style.md から抜粋

## Immutability (CRITICAL)

ALWAYS create new objects, NEVER mutate existing ones:

WRONG:  modify(original, field, value) → original が変更される
CORRECT: update(original, field, value) → 新しいコピーを返す

理由：イミュータブルなデータは隠れた副作用を防ぎ、
デバッグを容易にし、安全な並行処理を実現します。
```

このファイル 1 つを指定するだけで、AI がコーディング規約を自動適用。毎回プロンプトで説明する無駄がありません。

**ポイント**：ルール = 「変わらない設計決定をコード化して再利用する」

### 4-2. スキル（skills/）：AI の能力を事前定義

**役割**：「AI が自分で行動できる能力」をあらかじめ定義します。

ECC の `/question` スキルの設計イメージ：

```javascript
// skill定義の疑似コード
skill: {
  name: "question",
  description: "疑問を06_questions/open/に追加",
  input: { type: "string", description: "疑問のテキスト" },
  handler: (text) => {
    // スキル呼び出し時、自動的に疑問をファイルに保存
    appendToFile("06_questions/open/...", text)
  }
}
```

ユーザーが `/question 〇〇について知りたい` と実行すれば、システムが自動処理します。セッション内で AI が「どう記録しよう」と考える必要がありません。

**ポイント**：スキル = 「AI ができる操作をあらかじめ実装して提供する」

### 4-3. フック（hooks/）：自動化による運用効率化

**役割**：「特定のイベント時に自動実行される処理」を定義します。

ECC の `pre-commit` フック実装イメージ：

```javascript
// コミット前に自動チェック
module.exports = async () => {
  const status = await exec("git status");
  
  // 機密ファイルが含まれていないか確認
  if (status.includes("sensitive-file")) {
    throw new Error("Sensitive file detected");
  }
  // フック実行時、システムが自動的に中止
}
```

プロンプト（「気をつけてください」）ではなく、フック（「システムが自動ブロック」）で実装することで、人的ミスを完全に防ぎます。

**ポイント**：フック = 「ルーチン化できる処理を、AI ではなくシステムに任せる」

## 5. 実装のポイント

自分たちのシステムで実践するときの思考フロー：

1. **「何が変わらないか」を見つける**：毎回説明が必要なルール・ガイドラインを洗い出す
2. **「何ができるようにするか」を明確化**：AI が使える操作をあらかじめ定義する
3. **「何を自動化するか」を決める**：フックで自動化すべき処理を選定する
4. **複数エージェント間での一貫性を保つ**：共通のルール・設定を参照させる

---

## 6. まとめ

コンテキストエンジニアリングは「プロンプト vs コンテキスト」ではなく、**両者の組み合わせ**です。

プロンプトエンジニアリングはセッション内での即座な最適化に優れています。一方、コンテキストエンジニアリングは長期的・複数セッション間での安定性とスケーラビリティを提供します。

ECC のように「ルール・スキル・フック・エージェント構成」という層を意識することで、毎回プロンプトを工夫する疲弊から解放されます。システム全体を堅牢に保ちながら、AI の力を最大限に引き出してください。
