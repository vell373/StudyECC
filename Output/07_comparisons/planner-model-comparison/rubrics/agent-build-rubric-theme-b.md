# Theme B 採点ルーブリック: ドキュメント自動更新システム

**対象テーマ**: ドキュメント自動更新システム（マルチエージェント型ハーネス）  
**実装期間**: Phase 1（最小構成）+ Phase 2（拡張）  
**最終更新**: 2026-05-09

---

## 1. 概要（タスク説明）

このルーブリックは、Claude Code を用いて**複数の専門エージェントとハーネスを実装し、コード diff から自動的にドキュメント（README・API・CHANGELOG）を更新するシステム**を構築した場合の評価基準を定めます。

### 評価対象
- **API ドキュメント更新エージェント**（必須）
- **README 保守エージェント**（必須）
- **CHANGELOG / 使用例生成エージェント**（推奨）
- **統合ハーネス**（必須）
- **ドキュメント・設計説明**（必須）

### 評価の目的
- 要件定義に対する実装の完全性を測定
- エージェント設計の品質（粒度・独立性・協調性）を確認
- 実装の動作性・再現性を検証
- マルチエージェント設計の理解度と創造的な拡張を評価

## 2. 採点軸と基準

### 2.1 採点軸（4軸 × 10点満点）

| 採点軸 | 説明 |
|--------|------|
| **要件充足度（0-10点）** | Must-Have・Should-Have の実装率。タスク定義で求めている機能がどの程度実装されているか |
| **品質・構造（0-10点）** | エージェント設計の粒度・独立性・協調性。コード品質・ドキュメント完成度 |
| **完成度・動作性（0-10点）** | 実装されたシステムが実際に動作し、サンプル diff で期待通りの出力を生成できるか |
| **創造性・判断力（0-10点）** | Theme の要件を超えた工夫。エッジケース対応・ドキュメント設計・拡張性の工夫 |

### 2.2 評価テーブル（5段階評価）

#### 2.2.1 要件充足度（0-10点）

| スコア | 評価レベル | 基準 |
|--------|-----------|------|
| **9-10** | ✓ 完全実装 | Must-Have すべて + Should-Have の 80% 以上実装。Phase 1・2 が完了。受け入れ条件をすべて満たす |
| **7-8** | ◎ ほぼ完全 | Must-Have すべて実装。Should-Have の 50% 以上実装。Phase 1 は完全。Phase 2 が部分的 |
| **5-6** | ○ 部分実装 | Must-Have の 80% 実装（2体エージェント完成）。ハーネスが動作。Should-Have はほぼ未実装 |
| **3-4** | △ 最小限 | Must-Have の 50% 実装。エージェント 1-2 体は存在するが機能不完全。ハーネスが不完全 |
| **0-2** | × 不足 | Must-Have の 50% 未満。エージェント定義が不十分。動作確認ができない |

**具体的チェック項目**:
- [ ] API ドキュメント更新エージェントが実装され、関数シグネチャ・引数・戻り値の変更を検知できる
- [ ] README 保守エージェントが実装され、新機能・削除・変更を検知できる
- [ ] ハーネスが複数エージェントを呼び出し、提案をマージできる
- [ ] サンプル diff 2 件（新機能・バグ修正）で動作確認ができる
- [ ] 出力レポートが「更新ファイル」「before/after」「理由」を含む
- [ ] 3 体目エージェント（CHANGELOG or 使用例）が実装されている（Should-Have）
- [ ] コード変更属性の自動判定ロジックがある（Should-Have）
- [ ] エージェント間の矛盾検出ロジックがある（Should-Have）

**具体的チェック項目**:

## 3. 採点チェックリスト

### Phase 1 チェックリスト（Must-Have）

**エージェント実装**
- [ ] API ドキュメント更新エージェント（`.md` 定義）が存在する
  - [ ] 関数シグネチャの変更を検知できる
  - [ ] 引数の追加・削除・型変更を検知できる
  - [ ] 戻り値の変更を検知できる
- [ ] README 保守エージェント（`.md` 定義）が存在する
  - [ ] 新機能を検知できる
  - [ ] 削除機能を検知できる
  - [ ] 使い方変更を検知できる

**ハーネス実装**
- [ ] ハーネスが diff を入力として受け取れる
- [ ] ハーネスが複数エージェントを呼び出せる
- [ ] ハーネスが「更新が必要か」を判定できる
- [ ] ハーネスが「何を更新するか」を判定できる
- [ ] ハーネスが複数エージェント提案をマージできる
- [ ] ハーネスが重複を排除できる
- [ ] ハーネスが優先度付けできる

**入力・出力**
- [ ] unified diff 形式の入力に対応している
- [ ] 変更属性（新機能・バグ修正・廃止予告・セキュリティパッチ）が指定できる
- [ ] README.md・API.md のテンプレートが用意されている
- [ ] Markdown レポートで「更新ファイル」を明記している
- [ ] Markdown レポートで「before/after」を明記している
- [ ] Markdown レポートで「更新理由」を明記している

**動作確認**
- [ ] サンプル diff（新機能追加）でテスト実施
- [ ] サンプル diff（バグ修正）でテスト実施
- [ ] 出力レポートが期待通りか確認
- [ ] 「このドキュメント差分を実行してよいか」が判定できるか確認

### Phase 2 チェックリスト（Should-Have / Nice-to-Have）

**3 体目エージェント**
- [ ] CHANGELOG 自動生成エージェント または 使用例生成エージェント が実装されている
- [ ] セマンティックバージョニング ロジックが組み込まれている（CHANGELOG の場合）
- [ ] 新 API の使用例が自動生成される（使用例生成の場合）

**エージェント間協調**
- [ ] API 変更 × README 矛盾検出ロジックがある
- [ ] 「API.md に新機能あるのに README に記載がない」を検出できる
- [ ] CHANGELOG と README の粒度調整ロジックがある

**属性自動判定**
- [ ] diff から「新機能 / バグ修正 / 廃止予告 / セキュリティパッチ」を自動判定できる
- [ ] コード パターン（関数追加・条件分岐削除・セキュリティキーワード）から属性を推測できる

**ドキュメント**
- [ ] 各エージェントの判定ロジックが説明されている
- [ ] ハーネスの統合ロジック（矛盾検出・優先度付け）が説明されている
- [ ] diff → ドキュメント更新の処理フロー図が含まれている

**拡張機能（Nice-to-Have）**
- [ ] GitHub Actions 統合スクリプトが提供されている
- [ ] PR コメント or commit で提案ドキュメントが提示される
- [ ] バージョン管理（package.json / pyproject.toml 自動更新）が実装されている
- [ ] 多言語ドキュメント対応（日本語 + 英語）が実装されている
- [ ] ドキュメント品質チェック（可読性・正確性スコア）が実装されている

**エッジケース対応**
- [ ] 複数機能追加の場合に「機能 A」「機能 B」を分けて記載できる
- [ ] 古いドキュメントに対して「要レビュー」フラグを立てられる
- [ ] 廃止予告機能に対して「削除」ではなく「廃止予定」と記載できる
- [ ] 多言語混在（日本語 README + 英語 API.md）に対応できる

**再現性**
- [ ] サンプル diff + テンプレート で同じ更新提案が再現される
- [ ] 環境依存がない

## 4. 総合スコア算出式

```
総合スコア（0-100点）= 
  (要件充足度 × 0.35) + 
  (品質・構造 × 0.25) + 
  (完成度・動作性 × 0.25) + 
  (創造性・判断力 × 0.15)
```

| スコア帯 | 総合評価 | 合格判定 |
|---------|---------|--------|
| **90-100** | S: 優秀。Phase 1・2 をフルカバーし、拡張機能も実装。次段階へ推進可能 | **✓ 合格** |
| **80-89** | A: 良好。Must-Have + Should-Have の大部分実装。実用的 | **✓ 合格** |
| **70-79** | B: 標準。Must-Have をほぼ完全実装。Phase 2 は部分的 | **✓ 合格** |
| **60-69** | C: 最小限。Must-Have の主要部分が実装。改善の余地あり | **△ 保留** |
| **50-59** | D: 不十分。複数の要件が未実装または不完全 | **✗ 再提出** |
| **0-49** | F: 不合格。要件の 50% 未満が実装 | **✗ 再提出** |

## 6. 補足

### 6.1 動作確認の実施方法

1. **サンプル diff の用意**
   - 新機能追加の diff（関数追加）
   - バグ修正の diff（既存関数の修正）
   - 廃止予告の diff（関数の deprecated マーク追加）

2. **ハーネスの実行**
   - diff + 既存テンプレート（README.md・API.md）を入力
   - 出力レポートを確認

3. **検証項目**
   - 生成ドキュメント案の正確性
   - 矛盾検出の精度
   - before/after の明確性

### 6.2 ドキュメント評価の観点

- **判定ロジック説明**: 各エージェントが「どういう条件で更新が必要か」を明記しているか
- **フロー図**: テキストだけでなく図示があるか（Markdown フロー図など）
- **エッジケース**: 複数機能・古いドキュメント・廃止予告・多言語への対応を明記しているか

### 6.3 再現性の確認方法

- 同じサンプル diff + テンプレートで 2 回実行
- 出力が完全に一致するか確認
- 環境変数・外部 API に依存していないか確認
