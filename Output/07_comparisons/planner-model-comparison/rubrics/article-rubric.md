# 記事執筆競技 採点ルーブリック

## 概要

3つのテーマ（A：コードレビューの自動化、B：コンテキストエンジニアリング入門、C：マルチエージェント設計パターン）に対する記事執筆の品質を統一的に評価するための採点基準。

各テーマの記事を以下4軸で評価し、総合スコアを算出します。

---

## 採点軸と評価項目

### 1. 要件充足度（Must-Have 要件の完全性）

記事タスクで指定された**必須要素**をすべて満たしているかを評価します。

| 項目 | 配点 | 評価基準 |
|------|------|--------|
| **文字数** | 20点 | 2000～3000文字の範囲内に収まっているか<br>- 範囲内：20点 / 1999字以下または3001字以上：0点 |
| **構成（セクション数と流れ）** | 20点 | 指定された構成を満たし、論理的な流れになっているか<br>- 要求構成を完全に実装：20点 / 大部分実装：15点 / 部分的：10点 / 不足：0点 |
| **コード例・実装例の数と質** | 20点 | 要求数以上のコード例・実装例を含み、実行可能・参考になるか<br>- テーマA・C：最低2つ以上 / テーマB：最低2つ以上の具体例<br>- 実装例が完全で動作レベルの詳細：20点 / 有用だが不完全：15点 / 簡易的：10点 / なし：0点 |
| **実践性・応用可能性** | 20点 | 読者が記事内容を自分のプロジェクトに適用できるレベルか<br>- 実装手順が明確で応用可能：20点 / おおむね可能：15点 / 概念的で抽象的：10点 / 理論のみ：0点 |
| **参照・比較の充実度** | 20点 | 要求された比較視点や参照の具体性<br>- テーマA：マルチエージェント設計の複数観点を比較 / テーマB：プロンプト vs コンテキスト比較表あり / テーマC：3パターン以上の比較表<br>- すべて実装：20点 / 大部分：15点 / 部分的：10点 / なし：0点 |

**小計：100点**

---

### 2. 品質・構造（論理性と説得力）

記事全体の品質、構成の論理性、説得力を評価します。

| 項目 | 配点 | 評価基準 |
|------|------|--------|
| **論理構成の明確さ** | 20点 | 導入 → 問題提示 → 解決策 → 実装 → まとめの流れが明確か<br>- 非常に明確で追随しやすい：20点 / 概ね明確：15点 / やや曖昧：10点 / 不明確：0点 |
| **説得力・根拠の妥当性** | 20点 | 主張に根拠があり、読者を納得させるか<br>- 根拠が十分で説得的：20点 / 根拠はあるが弱い：15点 / 例示がのみ：10点 / 根拠なし：0点 |
| **対象読者への適切性** | 15点 | 想定読者レベルに合わせた説明難度か<br>- 初心者向けはわかりやすく、専門的な記事は適切に詳細。<br>- 適切な難度：15点 / おおむね適切：12点 / やや難しい or 簡単すぎる：8点 / 不適切：0点 |
| **Markdown 形式の正確さ** | 15点 | 見出し（H1から）、テーブル、リスト、コードブロックの形式が正確か<br>- すべて正確：15点 / 軽微な誤り：12点 / いくつか誤り：8点 / 形式が乱れている：0点 |
| **用語・表現の統一性** | 15点 | 専門用語の定義が一貫し、表現がぶれていないか<br>- 統一的で自然な流れ：15点 / おおむね統一：12点 / 部分的な表現ぶれ：8点 / ぶれが目立つ：0点 |
| **図解・テキストダイアグラム** | 15点 | 必要に応じてダイアグラムが含まれるか（テーマC：加点項目）<br>- テーマC でダイアグラム含む：15点 / 部分的：10点 / テーマA・B でテーブル充実：10点 / なし：0点 |

**小計：100点**

---

### 3. 完成度・動作性（実装の具体性と検証可能性）

実装例やコード例が実際に動作する品質か、細部まで完成しているかを評価します。

| 項目 | 配点 | 評価基準 |
|------|------|--------|
| **コード例の実行可能性** | 20点 | 記載されたコード・設定ファイルが実際に動作するか<br>- 実行可能で動作確認済みレベル：20点 / 実行可能と判断される：15点 / 軽微な修正で動作：10点 / 動作不確実：0点 |
| **設定ファイル・ファイル構成の完全性** | 20点 | ファイルパス、設定項目、必須パラメータが完全か<br>- 完全で即座に使用可能：20点 / おおむね完全：15点 / 主要部のみ：10点 / 不完全：0点 |
| **エージェント間連携・データフローの明確性** | 20点 | エージェント間のデータ受け渡し、実行順序が明確か<br>- 実装フローチャート or 詳細な説明あり：20点 / 大部分説明：15点 / 概略のみ：10点 / なし：0点 |
| **停止条件・エラーハンドリングの記載** | 20点 | ループ終了条件、エラー時の処理が明記されているか<br>- 詳細に記載：20点 / 基本的に記載：15点 / 触れている：10点 / なし：0点 |
| **実装上の注意点・制約の明記** | 20点 | 実装時の落とし穴、制限事項が明確に記載されているか<br>- 詳細に警告・注意点あり：20点 / 主要な注意点記載：15点 / 簡易記載：10点 / なし：0点 |

**小計：100点**

---

### 4. 創造性・判断力（新しい視点と洞察）

単なる情報整理ではなく、著者独自の分析、批判的思考、オリジナルな視点があるかを評価します。

| 項目 | 配点 | 評価基準 |
|------|------|--------|
| **オリジナルな分析・視点** | 20点 | ECC / Claude Code の実装を単に説明するのではなく、なぜそうなっているのか、独自の解釈があるか<br>- 深い分析と独自視点あり：20点 / 視点はあるが部分的：15点 / 説明が多く分析少ない：10点 / 単純な説明のみ：0点 |
| **メリット・デメリットのバランス** | 20点 | 提案する手法・パターンのメリットだけでなく、デメリット・制限を公正に述べているか<br>- メリット・デメリット両者記載、バランス良好：20点 / 大部分バランス：15点 / メリット偏向気味：10点 / 一方的：0点 |
| **応用例・発展性の提案** | 15点 | 記事の内容を他分野・他状況に応用する示唆があるか<br>- 複数の応用例・発展方向を提案：15点 / 応用例あり：12点 / 言及のみ：8点 / なし：0点 |
| **根拠のない主張の回避** | 15点 | 確信がない場合に「おそらく」「検討の余地」と明記されているか<br>- 不確実な主張は必ず明記：15点 / ほぼ明記：12点 / 時々不明記：8点 / 根拠なき主張が多い：0点 |
| **プライバシー・倫理への配慮** | 15点 | 個人・企業情報が隠匿されているか、倫理的な配慮があるか<br>- 十分な配慮：15点 / 基本的に配慮：12点 / 軽微な懸念：8点 / 配慮不足：0点 |
| **読者のアクション喚起** | 15点 | 読者が「自分たちも試してみたい」「判断できるようになった」と思えるか<br>- 強く喚起できる内容：15点 / 読んだ後の行動イメージあり：12点 / 理解は得られるが行動は曖昧：8点 / 行動喚起なし：0点 |

**小計：100点**

## 採点方法

### ステップ1：採点軸ごとの小計計算

各軸（要件充足度、品質・構造、完成度・動作性、創造性・判断力）の配点を合計。

```
各軸の小計 = 各評価項目の配点合計
```

### ステップ2：総合スコア計算

```
総合スコア = （要件充足度 × 0.35）+ （品質・構造 × 0.30）+ （完成度・動作性 × 0.20）+ （創造性・判断力 × 0.15）
```

重み付け理由：
- **要件充足度 35%**：Must-Have 要件を満たさないと論外。記事タスクの基礎
- **品質・構造 30%**：論理性と説得力が読者体験を左右。重要な第二要素
- **完成度・動作性 20%**：実装例の質が記事の実用性を決定
- **創造性・判断力 15%**：差別化。単なる説明ではなく、分析と洞察

### ステップ3：評価区分

| スコア範囲 | 評価 | 説明 |
|-----------|------|------|
| 90～100 | **S（優秀）** | 全ての要件を完全に満たし、創造性・分析も優れた記事。そのまま公開可能。 |
| 80～89 | **A（良好）** | 要件をほぼ満たし、品質も高い。軽微な修正で公開可能。 |
| 70～79 | **B（及第）** | 要件を満たし、基本的に良い。改善提案あれば、さらに質向上。 |
| 60～69 | **C（改善要）** | 要件は満たすが、品質・完成度に改善余地あり。再執筆・加筆推奨。 |
| 50～59 | **D（不合格寸前）** | 要件の一部不足。本質的な改善・追加が必要。 |
| ～49 | **F（不合格）** | 要件の複数不足。再提出推奨。 |

## 評価時の注意点

1. **テーマごとの観点の違いを認識**：テーマ A・C はコード例が中心、テーマ B は比較・説得が中心。適切に重み付けを調整。

2. **根拠の明記**：「なぜこのスコアか」を理由とともに示す。改善点の提案を含める。

3. **部分点の活用**：二者択一ではなく、段階的な配点を活用。例えば「実装例が2つ中1つ」なら15点など。

4. **記事の文脈を尊重**：著者の主張や視点が明確であれば、評価も寛容に。ただし根拠のない主張は減点。

5. **再評価の道**：スコア60未満の場合は、具体的な改善項目リストを提供。著者が改版したら再評価。

## まとめ

このルーブリックは、3つのテーマの記事を統一的に評価しながら、各テーマの特性（A：実装例、B：説得力、C：パターン比較）を尊重する設計になっています。

評価者は採点基準を明確に理解し、必ず根拠とともに点数を付けることで、著者への有用なフィードバックが実現できます。
