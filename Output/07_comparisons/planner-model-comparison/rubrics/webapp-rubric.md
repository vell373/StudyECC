# ウェブアプリ開発競技 採点ルーブリック

**作成日**: 2026-05-09
**対象テーマ**: マークダウンライブプレビューエディター / ポモドーロタイマー + タスク管理 / シンプルな習慣トラッカー

---

## 採点構成

本ルーブリックは、4つの採点軸で総合評価を行います。各軸は100点満点で配点されます。

| 採点軸 | 満点 | 説明 |
|-------|------|------|
| 1. 要件充足度 | 100 | Must-Have/Should-Have 要件の実装率と正確性 |
| 2. 品質・構造 | 100 | コード品質、設計、UI/UX の質 |
| 3. 完成度・動作性 | 100 | 動作安定性、ブラウザ互換性、ストレージ機能の信頼性 |
| 4. 創造性・判断力 | 100 | Nice-to-Have 実装、工夫・最適化、ドキュメント品質 |

**総合スコア** = (軸1 + 軸2 + 軸3 + 軸4) / 4

### 1-2. Should-Have 実装（20点）

**評価基準**: 「あると良い」機能の実装数と実装品質

| テーマ別の Should-Have 例 | 配点 | 評価基準 |
|------------------------|------|---------|
| テーマA: テーマ切り替え/フォント設定/スタイリング/印刷機能（各5点） | 20 | 実装されたものの数と完成度 |
| テーマB: 優先度設定/本日の統計/ダークモード/プログレスバー（各5点） | 20 | 実装されたものの数と完成度 |
| テーマC: 習慣削除/カラーコード/週ビュー/統計サマリー（各5点） | 20 | 実装されたものの数と完成度 |

**配点例**:
- 0個実装 → 0点
- 1～2個実装 → 5点
- 3個実装 → 10点
- 4個実装 → 20点（満点）

### 1-3. 要件の正確性・受け入れ条件の達成度（20点）

**評価基準**: Must-Have の「受け入れ条件」を厳密に満たしているか

| 項目 | 配点 | 評価基準 |
|------|------|---------|
| 受け入れ条件の達成度 | 20 | 要件ごとに定義された受け入れ条件をすべて満たす |

**例**:
- テーマA「プレビューが約100ms以内に更新」を満たすか
- テーマB「セッションカウント4のタイミングで長休憩提案」が動作するか
- テーマC「記録なし日の連続カウント=0」が正確か

## 採点軸2: 品質・構造（100点）

### 2-1. コード品質（40点）

**評価基準**: 関数分割、命名規則、可読性、保守性

| 項目 | 配点 | 評価基準 |
|------|------|---------|
| 関数分割・モジュール化 | 15 | 関数が単一責任を持つ、循環依存がない、1関数が30～50行以内 |
| 命名の一貫性 | 10 | 変数・関数名が意味的で一貫性がある（camelCase等） |
| コメント・ドキュメント | 10 | 複雑な処理に説明コメント、README が分かりやすい |
| DRY 原則・再利用性 | 5 | 重複コードがなく、共通処理が関数化されている |
| **小計** | **40** | |

**配点ガイド**:
- 優: 15/15 - よく設計された関数群
- 良: 12/15 - 軽微な改善の余地あり
- 可: 8/15 - 分割が不十分、重複あり
- 不可: 5点以下 - ほぼ全体が1～2個の巨大関数

### 2-2. UI/UX・見やすさ（35点）

**評価基準**: レイアウト、色使い、操作性、レスポンシブ性

| 項目 | 配点 | 評価基準 |
|------|------|---------|
| レイアウト・配置 | 10 | 左右2ペイン or 上下分割が明確、スペース配分が自然 |
| 色使い・コントラスト | 10 | 視認性が高い、アクセシビリティを考慮（WCAG 準拠） |
| ボタン・インタラクティブ要素 | 10 | マウスホバー、クリック時の視覚的フィードバックがある |
| レスポンシブ性 | 5 | PC/タブレット/モバイル で見やすい（最小スマートフォンサイズ対応） |
| **小計** | **35** | |

**配点ガイド**:
- 優: 34-35点 - プロダクションレベルのUI
- 良: 28-33点 - 実用的で見やすい
- 可: 20-27点 - 基本的だが使える
- 不可: 20点以下 - 見づらい、分かりにくい

### 2-3. 状態管理・設計の明確性（25点）

**評価基準**: グローバル状態の管理、状態更新の一貫性

| 項目 | 配点 | 評価基準 |

## 採点チェックリスト

### テーマA（マークダウンエディター）確認項目

**Must-Have の受け入れ条件**
- [ ] テキストエリアで改行が反映される
- [ ] プレビューが100ms以内に更新される（ラグ測定: DevTools で検証）
- [ ] H1～H6 見出しが正しく HTML に変換される
- [ ] 太字 (`**text**`), イタリック (`*text*`) が正しく `<strong>`, `<em>` になる
- [ ] 順序なしリスト、順序付きリストが正しく表示される
- [ ] コードブロック（````code````) が正しく `<pre><code>` で囲まれる
- [ ] リンク `[text](url)` が正しい `<a href>` になる
- [ ] ページリロード後、テキストが復元される

**動作テスト**
- [ ] Chrome / Firefox / Safari / Edge で動作確認
- [ ] 複数段落、複雑な記法を含むテキストで動作確認
- [ ] localStorage がフル時の動作確認

**品質確認**
- [ ] JavaScript ファイルが300行を超える場合、分割されている
- [ ] 関数が50行を超えない
- [ ] 変数名が camelCase で統一
- [ ] DevTools の Console にエラーがない

### テーマB（ポモドーロタイマー）確認項目

**Must-Have の受け入れ条件**
- [ ] 開始ボタンで 25分カウントダウンが始まる
- [ ] 一時停止ボタンで停止し、再開ボタンで再開される
- [ ] リセットボタンで 25:00 に戻る
- [ ] 25分終了時に通知（音/アラート）が出る
- [ ] 5分休憩が正確に 5分でカウントダウン
- [ ] 5分終了時に通知が出る
- [ ] セッションカウントが正確にインクリメント（25分完了時に +1）
- [ ] 4セッション完了後に「15分の長休憩」が提案される
- [ ] タスク追加がボタンクリックで動作
- [ ] チェックボックスで取り消し線が表示される
- [ ] 削除ボタンがタスクを削除
- [ ] ページリロード後、タスク一覧とセッションカウントが復元される

**動作テスト**
- [ ] 25分を短縮時間（例: 5秒）でテスト
- [ ] 複数ブラウザで同期が必要か確認
- [ ] localStorage 上限到達時の処理確認

**品質確認**
- [ ] タイマーロジックが独立した関数（startTimer, pauseTimer等）
- [ ] タスク状態と タイマー状態が分離されている
- [ ] イベントリスナーが適切に登録・削除される（メモリリーク防止）

### テーマC（習慣トラッカー）確認項目

**Must-Have の受け入れ条件**
- [ ] テキスト入力 + 「追加」ボタンで習慣がリストに加わる
- [ ] 最大10個に達した場合、警告が出る
- [ ] 習慣ごとに「今日実施」ボタンがある

## 採点の実施方法

1. **要件充足度（100点）** を最初に確認
   - Must-Have が全て実装されているか → 進めるか中止か判断
   - Should-Have の実装数をカウント

2. **品質・構造（100点）** をコードレビューで評価
   - GitHub での diff / 拡張機能でコード確認
   - 各項目にスコア付け

3. **完成度・動作性（100点）** を実機テストで評価
   - 各ブラウザで実際に操作
   - チェックリストを進める

4. **創造性・判断力（100点）** を総合的に評価
   - Nice-to-Have の実装確認
   - パフォーマンス・ドキュメント・アクセシビリティを検査

5. **総合スコア** = (軸1 + 軸2 + 軸3 + 軸4) / 4

## 備考

- このルーブリックは、3テーマ全てに共通する採点軸を基盤としながら、各テーマ固有の要件に対応したチェックリストを並列して用意しています
- 実装の「深さ」（Nice-to-Have の充実度）と「確実性」（バグがないか）の両方を評価します
- ブラウザ互換性やアクセシビリティは、実務で重要視される項目として配点を高めています
- 採点者は、このドキュメントを参照しながら、各アプリを系統的に評価することができます
