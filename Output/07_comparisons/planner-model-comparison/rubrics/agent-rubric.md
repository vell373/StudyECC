# エージェント構築競技 採点ルーブリック

## 概要

このルーブリックは、マルチエージェントシステムの構築タスクを評価するための採点基準です。
4つの採点軸に基づいて、段階的に品質を評価します。

---

## 採点軸（4つの観点）

### 1. **要件充足度** — 指定された要件をすべて実装しているか

実装の完成度と仕様への準拠を評価します。

| 評価等級 | 基準 |
|---------|------|
| **A（優秀）** | ・Must-Have を 100% 実装<br>・Should-Have の 80% 以上を実装<br>・Nice-to-Have も複数実装<br>・エージェント数が要件を超えている |
| **B（良好）** | ・Must-Have を 100% 実装<br>・Should-Have の 50% 以上を実装<br>・出力ファイル構成が要件通り |
| **C（合格）** | ・Must-Have をすべて実装<br>・エージェント数が最低要件を満たす<br>・基本的なオーケストレーションが動作可能 |
| **D（部分的）** | ・Must-Have の 70% 以上を実装<br>・核となるエージェント 2～3 体は実装<br>・オーケストレーションが不完全 |
| **F（要改善）** | ・Must-Have が 70% 未満<br>・エージェント数が要件に満たない<br>・実装が不十分で動作不可能 |

**チェックリスト（テーマA: PR レビュー）:**
- [ ] 3体以上の専門レビューエージェント実装
- [ ] 各エージェントが異なる観点（セキュリティ、パフォーマンス、テスト等）を担当
- [ ] 各エージェント `.md` 形式で ECC フロントマター完備
- [ ] オーケストレーションコマンド実装
- [ ] Git diff 入力対応
- [ ] 統合レポート生成機能

**チェックリスト（テーマB: ドキュメント更新）:**
- [ ] 変更検知エージェント実装
- [ ] ドキュメント生成エージェント実装（最低2種類）
- [ ] Git diff またはコミットメッセージ入力対応
- [ ] オーケストレーションコマンド実装
- [ ] 構造化フォーマット出力（JSON または Markdown）
- [ ] パッチ形式のドキュメント出力

**チェックリスト（テーマC: 技術選定）:**
- [ ] 要件分析エージェント実装
- [ ] 技術提案エージェント実装
- [ ] 比較評価エージェント実装
- [ ] 統合コマンド実装
- [ ] 複数軸スコアリング（最低5軸）
- [ ] 最終推奨技術スタック出力

---

### 2. **品質・構造** — エージェント間連携と設計が明確か

設計の一貫性、モジュール化、拡張性を評価します。

| 評価等級 | 基準 |
|---------|------|
| **A（優秀）** | ・エージェント間の責務分離が明確<br>・疎結合設計で、各エージェント独立動作可能<br>・データフロー図などの設計図が完備<br>・エージェント数を増やしやすい拡張性<br>・エラーハンドリング・フォールバック実装 |
| **B（良好）** | ・各エージェントの役割が明確に記述<br>・エージェント間の入出力インタフェースが定義<br>・最低限の拡張性がある<br>・設計説明が詳しい |
| **C（合格）** | ・各エージェントの役割が分かる<br>・エージェント間の連携が機能する<br>・基本的な設計説明がある<br>・拡張性は限定的だが実装可能 |
| **D（部分的）** | ・エージェント間の責務がやや曖昧<br>・連携に無駄・重複がある<br>・設計説明が最小限<br>・拡張は困難 |
| **F（要改善）** | ・エージェント間の責務が不明確<br>・連携に矛盾・エラーがある<br>・設計が秩序なく見える<br>・拡張性がない |

**チェックリスト（構造品質）:**
- [ ] 各エージェントの責務（何をするか）が 1 文で説明可能
- [ ] エージェント間のデータ受け渡しが明確（ファイル/API/プロンプト など）
- [ ] 循環依存がない
- [ ] 各エージェントが独立してテスト可能
- [ ] コマンド/オーケストレーターの実行フローが図示或いは明記
- [ ] 設計の「なぜそうしたか」が説明されている
- [ ] 新しいレビュー観点・ドキュメント種別・評価軸を追加しやすい
- [ ] エラーケース（入力エラー、出力エラー）の処理

---

### 3. **完成度・動作性** — システムが実装されており、実行可能か

実装の完成度、実行可能性、ドキュメント品質を評価します。

| 評価等級 | 基準 |
|---------|------|
| **A（優秀）** | ・すべてのコンポーネントが完全実装<br>・サンプル実行例が複数含まれている<br>・README が詳細で、すぐに実行可能<br>・エラーハンドリング完備<br>・設定ファイルなど実運用向け |
| **B（良好）** | ・主要コンポーネント完全実装<br>・サンプル実行例が提示されている<br>・README が詳しい<br>・基本動作確認可能<br>・運用への工夫がある |
| **C（合格）** | ・最低限のコンポーネント実装完了<br>・実行手順が記述されている<br>・README が要件を説明している<br>・基本動作は確認可能<br>・サンプルが最小限 |
| **D（部分的）** | ・コンポーネントが 80% 程度実装<br>・実行手順が不完全<br>・README が簡潔<br>・実行に試行錯誤が必要<br>・サンプルが不足 |
| **F（要改善）** | ・実装が未完了部分が多い<br>・実行手順が不明確<br>・README が不十分<br>・実行できない/試行錯誤が多い |

**チェックリスト（完成度・動作性）:**
- [ ] 全エージェント `.md` ファイルが完成（フロントマター + 指示）
- [ ] オーケストレーションコマンド完成
- [ ] README.md が存在し、以下を含む：
  - [ ] システム構成図（テキストまたは ASCII アート）
  - [ ] 各エージェントの役割説明
  - [ ] 実行手順（クイックスタート）
  - [ ] サンプル入力・出力
  - [ ] 設定方法（あれば）
  - [ ] トラブルシューティング
- [ ] サンプル出力ファイルが存在（example-outputs/ に複数例）
- [ ] 実行手順に沿って、実際に動作可能
- [ ] エラーメッセージが分かりやすい
- [ ] ファイル構成が要求ファイル構成と一致

---

### 4. **創造性・判断力** — 要件を超えた工夫、実装の工夫

要件の理解、実装の工夫、実用性を評価します。

| 評価等級 | 基準 |
|---------|------|
| **A（優秀）** | ・要件を明確に理解し、独自の工夫が多数<br>・Should-Have / Nice-to-Have を多く実装<br>・実装に創意工夫（例：キャッシング、最適化）<br>・実用レベルのシステム<br>・将来の改善提案も示唆される |
| **B（良好）** | ・要件を正しく理解<br>・Should-Have の複数を実装<br>・実装に工夫がある<br>・実用的なシステム<br>・拡張ポイントが明確 |
| **C（合格）** | ・要件を正しく理解<br>・基本要件を満たすレベルの工夫<br>・実装に最低限の工夫<br>・基本的に実用的 |
| **D（部分的）** | ・要件の理解が部分的<br>・工夫が少ない<br>・実装が機械的<br>・実用性に課題 |
| **F（要改善）** | ・要件の理解が不完全<br>・工夫がほとんどない<br>・実装が不完全で実用性低い |

**チェックリスト（創造性・判断力）:**
- [ ] Should-Have の機能が複数実装されている
- [ ] Nice-to-Have の機能が実装されている
- [ ] エージェント間の連携に工夫がある（例：キャッシング、並列実行）
- [ ] エラーハンドリングが充実
- [ ] ログ・デバッグ機能がある
- [ ] パフォーマンス最適化が考慮されている
- [ ] セキュリティ対策が考慮されている
- [ ] 将来の拡張・改善提案がコメント/ドキュメントに記載
- [ ] 類似プロジェクトへの応用性が高い

---

## 全体スコアリング

各採点軸を A～F で評価し、総合スコアを算出します。

### 総合等級の判定

| 判定 | 条件 |
|------|------|
| **合格** | 4軸すべてが **C 以上** |
| **優秀** | 4軸すべてが **B 以上** かつ、**A が 2軸以上** |
| **秀逸** | 4軸すべてが **A** |
| **要改善** | いずれかの軸が **D 以下** |

### スコア換算表（参考値）

| 等級 | 点数 |
|------|------|
| A | 90～100 点 |
| B | 75～89 点 |
| C | 60～74 点 |
| D | 40～59 点 |
| F | 0～39 点 |

**総合点 = (要件充足度 + 品質・構造 + 完成度・動作性 + 創造性・判断力) ÷ 4 の平均**

## テーマ別評価注釈

### テーマA（PR レビュー自動化ハーネス）の特有評価点

**品質・構造で重視:**
- 各レビューエージェントの責務分離の明確さ
- 実際の GitHub CI/CD 統合を想定した設計
- 異なるレビュー観点が衝突しない仕様

**完成度・動作性で重視:**
- 実際の Git diff（サンプル）で動作確認可能か
- 統合レポートの読みやすさ、実用性
- チームが実装に導入しやすいドキュメント

**創造性で重視:**
- false positive（誤検知）の除外機能
- レビュー指摘の優先度分類
- 自動修正サジェスション

---

### テーマB（ドキュメント自動更新システム）の特有評価点

**要件充足度で重視:**
- 2 種類以上のドキュメント対応
- Git diff 入力への対応
- パッチ形式の出力品質

**品質・構造で重視:**
- ドキュメント種別ごとのエージェント設計（スケーラビリティ）
- 既存ドキュメントスタイルの保持メカニズム

**完成度・動作性で重視:**
- README、API ドキュメント、設定ガイド等の実際の例示
- 生成されたドキュメント部分の正確性と一貫性

**創造性で重視:**
- 自動コミット機能
- 複数言語対応
- スタイル検証

---

### テーマC（技術選定エージェント群）の特有評価点

**要件充足度で重視:**
- 3 つのエージェント（分析、提案、評価）の実装
- 複数軸スコアリングの実装（最低 5 軸）
- 評価の一貫性と再現性

**品質・構造で重視:**
- 要件分析 → 提案 → 評価のデータフローの明確さ
- スコアリングロジックの透明性
- エージェント間の情報受け渡し仕様

**完成度・動作性で重視:**
- 実際のプロジェクト例での実行結果
- 推奨技術スタックの説得力
- 各軸の評価根拠の明記

**創造性で重視:**
- リスク評価エージェント
- コスト試算
- チームマッチング評価
- トレードオフ分析

---

## 採点手順

1. **要件充足度**: Must-Have チェックリストから始める
2. **品質・構造**: エージェント設計図、責務分離、データフローを確認
3. **完成度・動作性**: README の詳細さ、サンプル出力、実行可能性を確認
4. **創造性・判断力**: Should-Have / Nice-to-Have の実装、工夫を確認

各軸で段階的に判定し、最後に総合等級を決定します。

## 補足：採点時の着眼点

### 「実装」の定義

- **エージェント**: フロントマター（name, description, tools, model）+ 指示文 を含む `.md` ファイル
- **コマンド/オーケストレーター**: エージェント群を統合実行する `.md` または スクリプト
- **README**: システム構成、使い方、サンプルを含むマークダウン

### 「実行可能」の定義

- 記述されたすべてのステップを実行できる
- 所定の入力に対して、所定のフォーマットで出力が得られる
- ファイル（`.md`, サンプルデータ）が実際に存在する

### 「実用的」の定義

- チーム/プロジェクトで実際に使える品質
- ドキュメントが明確で、新人でも使える
- エラーメッセージ、ログが親切
- カスタマイズ・拡張が容易
