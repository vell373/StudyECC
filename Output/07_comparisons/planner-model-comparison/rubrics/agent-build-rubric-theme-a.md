# 採点ルーブリック: PR レビュー自動化ハーネス（agent-build / Theme A）

## 概要

このルーブリックは「PR レビュー自動化ハーネス」の実装品質を評価するための4軸採点基準。
マルチエージェント協調設計・Claude Code Agent SDK の実践的応用を評価対象とする。

- **満点**: 40点（各軸10点 × 4）
- **合格水準**: 28点以上（70%）
- **評価対象**: agent-build タスクの成果物全体（`.md` エージェント定義ファイル群・ハーネス・ドキュメント・サンプルコード）

---

## 採点軸と基準

| 軸 | 配点 | 0–3点 | 4–5点 | 6–7点 | 8–9点 | 10点 |
|---|---|---|---|---|---|---|
| **要件充足度** | 0–10 | Must-Have の2体エージェントが未実装、またはハーネスが存在しない | 2体エージェントとハーネスは存在するが、出力フォーマット（重要度別・カテゴリ別）が不完全 | Must-Have をほぼ達成。Should-Have の一部（3体目エージェントまたはドキュメント）が実装されている | Must-Have を完全達成 + Should-Have の大部分（3体以上・協調ロジック・サンプルコード・ドキュメント）を達成 | Must-Have + Should-Have を全て達成し、Nice-to-Have（GitHub Actions 連携・複数言語対応等）の1つ以上を実装 |
| **品質・構造** | 0–10 | `.md` 定義ファイルの構造が統一されておらず、責務が曖昧。エージェント間の役割分担が判別不能 | 各エージェントに責務の記述はあるが、ハーネスとの境界が不明確。命名・ファイル構成に一貫性がない | 各エージェントの責務が明確に分離されており、ハーネスが呼び出し・集約・フォーマットの役割を担っている。ただし可読性の改善余地あり | 責務分離が明確で、エージェント定義が200行以内に収まり、Claude Code から直接実行可能な形式。新規エージェント追加が容易な構造 | 上記に加え、拡張性のある設計（プラグイン式追加・標準化されたインターフェース）が実現されており、保守・拡張コストが最小化されている |
| **完成度・動作性** | 0–10 | サンプルコードが未提供、またはハーネスが実際に動作しない（エージェント呼び出しエラー等）。既知の脆弱性を検出できない | サンプルコード1件のみ。既知の脆弱性・品質問題の一部のみ検出できる。出力レポートが不完全 | サンプルコード2件（脆弱性含む・高品質）でテスト済み。既知問題の検出はほぼ正確だが、改善提案の質または出力フォーマットが不安定 | サンプルの既知問題を全件検出（検出率100%）。Markdown レポートが「Critical / High / Medium / Low」「Security / Quality 等のカテゴリ」で正確に整理され、実装者向けアクションが明確 | 上記に加え、エッジケース（大規模差分・矛盾提案・言語不明・差分なし）への対処が実装・検証されており、再現性が担保されている |
| **創造性・判断力** | 0–10 | 仕様の曖昧な部分（エッジケース・矛盾提案処理等）が全く考慮されていない。機械的な最小実装のみ | エッジケースの一部に言及があるが、実装・回避策が不十分。レポートが単純なリスト出力に留まる | エッジケースへの対処方針がドキュメントまたは実装に反映されている。矛盾提案を「トレードオフ分析」として提示するなど、読者視点の工夫がある | 仕様の曖昧な部分を自律的に解釈・補完し、実装に反映している。レポートが実装者にとって「次のアクション」を明示する実用的な品質を持つ。エージェント間協調（矛盾検出等）が機能する | 上記に加え、フィードバックループ・性能分析・多言語対応等の先進機能で独自性を発揮。設計上のトレードオフが明文化されており、ユーザーが意思決定しやすいアウトプットになっている |

## 採点チェックリスト

### 軸1: 要件充足度

**Must-Have（各1点、計5点相当）**
- [ ] セキュリティエージェントが `.md` 形式で定義されている
- [ ] コード品質エージェントが `.md` 形式で定義されている
- [ ] ハーネスが複数エージェントを呼び出し、結果を統合している
- [ ] 入力フォーマット（unified diff または JSON + ファイル情報）をハーネスが受け取れる
- [ ] 出力レポートに「Security Issues」「Quality Findings」「Recommendations」セクションが含まれ、重要度別・カテゴリ別に整理されている

**Should-Have（各0.5点、計2.5点相当）**
- [ ] 3体目の専門エージェント（テスト or アーキテクチャ）が実装されている
- [ ] エージェント間の矛盾検出・協調ロジックが実装されている
- [ ] 受け入れテスト用サンプルコードが2件以上用意されている
- [ ] 各エージェントの責務・ハーネスの動作フロー・呼び出し手順のドキュメントがある

**Nice-to-Have（ボーナス加点用）**
- [ ] GitHub Actions 連携スクリプトが存在する
- [ ] 複数言語（Python / JavaScript / Go 等）の差分を処理できる
- [ ] フィードバックループの仕組みが実装されている
- [ ] 各エージェントの実行時間・API 呼び出し回数がレポートに記録される

---

### 軸2: 品質・構造

- [ ] 各エージェント定義ファイルが200行以内に収まっている
- [ ] エージェントの役割（セキュリティ / 品質 / テスト etc.）がファイル名・ファイル冒頭で明示されている
- [ ] ハーネスの責務が「呼び出し・集約・フォーマット」の3つに明確に分離されている
- [ ] 新規エージェントを追加する際の手順・インターフェース仕様が定義されている
- [ ] ファイル構成・命名規則が一貫している（例: `security-agent.md`, `quality-agent.md`）
- [ ] エージェント出力の標準化スキーマ（フォーマット統一）が明示されている

---

### 軸3: 完成度・動作性

- [ ] サンプルコード1：SQLインジェクション・XSS 等の既知脆弱性が含まれる差分を用意している
- [ ] サンプルコード2：ネストが深い・関数が長い等の品質問題が含まれる差分を用意している
- [ ] サンプルコード1 の全脆弱性がレポートで検出されている（検出率100%）
- [ ] サンプルコード2 の全品質問題がレポートで検出されている（検出率100%）
- [ ] 出力 Markdown レポートが「Critical / High / Medium / Low」で整理されている
- [ ] 各指摘に「改善提案」が付いており、実装者が次のアクションを取れる
- [ ] 同じ入力で同じレポートが再現される（再現性確認）
- [ ] エッジケース（大規模差分・矛盾提案・言語不明・差分なし）の少なくとも1つへの対処が確認できる

---

### 軸4: 創造性・判断力

- [ ] 矛盾する提案（例：暗号化必須 vs 暗号化は遅い）を「トレードオフ分析」として提示している
- [ ] 言語判定失敗時の明示的なエラーレポートが実装されている
- [ ] 差分が微小（1行変更）の場合に「軽微」と判定・報告できる
- [ ] レポートが技術的な羅列でなく「実装者が読む文書」として書かれている
- [ ] 仕様の曖昧な部分（テーマに明示されていない判断）に対し、設計根拠が記述されている
- [ ] エージェント間協調（矛盾検出・統合指摘）が単純な結合以上の付加価値を提供している
