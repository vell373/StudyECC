## 3. 設計上の主要判断

### 3.1 エージェント間の分割（責務分担）

**判断**: API・README を完全に独立したエージェントに分割

**理由**:
- API の関数シグネチャ解析と README のセクション管理は異なるロジック
- 独立することで再利用性・テスト性向上
- ハーネスが「複数エージェント」の価値を実現

**代替案考慮**:
- ❌ 単一エージェント：機能が肥大化、保守困難
- ❌ 3+ エージェント（Phase 1では）：スコープ外張り

### 3.2 並列実行 vs 順序実行

**判断**: 並列実行をデフォルト

**理由**:
- API 提案と README 提案は互いに独立
- 処理時間短縮（max(t_api, t_readme) vs t_api + t_readme）
- 矛盾検出は「最終段階」で一括処理

**実装**: `harness.md` ステップ3 に `Promise.all()` パターンを記載

### 3.3 矛盾検出の「レベル」分け

**判断**: ERROR / WARNING / INFO に分類

**根拠**:
- **ERROR**: ドキュメント整合性の破綻（API削除 ∧ README記載残存）
  - 処理停止が妥当
- **WARNING**: 改善推奨だが動作可能（API新機能 ∧ README未更新）
  - 人間レビュー推奨
- **INFO**: 参考情報のみ

**実装**: `harness.md` ステップ4「矛盾検出」で定義

### 3.4 廃止予告の扱い

**判断**: 削除ではなく「廃止予告」セクションへ移動

**根拠**:
- `changeAttribute === "deprecation"` で機能完全削除でなく段階的廃止を表現
- 既存ユーザーへの移行期間を明確化
- API.md / README.md で統一的に扱う

**実装**: 
- `api-doc-agent.md` 判定ロジック参照
- `sample-execution.md` シーン3 で実装例

### 3.5 diff 解析の「正規表現」設計

**判断**: JSDoc + TypeScript 両対応の正規表現を定義

**パターン例**:
```
新規関数:  /^\+\s*(?:async\s+)?function\s+(\w+)\s*\((.*?)\)/
削除関数:  /^-\s*(?:async\s+)?function\s+(\w+)\s*\((.*?)\)/
```

**理由**: 
- JavaScript / TypeScript / Python 混在プロジェクトに対応
- 簡潔かつ実用的な抽出精度

**制限事項**:
- クラスメソッドは別パターンが必要（Phase 2）
- デコレータ付き関数は簡略化（Python @decorator など）

### 3.6 テンプレート形式の「柔軟性」

**判断**: テンプレートは「ガイド」であり、必須フォーマットではない

**理由**:
- 実際の README.md / API.md は構成がまちまち
- エージェントは「提案」を生成し、統合はハーネスで行う
- ユーザーが既存の構造に合わせてカスタマイズ可能

**実装**: `sample-execution.md` で複数テンプレート例を提示

## 7. 実装の工夫・創意工夫

### 7.1 矛盾検出の「優先度付け」

従来: ドキュメント差分を「実行可能 / 不可能」のみで判定

**実装**: ERROR / WARNING / INFO で段階的管理
- ERROR: 処理停止が必要（整合性破綻）
- WARNING: 改善推奨だが動作可能
- INFO: 参考情報

**効果**: ユーザーが「部分的に実行する」選択肢を獲得

### 7.2 変更属性の「自動推測」

従来: ユーザーが属性を手作業で指定

**実装**: `harness.md` ステップ2 で diff パターン + コミットメッセージから自動推測
- パターン例: `+ function` → new-feature（確信度0.9）
- 推測と指定が矛盾した場合は警告

**効果**: ユーザー入力エラーを検出・軽減

### 7.3 エージェント間の「データ受け渡しスキーマ」定義

従来: 各エージェント出力形式が未定義

**実装**: `harness.md` 「エージェント間データ受け渡しの仕様」で統一スキーマ定義
```json
{
  "agentName": "...",
  "changes": [ { "type", "section", "before", "after", "reason" } ],
  "confidence": 0.9,
  "requiresHumanReview": false,
  "warnings": [ { "level", "message" } ]
}
```

**効果**: ハーネスがエージェント出力を確実に処理可能

### 7.4 廃止予告機能の「段階的対応」

従来: 削除 = 記述削除のみ

**実装**: changeAttribute=deprecation で「廃止予告」セクション作成
- API.md に「[廃止予告]」マーク + v.X.X で削除予定
- README.md に廃止予告 + 代替案記載
- CHANGELOG で "Deprecated" セクション（Phase 2）

**効果**: ユーザーへ移行期間を明確化
