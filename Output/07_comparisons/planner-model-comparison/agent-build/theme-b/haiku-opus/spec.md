# Theme B: ドキュメント自動更新システム - 仕様書

## 概要

コード diff を入力し、複数の専門エージェント（API ドキュメント・README・CHANGELOG）が協調してドキュメントを自動更新するマルチエージェント型ハーネス。各エージェントの更新提案をマージし、「このドキュメント差分を実行してよいか」を判定できる Markdown レポートを生成する。

---

## 要件リスト

### Must-Have

1. **API ドキュメント更新エージェント（`.md` 形式）**
   - 関数シグネチャ・引数・戻り値の変更を検知
   - API.md の該当セクションを自動更新
   
2. **README 保守エージェント（`.md` 形式）**
   - 新機能・削除機能・使い方変更を検知
   - README.md の該当セクションを自動更新

3. **ハーネス（統合エージェント）**
   - unified diff 形式のコード変更を入力
   - 複数エージェントに diff を分配、「更新必要か」を判定
   - 複数提案をマージし、重複排除・優先度付け
   - 「更新ファイル」「旧テキスト→新テキスト」「更新理由」を含む Markdown レポートを出力

4. **テスト対応**
   - サンプル diff 2 件（新機能・バグ修正）で正確に更新提案を生成
   - 生成レポートが「そのまま実行可能」な完成度

### Should-Have

1. **3体目エージェント（CHANGELOG or 使用例生成）**
   - セマンティックバージョニング対応の CHANGELOG.md 自動更新
   - または新 API の使用例を examples/ に自動生成

2. **エージェント間協調ロジック**
   - 矛盾検出：API.md に新機能 → README に記載なし
   - 粒度調整：CHANGELOG は詳細、README は概要

3. **属性自動判定**
   - diff から「新機能 / バグ修正 / 廃止予告 / セキュリティパッチ」を自動判定
   - 判定結果をエージェントに通知

4. **ドキュメント**
   - 各エージェントの判定ロジック説明
   - diff → ドキュメント更新フロー図
   - エッジケース対応（廃止予告・言語混在）の説明

### Nice-to-Have

1. GitHub Actions 統合（PR での自動実行・コメント提示）
2. バージョン管理（CHANGELOG → package.json / pyproject.toml 更新）
3. 多言語対応（日本語 + 英語同時更新）
4. ドキュメント品質スコア化（低スコアは「要レビュー」フラグ）

## 実装フェーズ

### Phase 1: 最小構成（Must-Have）

**目標**: API・README エージェント + ハーネスの基本動作確認

1. API ドキュメント更新エージェント作成
2. README 保守エージェント作成
3. ハーネス実装（diff 入力 → エージェント呼び出し → Markdown 出力）
4. サンプル diff 2 件でテスト（新機能 1 件・バグ修正 1 件）
5. 出力レポート検証

**完成条件**:
- 各エージェントが独立して機能する
- 更新提案が正確（誤検知なし、見落としなし）
- レポートが「実装者の判定材料」になる粒度

### Phase 2: 拡張（Should-Have / Nice-to-Have）

**目標**: 3体エージェント・矛盾検出・属性判定の実装

1. 3体目エージェント追加（CHANGELOG or 使用例）
2. 属性自動判定ロジック実装
3. エージェント間矛盾検出ロジック
4. GitHub Actions or 多言語対応（いずれか）
5. ドキュメント・フロー図作成

**完成条件**:
- 複数エージェントが協調動作する
- セマンティックバージョニング or 多言語に対応
- 矛盾検出が「要レビュー」フラグを正確に立てる

## 品質基準

✓ API・README エージェントが独立して機能する（Phase 1）
✓ ハーネスが複数提案をマージ・重複排除できる
✓ サンプル diff 3 パターン（新機能・バグ修正・廃止予告）すべてに対応
✓ 生成レポートが before/after を明確に表示
✓ 矛盾検出ロジックが機能（API 追加 → README 未更新を検出）
✓ 「このドキュメント差分を実行してよいか」を判定できる完成度

## エッジケース対応

| ケース | 対応方法 |
|--------|--------|
| **複数機能追加** | ハーネスが「機能 A」「機能 B」に分割して記載 |
| **既存ドキュメント不正確** | 「要レビュー」フラグを立て、人間確認を促す |
| **廃止予告機能** | 削除でなく「廃止予告」に書き換え、段階を明記 |
| **言語混在** | 言語自動判定・各言語別エージェント起動（Phase 2） |
| **エージェント間矛盾** | ハーネスが「API 新機能 ↔ README 未更新」を検出・フラグ |

## 実装ガイダンス

### エージェント設計（各 200 行以下）

1. **入力**: diff テキスト + 既存ドキュメント + 属性（新機能/バグ修正/廃止予告/セキュリティパッチ）
2. **処理**: テンプレートマッチング or パターン認識で変更箇所を抽出
3. **出力**: JSON （`{ "file": "API.md", "old": "...", "new": "...", "reason": "..." }`）

### ハーネス設計

1. **Phase 1**: 単純な分配・マージ（複数エージェント呼び出し → JSON 結果を Markdown に変換）
2. **Phase 2**: 矛盾検出ロジック（API 変更 × README チェック）

### テスト対象

- 新機能追加（関数・パラメータ）
- バグ修正（既存機能の内部変更）
- 廃止予告（deprecated マーク）
- セキュリティパッチ（重大度・対応方法）
