# Theme B: ドキュメント自動更新システム

## テーマ概要

コード変更を検知し、自動的に README.md・API ドキュメント・CHANGELOG を更新する**マルチエージェント型ドキュメント自動化ハーネス**を設計・構築する。

新機能追加・API 変更・廃止予告・セキュリティパッチなど、複数のコード変更パターンに対応し、各パターン別に最適なドキュメント更新を実施するエージェント群の実装を通じて、Claude Code での「推論 → 実装」の自動化を習得する。

---

## 要件定義

### Must-Have（必須）

1. **最低2体の専門エージェント実装**
   - API ドキュメント更新エージェント：関数シグネチャ・引数・戻り値の変更を検知し、API.md を自動更新
   - README 保守エージェント：新機能・削除機能・使い方変更を検知し、README.md の該当セクションを自動更新
   - 両者は `.md` 形式の Claude Code Agent として定義する

2. **ハーネス（統合エージェント）の実装**
   - コード diff を入力として受け取る
   - 複数の専門エージェントに diff を渡し、「更新が必要か」「必要なら何を」を判定させる
   - 複数エージェントからの更新提案をマージし、最終的なドキュメント更新差分を生成
   - 「更新対象ファイル」「変更行」「旧テキスト→新テキスト」を明記した Markdown レポートを出力

3. **入力フォーマット**
   - コード diff（unified diff 形式）
   - 変更の属性（「新機能」「バグ修正」「廃止予告」「セキュリティパッチ」のいずれか）
   - 既存のドキュメントテンプレート（README.md・API.md）

4. **出力フォーマット**
   - Markdown レポート
   - 「更新ファイル」「更新内容（before / after）」「更新理由」を含む
   - 「このドキュメント差分を実行してよいか」を判定できる情報を含む

---

### Should-Have（強く推奨）

1. **3体目の専門エージェント追加**
   - CHANGELOG 自動生成エージェント：セマンティックバージョニングに基づき CHANGELOG.md を自動更新
   - または「使用例 / チュートリアル更新エージェント」：新 API の使用例を examples/ に自動生成

2. **ハーネス内でのエージェント間協調**
   - API 変更 × README の矛盾検出（「API.md に新機能あるのに README に記載がない」）
   - CHANGELOG と README の粒度調整（「CHANGELOG は詳細、README は概要」）

3. **コード変更の「属性」自動判定**
   - diff から「新機能 / バグ修正 / 廃止予告 / セキュリティパッチ」を自動判定
   - → エージェントに「このコード変更は XXX です」と伝える

4. **ドキュメント**
   - 各エージェントの判定ロジック
   - ハーネスの統合ロジック（矛盾検出・優先度付け）
   - diff → ドキュメント更新の動作フロー図

---

### Nice-to-Have（あると嬉しい）

1. **GitHub Actions 統合**
   - PR が開かれた時点でハーネスを自動実行
   - 提案ドキュメント更新を PR コメント or commit で提示

2. **バージョン管理**
   - CHANGELOG から自動的に `version` 定義を抽出し、`package.json` / `pyproject.toml` を更新

3. **多言語ドキュメント対応**
   - README (日本語) + README.en (英語) を同時更新する協調ロジック

4. **ドキュメント品質チェック**
   - 生成されたドキュメント案に対し「可読性・正確性」をスコア化
   - 低スコア案は「要レビュー」として人間確認を促す

---

## 技術スタック（推奨）

- **言語**: Markdown（`.md` Agent 定義） + テンプレート言語（Jinja2 等の利用も可）
- **エージェント数**: 最低2体（API・README）、推奨3体（+CHANGELOG or +使用例）
- **ハーネス実装**: Claude Code Agent SDK（`.md` 形式）
- **入力サンプル**: 実際のコード diff（JavaScript / Python の新機能実装例）
- **出力形式**: Markdown 差分レポート（ファイル名・before-after を明示）

---

## 実装フェーズ計画

### Phase 1: 最小構成（Must-Have）

1. API ドキュメント更新エージェント実装
2. README 保守エージェント実装
3. ハーネス実装（diff 入力 → 複数エージェント呼び出し → 統合出力）
4. サンプルコード diff 2 件でテスト
5. 「更新提案」レポート出力確認

**完成条件**: API・README それぞれの更新提案を正確に生成。矛盾がなく、実装者が「このドキュメント差分を実行してよいか」を判定できること

### Phase 2: 拡張（Should-Have / Nice-to-Have）

1. 3体目エージェント追加（CHANGELOG or 使用例生成）
2. コード変更属性の自動判定ロジック
3. エージェント間の矛盾検出・調整ロジック
4. GitHub Actions スクリプト（or 多言語対応）

**完成条件**: 3 体以上が協調し、セマンティックバージョニング・多言語・属性判定を実装した高度なドキュメント更新を実施できること

---

## 品質基準（これが揃えば完成）

✓ API・README・（CHANGELOG or 使用例）の複数エージェントが独立して機能する  
✓ ハーネスが複数エージェントの提案をマージし、重複排除・優先度付けができる  
✓ サンプルコード（新機能・バグ修正・廃止予告）の 3 パターンすべてに対応  
✓ 生成ドキュメント案が「この変更で実装完了か」を判定できる粒度である  
✓ 矛盾検出ロジック（API 追加 → README 未更新など）が機能  
✓ ドキュメント更新の「before / after」が明確で、レビュー不可能な曖昧さがない  

---

## エッジケース

1. **diff に複数の機能追加が含まれた場合**
   - 「新機能 A」「新機能 B」を分けて記載すべき
   - → ハーネスが「複数変更セットの検出」を実施

2. **既存ドキュメントが古い / 不正確な場合**
   - 「このドキュメントは古い。上書きしてよいか」と判定不可
   - → ハーネスが「要レビュー」フラグを立てる

3. **廃止予告機能の場合**
   - 「削除」ではなく「廃止予告」なので、README は「～ は廃止予定です」に書き換える必要がある
   - → エージェントが廃止予告ロジックを持つ

4. **ドキュメント言語が混在する場合**
   - 日本語 README + 英語 API.md など
   - → 言語の自動判定・各言語別エージェントの起動

---

## 受け入れ条件（Acceptance Criteria）

1. **機能完全性**
   - 最低2体のエージェントが定義ファイル（`.md`）として作成される
   - ハーネスがコード diff を入力として受け取り、複数エージェントを呼び出せる
   - サンプル diff 2 件（新機能・バグ修正）でテスト、「更新提案レポート」を出力できる

2. **品質要件**
   - エージェント定義ファイルが 200 行を超えない
   - 生成ドキュメント案が「そのまま実行可能」な完成度
   - 矛盾検出（API 変更 × README 未更新など）が機能すること

3. **ドキュメント**
   - 各エージェントの判定ロジックが説明されていること
   - 「diff → 更新提案」の処理フロー図を含むこと
   - 「廃止予告・複数言語」などのエッジケース対応が明記されていること

4. **再現性**
   - サンプル diff + テンプレート を使って、同じ更新提案が再現される
   - 環境依存がないこと
