# Theme A: PR レビュー自動化ハーネス

## テーマ概要

コード差分を受け取り、複数の観点（セキュリティ・パフォーマンス・可読性・テスト適否・アーキテクチャ）からレビューを自動実施する**エージェント群**を設計・構築する。

単一エージェントではなく、複数のレビュー専門エージェント（セキュリティエージェント、パフォーマンスエージェント等）が協調する**マルチエージェントハーネス**を実装することで、Claude Code の Agent SDK の実際的な応用を習得する。

---

## 要件定義

### Must-Have（必須）

1. **最低2体の専門エージェント実装**
   - セキュリティエージェント：SQL インジェクション・XSS・認証・暗号化等の脆弱性を検査
   - コード品質エージェント：可読性・保守性・命名規則・関数サイズ等を検査
   - 両者は `.md` 形式の Claude Code Agent として定義する

2. **ハーネス（統合エージェント）の実装**
   - 複数の専門エージェントを呼び出し、結果を集約して統一レポートを生成する
   - 各エージェントの出力を標準化し、重複排除・優先度付けを実施
   - 最終的に「Security Issues」「Quality Findings」「Recommendations」セクションを含むレポートを出力

3. **入力フォーマット**
   - コード差分（unified diff 形式または JSON で変更箇所を記述）
   - ファイル情報（ファイル名・言語・プロジェクトコンテキスト）
   - をハーネスが受け取れること

4. **出力フォーマット**
   - Markdown レポート
   - 「重要度」別に整理（Critical / High / Medium / Low）
   - 「カテゴリ」別に整理（Security / Performance / Readability / Testing / Architecture）
   - 改善提案を含む

---

### Should-Have（強く推奨）

1. **3体目の専門エージェント追加**
   - テスト適否エージェント：テストカバレッジ・エッジケース対応・テスト可能性設計を検査
   - または「アーキテクチャエージェント」：設計パターン・依存関係・レイヤー分離を検査

2. **ハーネス内での エージェント間協調**
   - セキュリティ×パフォーマンスの矛盾検出（例：キャッシング vs タイムリーネス）
   - セキュリティ×テストの統合指摘（例：暗号化テスト困難性）

3. **受け入れテスト用サンプルコード**
   - セキュリティ脆弱性を含むサンプルコード（SQLインジェクション例等）
   - 品質問題を含むサンプルコード（ネストが深い、関数が長い等）
   - ハーネスが正確に検出できることを実証

4. **ドキュメント**
   - 各エージェントの責務説明
   - ハーネスの動作フロー
   - ハーネス外部から呼び出す際の使い方

---

### Nice-to-Have（あると嬉しい）

1. **GitHub Actions 統合スクリプト**
   - PR が開かれた時点で自動的にハーネスを呼び出す例（shell script）

2. **複数言語対応**
   - Python / JavaScript / Go など複数言語の差分を同時に処理可能

3. **フィードバックループ**
   - ハーネスの提案に対し「実施した / スキップした / 異議あり」の反応を入力
   - 後続の同じ項目の提案ではそれを学習・加味する

4. **Performance 分析**
   - 各エージェントの実行時間・API 呼び出し回数をレポートに追加

---

## 技術スタック（推奨）

- **言語**: Markdown（`.md` Agent 定義） + 必要に応じて補助スクリプト（shell / Python）
- **エージェント数**: 最低2体（セキュリティ・品質）、推奨3体（+テスト or +アーキテクチャ）
- **ハーネス実装**: Claude Code Agent SDK（`.md` 形式の Planner / Generator / Orchestrator）
- **入力サンプル**: 実際の GitHub PR diff（JavaScript / Python の既知脆弱性パターン）
- **出力形式**: Markdown レポート（統一スキーマ）

---

## 実装フェーズ計画

### Phase 1: 最小構成（Must-Have）

1. セキュリティエージェント実装（`.md`）
2. コード品質エージェント実装（`.md`）
3. ハーネス実装（複数エージェント呼び出し・結果統合）
4. サンプルコード2件での実装テスト
5. Markdown レポート出力確認

**完成条件**: 「Security Issues」「Quality Findings」の 2 セクションを含むレポートを生成し、サンプルの脆弱性・品質問題を正確に検出

### Phase 2: 拡張（Should-Have / Nice-to-Have）

1. 3体目エージェント追加（テスト or アーキテクチャ）
2. エージェント間の協調ロジック実装
3. GitHub Actions スクリプト作成（OR 複数言語対応）
4. パフォーマンス測定・ドキュメント充実

**完成条件**: 3体以上のエージェントが協調してレポートを生成し、矛盾検出・提案の高度な統合ができること

---

## 品質基準（これが揃えば完成）

✓ セキュリティ・コード品質・（テスト or アーキテクチャ）の 2〜3 体のエージェントが独立して機能する  
✓ ハーネスが複数エージェントの結果を正確に統合し、重要度順・カテゴリ別に整理できる  
✓ サンプルコードの既知脆弱性（SQLインジェクション・XSS・長関数 等）をすべて検出  
✓ エージェント定義が `.md` 形式で Claude Code から直接実行可能  
✓ Markdown レポートが「改善提案」を含み、実装者が次のアクションを明確に把握できる  
✓ ドキュメント（各エージェント責務・ハーネスの動作フロー）が充実し、新たなレビュー観点の追加が容易

## エッジケース

1. **ファイルサイズが大きい場合**
   - 1000行超えのファイル差分：エージェントがコンテキスト不足にならないよう工夫
   - → 差分を複数チャンクに分割し逐次処理する OR ハーネスが優先順位付けして重要部分のみ詳査

2. **矛盾する提案が生じた場合**
   - セキュリティ「暗号化必須」 vs パフォーマンス「暗号化は遅い」
   - → ハーネスが「トレードオフ分析」として両方を報告

3. **言語不明な diff を受け取った場合**
   - ファイル拡張子が不正 / Markdown が混在
   - → ハーネスが「言語判定失敗」と明示的に報告

4. **差分がない or 差分が微小（1行変更）**
   - → ハーネスが「変更なし / 軽微」と判定・報告

---

## 受け入れ条件（Acceptance Criteria）

1. **機能完全性**
   - 最低2体のエージェントが定義ファイル（`.md`）として作成される
   - ハーネスが各エージェントを正確に呼び出し、結果を統合できる
   - サンプルコード2件（脆弱性含む・高品質）でテストし、検出率 100% を達成

2. **品質要件**
   - エージェント定義ファイルが 200行を超えない（簡潔さ）
   - ハーネスの責務が明確（呼び出し・集約・フォーマット）
   - Markdown レポートが「実装者向けアクション」として読める

3. **ドキュメント**
   - 各エージェントの検査項目が 1 ファイル 1 ページで説明できる
   - ハーネスの動作フロー図（テキスト図）を含む
   - 「新規エージェント追加手順」が明記されていること

4. **再現性**
   - サンプルコード + ハーネスを実行し、同じレポートが再現される
   - 環境依存（APIキー等）がないこと
