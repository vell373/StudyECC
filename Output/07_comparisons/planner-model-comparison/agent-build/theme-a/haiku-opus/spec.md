# PR レビュー自動化ハーネス — 仕様書（Haiku版）

## 概要

複数の専門エージェント（セキュリティ・品質・テスト）が協調して PR 差分を評価し、重要度別・カテゴリ別に整理した Markdown レポートを生成する Agent SDK アプリケーション。最小構成として 2 体のエージェント + ハーネスで実装し、段階的に拡張する設計。

---

## 要件サマリー

### Must-Have（Phase 1: 最小構成）

| # | 要件 | 詳細 |
|---|------|------|
| 1 | **セキュリティエージェント** | SQLi・XSS・認証・暗号化脆弱性を検査（`.md` 形式） |
| 2 | **品質エージェント** | 可読性・命名・関数サイズ・ネスト深度を検査（`.md` 形式） |
| 3 | **ハーネス実装** | 2 体以上の専門エージェントを呼び出し、結果を統合・レポート出力 |
| 4 | **入力フォーマット** | diff 形式のコード差分 + ファイル情報（名前・言語） |
| 5 | **出力フォーマット** | Markdown レポート（「Security Issues」「Quality Findings」セクション） |
| 6 | **重要度・カテゴリ整理** | 検出結果を Critical/High/Medium/Low ×「Security / Quality / Performance」で分類 |

### Should-Have（Phase 2: 拡張）

| # | 要件 | 詳細 |
|---|------|------|
| 7 | **3 体目エージェント** | テストまたはアーキテクチャ観点を追加 |
| 8 | **エージェント間協調** | 矛盾検出（セキュリティ ↔ パフォーマンスなど） |
| 9 | **受け入れテスト** | 既知脆弱性・品質問題を含むサンプルコード 3～5 件 |
| 10 | **ドキュメント** | 各エージェント責務・ハーネスフロー・新規エージェント追加手順 |

### Nice-to-Have（Phase 3: 最適化）

| # | 要件 | 詳細 |
|---|------|------|
| 11 | **CI/CD 統合** | GitHub Actions で PR オープン時に自動実行する shell スクリプト |
| 12 | **複数言語対応** | Python / JavaScript / Go の差分を統一レポートで処理 |
| 13 | **フィードバックループ** | 提案の採用状況を学習、同じパターンの再提案を調整 |

**合計**: Must-Have 6項目、Should-Have 4項目、Nice-to-Have 3項目

## 技術スタック

| 項目 | 選定 | 理由 |
|-----|------|------|
| **エージェント定義** | Markdown（`.md`） | Claude Code Agent 形式、バージョン管理容易、可読性高い |
| **ハーネス実装** | Claude Agent SDK + 補助スクリプト | 複数エージェント呼び出し・結果統合、Markdown 出力ネイティブ |
| **入力形式** | unified diff OR JSON | diff は標準的、JSON は構造化処理に有利 → 当初 diff, Phase 2 で JSON 併用 |
| **出力形式** | Markdown（単一ファイル） | 実装者向け、diffs 記録可能、git 親和的 |
| **言語サポート** | Phase 1: JS/Python、Phase 2: Go/Ruby 追加 | 一般的、脆弱性パターン豊富 |

## 実装フェーズ計画

### Phase 1: 最小構成（Must-Have 6項目）

**目標**: セキュリティ × 品質の 2 体エージェント + ハーネスを実装し、サンプルコード 2 件での検出実証

**実装タスク**:

1. **エージェント定義**（各 100～150 行）
   - `security-agent.md`: SQL インジェクション・XSS・認証フロー・暗号化を検査ルール化
   - `quality-agent.md`: 可読性（命名・関数長）・ネスト・重複コードを検査ルール化

2. **ハーネス実装**（200～250 行）
   - 入力パースする（diff 形式 → 言語・変更箇所の抽出）
   - 各エージェントを順序実行（コンテキストパス）
   - 結果を重要度・カテゴリ別にマージ
   - Markdown レポートを生成・stdout 出力

3. **サンプルコード**（2 件）
   - `sample-vulnerable.js`: SQLi, XSS, 長関数, ネスト深い例
   - `sample-good.py`: ベストプラクティス コード

4. **完成条件**
   - サンプルコードの既知脆弱性 100% 検出
   - ハーネス実行時間 < 30 秒
   - レポートに「改善提案」が含まれ、実装者がアクション明確

**完了予定**: 1～2 日

---

### Phase 2: 拡張（Should-Have 4項目）

**目標**: 3 体目エージェント + 協調ロジック + テストスイート

**実装タスク**:

1. **テストエージェント追加**
   - `testing-agent.md`: テストカバレッジ・エッジケース・テスト可能性設計を検査

2. **エージェント間協調**
   - ハーネス内に「矛盾検出ロジック」を追加
     - 「セキュリティは暗号化要求」「パフォーマンスは高速化要求」の場合、「トレードオフ分析」を報告
   - 優先度の自動調整（Security > Testing > Quality > Performance）

3. **受け入れテスト（3～5 件）**
   - SQLi パターン
   - XSS パターン
   - 関数サイズ超過 + ネスト深い
   - テストカバレッジ不足
   - 矛盾する設計提案

4. **ドキュメント**
   - `README-agents.md`: 各エージェントの検査項目・出力例
   - `README-harness.md`: ハーネスの動作フロー・新規エージェント追加手順
   - `TROUBLESHOOTING.md`: 一般的なエラーと対処法

**完了予定**: 1 週間

---

### Phase 3: 最適化（Nice-to-Have 3項目）

**実装タスク**:

1. **GitHub Actions 統合**: `pr-review.yml` スクリプト
2. **複数言語対応**: Go / Ruby パターン追加
3. **フィードバックループ** (オプション): SQLite で提案履歴を記録

**完了予定**: 追加 3～5 日

## エッジケース対応戦略

| ケース | 対応方法 |
|--------|---------|
| **ファイル > 1000 行** | 差分を 500 行ごとにチャンク分割、各チャンクを順序実行、結果統合 |
| **矛盾する提案** | ハーネスが「トレードオフ分析」として両方を報告、優先度を明記 |
| **言語不明な diff** | ファイル拡張子から言語判定、失敗時は「言語判定失敗」と明示的に報告 |
| **変更なし or 微小変更** | ハーネスが「変更なし」または「軽微」と判定、処理短縮 |

---

## 実装優先度（Haiku 効率性重視版）

1. **セキュリティエージェント**: 最も価値が高く、パターン化容易
2. **品質エージェント**: 次点、ツール的価値大
3. **ハーネス**: コアロジック、最後に統合
4. **テストサンプル**: 早期からテスト駆動で検証

**推奨初期スコープ**: セキュリティ + 品質 + ハーネス で Phase 1 完成
→ テストエージェント追加で Phase 2 へ進む
