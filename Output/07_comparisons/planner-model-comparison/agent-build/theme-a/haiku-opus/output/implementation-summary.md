## 設計上の判断

### 判断 1: エージェント定義形式（`.md`）

**背景**: 仕様書では「Markdown（`.md`）形式」を推奨

**実装判断**: 
- ✓ 採用理由:
  - Claude Code Agent SDK のネイティブ形式
  - ファイルの可読性・保守性が高い
  - バージョン管理（git diff）が直感的
  - 拡張ルール追加時のテンプレート明確

**注意点**: 
- 実行時には `.md` → JSON への変換が必要（ハーネス内で実施）
- Python / JavaScript 等の実装言語との互換性は確保（pseudocode で示）

---

### 判断 2: 重要度ラベル（4 段階）

**背景**: 仕様「Critical / High / Medium / Low」を採用

**実装判断**:
- Security Issues:
  - Critical: SQLi, 認証欠落, コマンドインジェクション
  - High: XSS, 暗号化弱点, パストラバーサル, 不安全な通信
  
- Quality Findings:
  - Medium: 関数長超過, ネスト深度, 命名違反, 重複コード
  - Low: コメント不足, パラメータ数過多, 行長超過

**根拠**: Security > Quality（優先度順）で配置。実装者が優先すべき修正を明確化

---

### 判断 3: エージェント間協調（矛盾検出）

**背景**: 仕様「矛盾検出・トレードオフ分析」を要件化

**実装判断**:
- ハーネスに `detect_conflicts()` ロジックを実装
- 矛盾例: 「暗号化必須（Security）」 vs 「高速化優先（Quality）」
- 出力: "Tradeoff Analysis" セクション（解決策付き）
- ハーネス pseudocode に実装例を明記

**実装の曖昧さ**: 
- 「矛盾の定義」は具体例から推測（仕様書に明示なし）
- Security と Quality の優先度は Security > Quality で統一
- Phase 2 実装時に実際のテストケースで精度向上

---

### 判断 4: エッジケース対応の実装戦略

**背景**: 仕様「ファイルサイズ > 1000 行」「言語判定失敗」等を先読み対応

**実装判断**:

| ケース | 対応方法 |
|--------|---------|
| ファイル > 1000 行 | 500 行ごとにチャンク分割、順序実行、結果統合 |
| 言語判定失敗 | 拡張子確認 → 失敗時「言語判定失敗」と明示報告 |
| 矛盾する提案 | Tradeoff Analysis セクションで優先度を明記 |
| 差分が微小（1-3行） | 軽微な変更と判定、検査の軽減 |
| 差分なし | 処理スキップ、「変更なし」を報告 |

**実装詳細**: `pr-review-harness.md` の "エッジケース対応" セクションで Markdown レポート例を明記。実装時には各ロジックを pseudocode で記述。

---

### 判断 5: 新規エージェント追加手順（拡張性設計）

**背景**: Phase 2 で 3 体目エージェント（testing / architecture）追加予定

**実装判断**:
- ハーネス内に "新規エージェント追加手順" セクションを明記
- 5 ステップ: (1) ファイル作成, (2) 呼び出し順序追加, (3) 出力フォーマット拡張, (4) 統合ロジック更新, (5) テスト検証
- インターフェース標準化: 各エージェント出力は `{ "findings": [...], "summary": {...} }` JSON 形式

**効果**: 新規エージェント開発者（他のジェネレーター）が独習可能な設計を実現

## 既知の制限・改善点

### 制限 1: 静的解析のみ

実装の制約上、以下は検出不可:
- 実行時のタイミング攻撃
- 動的に生成される脆弱性
- コンテキスト依存の判断（例：ロギング用ならフォーマット文字列は安全）

**対処**: ドキュメントに明記。Phase 2 で動的解析エージェント追加を検討。

---

### 制限 2: 言語特化性

Python / JavaScript / Go を主対象。その他言語は「言語判定失敗」と報告。

**対処**: Phase 2 で言語別ルール追加（Ruby / C# / Java 等）

---

### 制限 3: コンテキスト非依存

例：`os.system()` 呼び出しをすべてコマンドインジェクション脆弱性と判定。
実際には安全な用途も存在。

**対処**: 誤検知は「Low」重要度で報告。実装者がコンテキスト確認。
