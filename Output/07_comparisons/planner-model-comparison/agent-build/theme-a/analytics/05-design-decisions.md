# 設計決定記録 — Theme A 仕様の差分分析

**分析日**: 2026-05-09  
**対象**: Opus版仕様（spec.md） vs Haiku版仕様（spec.md）

---

## 1. エージェント分割方針の違い

### Opus版の設計判断

#### 基本戦略: 「2体による最小構成」

```
Opus版の背景:
- Phase 1: セキュリティ + 品質の2体で必須要件（M1-M9）を完全カバー
- Phase 2: その後に3体目（Testing/Architecture）を追加

理由:
  1. 複雑度を段階的に管理
  2. 2体でも脆弱性・品質問題は網羅可能
  3. Phase 1 での高い完成度を優先
```

#### エージェント設計の粒度

| エージェント | 責務の粒度 | パターン数 | 特性 |
|----------|---------|---------|------|
| セキュリティ | 細粒度 | 10パターン | OWASP Top10 に準拠 |
| 品質 | 細粒度 | 12パターン | 数値しきい値で機械的判定 |

**特徴**: 各エージェントが「専門分野に特化」

---

### Haiku版の設計判断

#### 基本戦略: 「3体による将来対応」

```
Haiku版の背景:
- Phase 1+2を見据えた統合設計
- 初期実装は2体、3体目追加の手順を明確化
- エージェント間協調ロジックを先取り仕様化

理由:
  1. 拡張パスが明確なら長期運用が容易
  2. 新規開発者が独習可能な標準化
  3. 矛盾検出など高度な機能を初期仕様に組み込む
```

#### エージェント追加の標準化（5ステップ）

```markdown
## 新規エージェント追加手順（Haiku版の独自工夫）

1. ファイル作成（.md フォーマット、200行以内）
2. ハーネスの呼び出し順序に追加
3. 出力フォーマット拡張（JSON スキーマ統一）
4. 統合ロジック更新（重複排除・優先度付け）
5. テスト検証（既知パターンで検出率確認）

→ 誰でも追加可能な標準化プロセス
```

**特徴**: 「拡張可能性」を設計に組み込む

---

## 2. 重要度分類の粒度の違い

### Opus版の重要度分類

#### 4段階: Critical / High / Medium / Low

**セキュリティ観点での分類**:
```
Critical: 直接的なRCE/データ漏洩/権限昇格に繋がる
  ├─ SQLi、ハードコード秘密、認証欠如、デシリアライズ
  
High: 攻撃成功で実害が出るが追加条件が必要
  ├─ XSS、CSRF、弱い暗号、パストラバーサル
  
Medium: 防御の弱体化・将来的リスク
  ├─ 弱い乱数、オープンリダイレクト
  
Low: ベストプラクティス違反
  └─ 推奨設定欠如
```

**品質観点での分類**:
```
High: 保守を著しく困難にする（長関数、深ネスト、空catch）
Medium: 中長期の負債化（重複、デッドコード、命名違反）
Low: ベストプラクティス違反（マジックナンバー、TODO残置）
```

**特徴**: セキュリティと品質で**異なる分類軸**

---

### Haiku版の重要度分類

#### 4段階 + 優先度カテゴリ: 「即座・次回・次リファクタ・ベストプラ」

**Haiku版の独自細分化（Opus未定義）**:

```markdown
## 優先度分類（Haiku版の工夫）

即座（Immediate）
  ├─ このSprint 内に修正必須
  ├─ セキュリティ Critical、品質 High の一部
  └─ 例: SQL インジェクション、認証欠如

次回（Next Sprint）
  ├─ 1-2週間以内の修正計画
  ├─ セキュリティ High、品質 Medium の大部分
  └─ 例: XSS、長関数（60行超）

次リファクタ（Upcoming Refactoring）
  ├─ 次のリファクタリングサイクルで対応
  ├─ 品質 Low、一部 Medium
  └─ 例: マジックナンバー、命名規則違反

ベストプラ（Best Practice）
  ├─ 優先度低（参考情報）
  ├─ ドキュメント・スタイル指南
  └─ 例: コメント不足、推奨設定欠如
```

**特徴**: 実装者の**アクション計画を明確化**

---

## 3. エッジケース対応戦略の差分

### Opus版のエッジケース対応

#### 定義されているケース

| ケース | 対応策 | 詳細 |
|--------|--------|------|
| **大規模差分（1000行超）** | ✓定義済み | チャンク分割（500行単位）+ マージ |
| **矛盾する提案** | ✓定義済み | Trade-offs セクションで並記（Phase 2） |
| **言語判定失敗** | ✓定義済み | Warning 付きレポート |
| **差分なし / 微小** | ✓定義済み | 短縮レポート / 軽微判定 |

**対応率**: 4/4 = 100%（ただし部分的）

---

### Haiku版のエッジケース対応

#### より詳細な対応定義

| ケース | Opus | Haiku | 差分 |
|--------|------|--------|------|
| **大規模差分** | チャンク分割 | + Note 表示、チャンク結果をマージ時のソート方法 | 詳細化 |
| **矛盾する提案** | Trade-offs セクション | + 既知矛盾ペア表、優先度自動調整ロジック | 実装化 |
| **言語判定失敗** | Warning | + 言語別ヒューリスティック判定、failure mode 定義 | 堅牢化 |
| **差分なし** | スキップ | + 「変更なし」の短縮レポート形式 | 標準化 |
| **微小差分** | 軽微判定 | + 「1-3行」「4-10行」の細分定義 | 精緻化 |

**新規ケース**:
- **矛盾検出の既知ペア表**: セキュリティ ↔ パフォーマンス、キャッシング ↔ タイムリーネス等を明記

**対応率**: 5/5 + 新規項目 = 拡張

## 4. エージェント間協調ロジックの設計差分

### Opus版の協調設計

#### Phase 1: 順次呼び出し（協調なし）

```
セキュリティエージェント実行
          ↓
        結果取得
          ↓
コード品質エージェント実行
          ↓
        結果取得
          ↓
ハーネスが重複排除・優先度付け
          ↓
    Markdown 生成
```

**特徴**: 単純な「順次実行」。協調ロジックはなし。

#### Phase 2計画: 矛盾検出ロジック

```
3体全エージェント実行
          ↓
  findings 統合
          ↓
  矛盾ペア検出
  （既知ペアの照合）
          ↓
Trade-off セクション生成
```

**ポイント**: 「Phase 2 での実装」として延期

---

### Haiku版の協調設計

#### Phase 1から組み込む: 高度な協調

```
セキュリティエージェント実行
        + コード品質エージェント実行
          ↓
  両者の findings 統合
          ↓
既知矛盾ペアをマッチング
  セキュリティ↔パフォーマンス等
          ↓
矛盾検出時：
  1. 両提案を「Trade-off」として並記
  2. 優先度を判定（Security > Quality > Performance）
  3. トレードオフ分析セクションに追記
          ↓
    Markdown 生成
```

**新規定義**: 既知矛盾ペア表

```markdown
## 既知矛盾ペア（Haiku版の工夫）

| 矛盾番号 | 主張1（Security） | 主張2（Performance） | トレードオフ解析 |
|--------|----------------|------------------|-------------|
| TRD-001 | 全データ暗号化必須 | TLS ハンドシェイク遅延 | 機密度で判定 |
| TRD-002 | キャッシング禁止 | 応答時間短縮 | 有効期限で折衷 |
```

**特徴**: 「Phase 1 から実装化」、自動優先度調整

## 5. 具体的な設計判断の差異表

### 判断1: エージェント数（Phase 1）

| 判断項目 | Opus版 | Haiku版 | トレードオフ |
|---------|--------|--------|----------|
| **Phase 1 エージェント数** | 2体 | 2体 | 同一 |
| **理由** | 最小構成で要件達成 | 拡張パスを明確化 | 異なる視点 |

**Opus判断**: 「今のニーズに絞る」  
**Haiku判断**: 「将来の拡張を考えて設計」

---

### 判断2: 矛盾検出タイミング

| 判断項目 | Opus版 | Haiku版 |
|---------|--------|--------|
| **実装時期** | Phase 2 | Phase 1 仕様化 |
| **仕様の詳細度** | 概要のみ | 既知ペア表、優先度ロジック含む |
| **実装難度** | 中程度（新規実装） | 低い（仕様に従うだけ） |

**Opus判断**: 「段階的実装」（保守保険）  
**Haiku判断**: 「先制的仕様化」（拡張加速）

---

### 判断3: エージェント追加ガイド

| 判断項目 | Opus版 | Haiku版 |
|---------|--------|--------|
| **新規追加手順** | 未定義 | 5ステップで標準化 |
| **インターフェース標準** | 暗黙 | JSON スキーマで明示 |
| **保守性への影響** | 次回手戻りリスク | 独習可能、リスク低 |

**Opus判断**: 「実装ありき」（経験から学ぶ）  
**Haiku判断**: 「スケーラビリティ設計」（予測から設計）

---

### 判断4: 出力フォーマット

| 観点 | Opus版 | Haiku版 |
|------|--------|--------|
| **基本形式** | Markdown | Markdown |
| **中間表現** | JSON (暗黙) | JSON (スキーマ明示) |
| **拡張方法** | エージェント毎にカスタマイズ | スキーマ統一で標準化 |

**Opus判断**: 「フレキシブル」  
**Haiku判断**: 「コンシステント」

## 6. 設計哲学の比較

### Opus版の設計哲学

```
「最小構成で最大価値を」
  ├─ Phase 1: 脆弱性 + 品質の2軸で十分
  ├─ Phase 2: 必要なら追加（段階的）
  └─ 実装を通じて拡張パスが明確化する（漸進的改善）
```

**特徴**: 「実装最優先」「検証駆動」

---

### Haiku版の設計哲学

```
「スケーラブルな基盤を最初から」
  ├─ Phase 1+2を見据えた統合設計
  ├─ 新規エージェント追加が自明な手順
  └─ 既知矛盾の検出も初期仕様に含める（先制的）
```

**特徴**: 「設計最優先」「拡張駆動」

---

## 7. 実装への影響

### Opus版を採用した場合

**メリット**:
- 実装工数が少ない（2エージェント）
- テストが明確（既知脆弱性100%検出）

**デメリット**:
- Phase 2 追加時に設計見直しが発生
- 新規エージェント追加が毎回一からの設計

---

### Haiku版を採用した場合

**メリット**:
- 3体目追加が手順書に従うだけ
- 矛盾検出など高度な機能が初期組み込み
- 次世代開発者が参入しやすい

**デメリット**:
- 初期実装が複雑（Tradeoff検出ロジック）
- pseudocode から実装への翻訳コストあり

---

## 8. 今後の設計改善案

### 両方のいいとこ取り

**推奨ハイブリッド設計**:

1. **基盤**: Haiku版の「新規エージェント追加5ステップ」を採用
   - 拡張性を保証

2. **矛盾検出**: Phase 1.5 で段階的に実装
   - Haiku版の既知ペア表を参考
   - ただし最初は「Warning付き」で保留

3. **実装検証**: Opus版のサンプルテスト方法を参考
   - 100% 検出率を確認してから本運用

4. **ドキュメント**: Opus版の実装記録 + Haiku版の設計記録を統合
   - 「なぜこう設計したか」を明記
