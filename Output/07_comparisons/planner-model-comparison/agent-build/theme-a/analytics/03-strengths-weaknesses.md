# 強弱分析 — Opus-Haiku vs Haiku-Opus

**評価日**: 2026-05-09  
**テーマ**: PR Review Automation Harness

---

## Opus-Haiku版（Opusプランナー × Haikuジェネレーター）

### 強み

#### 1. 実装による検証済みの高精度 ⭐⭐⭐⭐⭐

**内容**:
- サンプル脆弱性：5パターン全検出（100%）
- サンプル品質問題：9パターン全検出（100%）
- 期待レポートと実装結果が完全一致

**具体例**:
```
Sample 1（脆弱性）検出結果：
- SEC-001 SQL インジェクション ×2 件
- SEC-004 ハードコード秘密 ×1 件
- SEC-005 弱い暗号化 ×1 件
- SEC-007 安全でないデシリアライズ ×1 件
→ 期待値（5件）と完全一致、過検出なし
```

**価値**:
- 本運用で即座に利用可能
- 検出精度への信頼が高い
- バグリスク最小化

---

#### 2. エージェント定義の簡潔性 ⭐⭐⭐⭐

**内容**:
- セキュリティエージェント：184行（200行制限内）
- コード品質エージェント：195行（200行制限内）
- 詳細度を損なわない簡潔性

**具体例**:
```markdown
## 検出ルール（簡潔）

| ID | パターン | 検出ヒント |
|----|---------|---------|
| SEC-001 | SQL インジェクション | `+`、f-string、テンプレートリテラル |
| QUAL-001 | 関数長すぎ | 50行超 |
```

**価値**:
- 保守が容易（変更影響が限定的）
- ドキュメントとして読みやすい
- メモリフットプリントが小さい

---

#### 3. Markdown出力形式の実装者向け最適化 ⭐⭐⭐⭐

**内容**:
- Evidence（証拠）：脆弱コード片を明示
- Description：背景と影響を簡潔に説明
- Recommendation：修正案をコード例で提示

**具体例**:
```markdown
### [Critical] SEC-001: SQL Injection via string concatenation
- **File**: `src/api/user.py:12`
- **Evidence**: query = "SELECT * FROM users WHERE id = " + user_id
- **Description**: ユーザー入力を直接SQL文に連結している
- **Recommendation**: 
  cursor.execute("SELECT * FROM users WHERE id = ?", (user_id,))
```

**価値**:
- 実装者がすぐに対応できる
- コードレビューの質が向上
- アクション漏れを防止

---

#### 4. 完全なPhase 1実装 ⭐⭐⭐

**内容**:
- Must-Have 9項目をすべて実装
- セキュリティと品質の2エージェント完成
- ハーネスの統合ロジック動作確認

**価値**:
- 「完成」の状態に到達
- 付け足しや試行錯誤が不要

---

### 弱み

#### 1. Phase 2への拡張手順が不明確 ⭐⭐

**問題**:
- 新規エージェント追加ガイドがない
- 3体目エージェント（Testing/Architecture）の実装パターンが不明
- エージェント間協調ロジックの設計が未定義

**具体例**:
- 新規エージェント X を追加するには？
  - どのファイル形式か
  - ハーネスに どう組み込むか
  - 出力フォーマットは どう統一するか
  → 答えが specification に書かれていない

**影響**:
- Phase 2 開始時に手戻りが多い
- 新規開発者の参入障壁が高い

---

#### 2. エージェント間協調ロジックの未実装 ⭐⭐

**問題**:
- Tradeoff分析の実装例がない
- 「暗号化必須 vs パフォーマンス優先」のような矛盾検出が機能していない
- 優先度調整ロジックが仕様にも実装にも未定義

**具体例**:
```
セキュリティが「全データを暗号化」と提案
パフォーマンスが「暗号化で遅延」と指摘
→ この矛盾をどう報告するか未定義
```

**影響**:
- 複雑な設計判断を支援できない
- PR レビューの深さが制限される

---

#### 3. 設計判断の詳細説明不足 ⭐

**問題**:
- implementation-summary.md が最小限の説明に留まる
- なぜこの設計か、他の選択肢は何か、の記録がない
- エッジケース対応の実装例が散在

**具体例**:
```
「なぜ400行サイズのハーネスをこう設計したのか」
→ 説明がない
```

**影響**:
- 次のジェネレーターが判断根拠を理解できない
- 設計哲学の継承が困難

## Haiku-Opus版（Haikuプランナー × Opusジェネレーター）

### 強み

#### 1. 明確な拡張性設計 ⭐⭐⭐⭐⭐

**内容**:
- 新規エージェント追加を5ステップで標準化
  1. ファイル作成（`.md` フォーマット）
  2. ハーネスの呼び出し順序に追加
  3. エージェント出力フォーマット拡張（JSON スキーマ）
  4. 統合ロジック更新（重複排除・優先度付け）
  5. テスト検証
- インターフェース標準化が JSON スキーマで明示
  ```json
  {
    "agent": "new-agent-name",
    "findings": [
      {
        "id": "NEW-001",
        "severity": "High",
        "category": "...",
        "file": "...",
        "line": 42
      }
    ],
    "summary": { "critical": 0, "high": 1, ... }
  }
  ```

**価値**:
- Phase 2 開始時の迷走がない
- 新規開発者が独習可能
- Template Method パターンで拡張を標準化

---

#### 2. 仕様以上の工夫 ⭐⭐⭐⭐⭐

**内容**:

**2.1 Tradeoff分析セクションの実装化**
- Opus仕様には「矛盾検出は Phase 2」と書かれているが、Haiku-Opus版は実装化
- 既知矛盾ペアを表で列挙（セキュリティ ↔ パフォーマンス、等）

**2.2 優先度分類の独自細分化**
- 通常: Critical → High → Medium → Low
- Haiku-Opus版: 「即座・次回・次リファクタ・ベストプラ」4段階
  - 実装者が短期（Sprint）と長期（Roadmap）で優先度区分可能

**2.3 エッジケース対応の先読み**
- 5パターンの対応表を事前定義
  - ファイル > 1000行 → チャンク分割
  - 言語判定失敗 → 明示的エラー報告
  - 矛盾する提案 → Tradeoff分析
  - 微小差分 → 軽微判定
  - 差分なし → スキップ

**価値**:
- 仕様レベルでの完全性が高い
- Phase 2 の実装がスムーズ
- ドメイン知識が十分に蓄積

---

#### 3. エッジケース対応の充実 ⭐⭐⭐⭐

**内容**:
- 5パターン × 対応方法を対応表で網羅的に整理
- 各ケースの処理フロー（チャンク分割サイズ、エラー報告形式等）が明記

**具体例**:
```markdown
## エッジケース対応

| ケース | 対応方法 |
|--------|---------|
| ファイル > 1000行 | 差分を 500 行ごとにチャンク分割、各チャンクを順序実行、結果統合 |
| 言語不明な diff | ファイル拡張子から言語判定、失敗時は「言語判定失敗」と明示的に報告 |
| 矛盾する提案 | ハーネスが「トレードオフ分析」として両方を報告、優先度を明記 |
| 変更なし or 微小変更 | ハーネスが「変更なし」または「軽微」と判定、処理短縮 |
```

**価値**:
- 予期しない状況での対応が準備済み
- ランタイムエラー・ハングが少ない

---

#### 4. エージェント間協調ロジックの仕様化 ⭐⭐⭐⭐

**内容**:
- Tradeoff分析の具体例を記載（暗号化 vs パフォーマンス）
- 優先度自動調整：Security > Testing > Quality > Performance
- 既知矛盾ペアの検出パターン表

**価値**:
- 実装時の判断基準が明確
- 複雑な設計判断を支援できる機構ができる
- スケーラブルな協調設計

---

### 弱み

#### 1. Phase 2要件の部分実装 ⭐⭐

**問題**:
- 3体目エージェント（Testing/Architecture）は未実装
- Should-Have として挙げられているが、完成度スコアに影響

**具体例**:
```
仕様: 「Phase 2 では Testing エージェントを追加」
実装: 未実装（pseudocode）
```

**影響**:
- スコア上 38/40 に留まる（満点ではない）
- 実装者が Testing エージェントの実装例を参照できない

---

#### 2. 実装レベルの検証不足 ⭐

**問題**:
- エージェント定義が pseudocode レベル
- 実装では検証されていない
- Tradeoff検出ロジックの精度が未確認
- 矛盾検出の誤検出率が不明

**具体例**:
```markdown
## ハーネスの集約処理（pseudocode）

1. 入力 diff をパース
2. 各エージェントを順次呼び出し
3. 結果を重要度でソート
4. 重複排除（file:line + id）
→ 実装形式は明記されていない
```

**影響**:
- 実装開始時に予期しない困難に直面する可能性
- テスト駆動開発ができていない

---

#### 3. ハーネスの複雑性（軽微） ⭐⭐⭐

**問題**:
- pr-review-harness.md が 388行と比較的長い
- 責務が拡大気味（入力解析 + 呼び出し + 重複排除 + 優先度付け + Tradeoff検出）

**具体例**:
```markdown
## ハーネスの責務（多数）
1. 言語判定
2. 差分パース
3. エージェント呼び出し（順次 or 並列）
4. 結果パース
5. 重複排除
6. 優先度付け
7. Tradeoff検出
8. Markdown生成
→ 単一責務原則が若干違反気味
```

**影響**:
- 保守時の変更影響が大きい
- 単体テストが複雑になる（軽微）

## 適用パターン別の推奨

### シナリオ1: 「今すぐ本番運用したい」

**推奨**: Opus-Haiku版  
**理由**:
- 実装検証済み（100%）
- 脆弱性・品質問題の検出精度が確認済み
- そのまま PR レビューに使用可能

**注意点**:
- Phase 2 拡張を計画しているなら、事前に新規エージェント追加ガイドを作成

---

### シナリオ2: 「長期保守・拡張を計画している」

**推奨**: Haiku-Opus版  
**理由**:
- 新規エージェント追加が5ステップで標準化
- Tradeoff分析など高度な協調ロジックが実装化
- Phase 2 への拡張パスが明確

**注意点**:
- 実装開始前に pseudocode を実装言語に翻訳
- サンプルコードでの検証を早期に実施

---

### シナリオ3: 「両方を活かしたい」

**推奨**: Opus-Haiku版を基盤 + Haiku-Opus版の設計を参照  
**戦略**:
1. Opus-Haiku版を本運用開始（実装検証済み）
2. Haiku-Opus版から新規エージェント追加ガイドを拝借
3. Phase 2 では Haiku-Opus版の Tradeoff分析を参考に実装
4. 最終的には両者のハイブリッド実装

---

## 総合的な判定

| 項目 | Opus-Haiku | Haiku-Opus |
|-----|-----------|-----------|
| **即戦力** | ⭐⭐⭐⭐⭐ | ⭐⭐⭐ |
| **拡張性** | ⭐⭐ | ⭐⭐⭐⭐⭐ |
| **保守性** | ⭐⭐⭐⭐ | ⭐⭐⭐⭐ |
| **実装難易度** | ⭐（完成） | ⭐⭐（実装要） |

**総合推奨**: Haiku-Opus版（将来性重視）  
**但し**: Opus-Haiku版の実装検証結果を参考に検出ロジックを精緻化すること
