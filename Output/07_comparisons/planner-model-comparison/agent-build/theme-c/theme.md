# Theme C: 技術選定エージェント群

## テーマ概要

新規プロジェクトの要件定義から、最適なテック スタック（言語・フレームワーク・ DB・インフラ）を提案・比較・決定する**複数エージェントの協調システム**を設計・構築する。

要件分析 → 候補技術抽出 → 比較・スコアリング → トレードオフ分析 → 最終推奨という多段階の意思決定プロセスを複数エージェントで分担し、人間の CTO / アーキテクトの判断をサポートするハーネスの実装を通じて、マルチエージェント「協調と競合」パターンを習得する。

---

## 要件定義

### Must-Have（必須）

1. **最低2体の専門エージェント実装**
   - 要件分析エージェント：「スケーラビリティ・セキュリティ・パフォーマンス・開発速度」の重要度を要件から判定
   - 技術提案エージェント：分析結果から「言語・フレームワーク・DB」の候補 3〜5 セットを提案
   - 両者は `.md` 形式の Claude Code Agent として定義する

2. **ハーネス（意思決定統合エージェント）の実装**
   - 複数エージェントの結果を受け取り、統一されたスコアリング基準で比較
   - 「優位性マトリックス」（要件軸 × 技術候補）を生成
   - トレードオフ分析（「言語 A は速いが学習コストが高い」等）を実施
   - 最終推奨テック スタック + 理由を出力

3. **入力フォーマット**
   - プロジェクト要件書（3〜5 ページ）：スケール・セキュリティ・パフォーマンス・チーム規模・リリース期限等
   - 制約条件（予算・既存スキル・商用サポート等）

4. **出力フォーマット**
   - Markdown レポート
   - 「候補テック スタック」表（言語・フレームワーク・DB・クラウド毎に 3 案提示）
   - 「スコアリング表」（要件軸 × 技術、0-10 点で採点）
   - 「トレードオフ分析」セクション（「技術 A は◎だが××」形式）
   - 「最終推奨」+ 根拠

---

### Should-Have（強く推奨）

1. **3体目の専門エージェント追加**
   - コスト見積もりエージェント：各テック スタックのインフラ・開発・保守コストを推定
   - または「競合技術スコアリングエージェント」：候補技術を客観的指標で点数化

2. **ハーネス内でのエージェント間協調**
   - 要件 → 提案 → スコアリング → コスト推定 のパイプライン
   - 「要件が矛盾する」場合のエージェント間での優先度調整

3. **複数業界パターンの対応**
   - SaaS（スケーラビリティ重視）
   - ヘルスケア（セキュリティ・コンプライアンス重視）
   - IoT/組込（パフォーマンス重視）
   - など異なる優先軸を持つプロジェクトに対応

4. **ドキュメント**
   - スコアリングの評価基準（なぜこの技術にこのスコアか）
   - 各テック スタックの学習曲線・チームオンボーディング時間の推定
   - リスク分析（「この技術は廃止予告」等）

---

### Nice-to-Have（あると嬉しい）

1. **マイグレーションパス分析**
   - 「将来 A 言語から B 言語に移行する場合」のコスト / 工期を推定

2. **OWASPトップ10 / セキュリティフレームワークに基づくスコアリング**
   - セキュリティスコアを業界標準に基づいて算出

3. **開発体験（DX）スコア**
   - テック スタックの「開発者の満足度」を客観化（GitHub stars, npm downloads 等から推定）

4. **AI モデル検討エージェント**
   - 「プロジェクトに LLM API / 推論モデルが必要か」を判定し、統合方法を提案

---

## 技術スタック（推奨）

- **言語**: Markdown（`.md` Agent 定義）
- **エージェント数**: 最低2体（要件分析・技術提案）、推奨3体（+コスト推定 or +スコアリング）
- **ハーネス実装**: Claude Code Agent SDK（`.md` 形式）
- **入力サンプル**: 実際のプロジェクト要件書 3〜5 パターン（SaaS / ヘルスケア / IoT など）
- **出力形式**: Markdown レポート（スコアリング表・トレードオフ分析を含む）

---

## 実装フェーズ計画

### Phase 1: 最小構成（Must-Have）

1. 要件分析エージェント実装
2. 技術提案エージェント実装
3. ハーネス実装（結果統合・スコアリング・トレードオフ分析）
4. サンプル要件書 2 件（SaaS・ヘルスケア）でテスト
5. 「推奨テック スタック + 理由」レポート出力確認

**完成条件**: 異なる要件に対して、適切に異なるテック スタックを推奨できること。スコアリング表がロジカルであること

### Phase 2: 拡張（Should-Have / Nice-to-Have）

1. 3体目エージェント追加（コスト推定 or スコアリング）
2. 業界パターン対応（SaaS / ヘルスケア / IoT の判別ロジック）
3. マイグレーションパス分析（or セキュリティスコアリング）
4. ドキュメント充実

**完成条件**: 「スケーラビリティ重視」「セキュリティ重視」など異なる優先軸のプロジェクト要件に対して、柔軟にテック スタック推奨を実施できること

## 品質基準（これが揃えば完成）

✓ 要件分析・技術提案・（コスト or スコアリング）の複数エージェントが独立して機能  
✓ ハーネスが複数エージェントの結果をマージし、統一スコアリングできる  
✓ スコアリング表が「なぜこのスコアか」を説明可能  
✓ トレードオフ分析が「技術 A は◎だが××」として対立軸を明示  
✓ 最終推奨テック スタックが「複数候補の中から 1 つを選んだ根拠」を記載  
✓ サンプル要件書（SaaS・ヘルスケア・IoT など異なる優先軸）に対して、異なる推奨結果を出力  

---

## エッジケース

1. **要件が矛盾する場合**
   - 「スケーラビリティ重視」「開発速度重視」「セキュリティ重視」が同時に必須
   - → ハーネスが「優先度付け」を実施して、トレードオフを明示

2. **レガシーシステムからのマイグレーション**
   - 「既存 PHP コードベースからの移行」など
   - → 「学習コスト」「マイグレーション工期」を考慮した提案が必要

3. **新興技術（Rust・Go・Kotlin など）の位置付け**
   - 「これから学ぶべき言語か」vs「今すぐ採用すべき技術か」の判定
   - → ハーネスがリスク評価を実施

4. **予算制約が非常に厳しい場合**
   - 「オープンソースのみ」「商用サポート必須」などの制約
   - → コスト見積もりエージェントが制約を優先

## 受け入れ条件（Acceptance Criteria）

1. **機能完全性**
   - 最低2体のエージェント（要件分析・技術提案）が定義ファイル（`.md`）として作成される
   - ハーネスが複数エージェントの結果を統合し、「推奨テック スタック」を出力できる
   - サンプル要件書 2 件（異なる優先軸）でテスト、異なる推奨結果を出力できること

2. **品質要件**
   - エージェント定義ファイルが 200 行を超えない
   - スコアリング表が「なぜこのスコアか」を説明可能な形式
   - トレードオフ分析が「複数候補の優位性の違い」を明確に表現

3. **ドキュメント**
   - 各エージェントの判定ロジック（「要件から優先軸をどう判断したか」等）
   - スコアリングの評価基準表
   - ハーネスの統合ロジック（優先度付け・トレードオフ分析）

4. **再現性**
   - サンプル要件書を何度も実行し、同じ推奨結果が出力される
   - 環境依存がないこと
