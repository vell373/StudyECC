## 利用シーンごとの推奨パターン

### シーン 1: スタートアップ MVP（予算 $50K-150K）

**推奨**: **Opus-Haiku パターン**（80% 品質）

#### 理由

| 要因 | 評価 |
|------|------|
| 実装速度 | 4 週間 vs 5.5 週間 → **Opus が 1.5 週間短縮** |
| 初期コスト | $18.5K tokens vs $31.2K tokens → **Opus が 40% 低コスト** |
| スコープ適合 | Phase 1（RequirementAnalyzer + TechnologyProposal）で十分 | 
| リスク | 低い。3 業界テスト済み実装 |

#### 実装ロードマップ

**Week 1-2: 要件定義**
```
Step 1: RequirementAnalyzer 仕様の MVP 化
  - 3 業界（SaaS / Healthcare / IoT）に限定
  - 優先度軸を 4 軸（Scalability / Security / Performance / DevSpeed）に固定
  - 15 言語の言語別スコア表を作成

Step 2: TechnologyProposal 仕様の MVP 化
  - 候補スタック 20 種類に限定（Phase 1 バージョンの 30 種類から削減）
  - Tier 1-2（Proven Cloud-Native, Developer-Friendly）のみ採用
  - トレードオフ分析は 2 要素で十分
```

**Week 3-4: Python 実装**
```
Step 3: TechSelectionHarness クラスの実装
  - 273 行のコンパクト実装を目標
  - calculate_weights() → score_candidate() → recommend_stack() の順序で実装
  - 単体テスト 10 個（3 業界 × 3 パターン + Edge cases）

Step 4: テスト実行と検証
  - SaaS: API Gateway + Node.js + PostgreSQL の組み合わせで検証
  - Healthcare: Java + Spring + Oracle で検証
  - IoT: Go + gRPC + TimescaleDB で検証
```

**Week 4: デプロイ準備**
```
Step 5: CI/CD パイプライン構築
  - GitHub Actions で自動テスト
  - コード品質スコア（SonarQube）を設定
  - 本番ドキュメントの作成
```

#### MVP リリース後の拡張計画

**Q2 2026**: Phase 2 Should-Have の企画
- 技術選定後の「コスト見積もり」が顧客要望で出てきた場合
- CostEstimator Agent の追加実装を検討

#### 成功メトリクス

```
評価指標               ターゲット    測定方法
------------------    ----------    ----------
実装期間              4 週間        実装開始〜テスト完了
テストカバー率        >80%          pytest --cov
顧客対応時間          <30 分        案件ごとのログ記録
推奨スタック満足度    >85%          6 ヶ月後のフォローアップ
```

## ハイブリッド・アプローチ

### Phase 別段階的実装（推奨）

**最適戦略**: Opus-Haiku パターンで開始 → Haiku-Opus へ段階的拡張

```
Timeline           Scope               Effort      Cost
--------           -----               ------      ----
Month 1-4          Opus (Phase 1)      4 weeks     $18.5K tokens
Month 5-8          → Phase 2 Agent     2 weeks     +$8.0K tokens
Month 9-12         → 5 業界ルール      3 weeks     +$5.0K tokens
                   ----------------------------------------
Total (Year 1)     Haiku-Opus equivalent          ~$31.5K tokens
```

#### 実装理由

**利点**:
1. **早期リリース**: Month 1 で 80% 品質の MVP を市場投入
2. **リスク分散**: Phase 1 の安定化後に Phase 2 へ進む
3. **顧客フィードバック**: 実運用で要件を精緻化してから Phase 2 実装
4. **チーム成長**: Phase 1 習得後に Phase 2 の複雑性に対応

**デメリット**:
- 総実装期間が 8 ヶ月（フルスクラッチの 5.5 ヶ月より長い）
- API の後方互換性保証が必要
- ドキュメント更新コストが増加

#### 段階的実装の詳細

**Phase 1 リリース（Month 4）**
```
納品物:
  - TechSelectionHarness クラス（273 行）
  - テスト 3 パターン（SaaS / Healthcare / IoT）
  - 基本ドキュメント + API リファレンス

ユーザーシーン:
  - 初期段階の技術選定
  - MVP 構築の技術スタック決定
  - 予算 $50K-150K のプロジェクト
```

**Phase 2 拡張（Month 8）**
```
追加納品物:
  - CostEstimator Agent（220 行）
  - 業界別ルール表（35 ルール）
  - Year 1/3/5 ROI 計算エンジン

新ユーザーシーン:
  - スケーリング期の意思決定
  - 長期 TCO 分析が必要な大型案件
  - 予算 $200K-500K のプロジェクト

API 変更:
  - `/recommendation` エンドポイント に cost_projection 属性追加
  - 後方互換性を維持（cost_projection はオプション）
```

**Phase 3 最適化（Month 12）**
```
追加納品物:
  - 5 業界ルール表の完成（SaaS/Healthcare/IoT/Enterprise/EdTech）
  - 業界ごとのシナリオテンプレート 25 パターン
  - 高度な紛争解決フレームワーク（4 段階優先度）

新ユーザーシーン:
  - エンタープライズ用意思決定支援
  - 規制業界の要件対応
  - マルチステークホルダーの利害調整

API 変更:
  - `/conflict-resolution` エンドポイント新規追加
```

## リスク管理と緩和戦略

### Opus-Haiku パターンのリスク

| リスク | 確率 | 影響度 | 緩和策 |
|--------|------|--------|--------|

| 新業界への対応遅延 | 中 | 中 | ルール追加の仕組みを事前構築（テンプレート化） |

## 成功事例シナリオ

### 事例 1: スタートアップ AI Software 企業（Opus-Haiku）

**状況**:
- Series A 資金調達済み（$2M）
- エンジニア 8 人（Python 強い、インフラ経験浅い）
- 市場投入を 8 週間以内に予定

**実装選択**:
1. Week 1-2: 3 業界（SaaS / EdTech / IoT）に限定
2. Week 3-4: TechSelectionHarness の MVP 実装（273 行）
3. Week 5-6: 統合テスト + 本番デプロイ
4. Week 7-8: 顧客トライアル + フィードバック収集

**成果**:
- Month 4: 初版リリース（80% 品質）
- 顧客反応: 「十分。むしろシンプルで信頼できる」
- Month 6-8: 顧客フィードバックに基づき Phase 2 企画開始

**得られた知見**:
> 「最初から完全性を目指さず、必要最小限から始める方が、
>  顧客の真のニーズを理解できる。Phase 2 の要件が明確になった」

### 事例 2: グローバル金融企業（Haiku-Opus）

**状況**:
- 5 つの子会社がそれぞれ異なる技術スタック採用予定
- 合計 200M ドルの技術投資決定が控えている
- 12 ヶ月の実装期間を確保

**実装選択**:
1. Month 1-2: 5 業界ルール（金融 / ヘルスケア / 製造 / 通信 / 小売）の詳細設計
2. Month 3-4: 4 エージェント仕様書作成（717 行）
3. Month 5-8: Go + Python 実装（本番グレード）
4. Month 9-10: Cost Estimator で各子会社の Year 1/3/5 ROI 計算
5. Month 11-12: 経営層への提案資料作成

**成果**:
- Month 12: 5 子会社全社の技術選定完了（92.5% 品質）
- 推定節減: Cost Estimator の精度により $15M の不要投資を回避
- Month 13+: Haiku-Opus パターンを社内標準化、他部門での利用

**得られた知見**:
> 「前半の詳細設計に 2 ヶ月要したが、後半の実装がスムーズで、
>  最終的には時間短縮。またコスト見積もり機能により、
>  経営層の信頼を大幅獲得できた」

### 「どちらを選ぶか」の最終判定表

| 判定軸 | Opus-Haiku | Haiku-Opus |
|--------|-----------|-----------|

| **保守性** | 高い（273 行） | 中程度（615 行） |
| **拡張性** | 良い（Phase 2 へ容易） | 既に高度（カスタマイズ向け） |
| **チーム要件** | ジュニア〜ミッドレベル | ミッド〜シニア |

### 最高の選択肢

**最も推奨**: **ハイブリッド段階的実装**
- Month 1-4: Opus パターンで MVP リリース
