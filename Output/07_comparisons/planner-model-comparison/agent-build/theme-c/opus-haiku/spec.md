# Theme C 設計仕様書：Tech Selection Agent Group

**作成日**: 2026-05-09  
**対象**: Opus Planner → Haiku Generator  
**テーマ**: 技術選定エージェント群  
**フェーズ**: 設計フェーズ（spec 策定）

---

## 概要

このシステムは、新規プロジェクトの要件定義から最適なテック スタック（言語・フレームワーク・DB・インフラ）を提案・比較・決定する複数エージェントの協調システムである。要件分析 → 候補技術抽出 → スコアリング → トレードオフ分析 → 最終推奨という多段階の意思決定プロセスを実装し、人間の CTO / アーキテクトの判断をサポートする。本テーマを通じて、複数エージェント間の「協調と競合」パターン、責務分離とインタフェース設計、意思決定ハーネスの実装方式を習得することを目的とする。

## 要件定義

### Must-Have（Phase 1: 最小構成）

#### 1. エージェント実装（2体以上）

##### 1.1 RequirementAnalyzerAgent
- **責務**: プロジェクト要件書を受け取り、4つの優先軸（スケーラビリティ・セキュリティ・パフォーマンス・開発速度）の重要度を判定
- **入力**: テキスト形式のプロジェクト要件書（3〜5ページ）、制約条件（予算・既存スキル・商用サポート）
- **出力**: 構造化JSON
  ```json
  {
    "priority_axes": {
      "scalability": {"score": 8, "reason": "..."},
      "security": {"score": 6, "reason": "..."},
      "performance": {"score": 5, "reason": "..."},
      "development_speed": {"score": 7, "reason": "..."}
    },
    "project_type": "SaaS|Healthcare|IoT|Enterprise",
    "team_size": "small|medium|large",
    "timeline": "months",
    "constraints": ["..."],
    "key_findings": ["..."]
  }
  ```
- **実装形式**: Markdown + YAML frontmatter 定義ファイル（`.md`形式、100-150行）
- **判定ロジック**:
  - 「スケーラブル」「マルチテナント」「数百万ユーザー」などのキーワード → スケーラビリティ高
  - 「GDPR」「PHI」「PCI-DSS」などのコンプライアンス要件 → セキュリティ高
  - 「リアルタイム」「低遅延」「IoT」 → パフォーマンス高
  - 「3ヶ月以内」「新規チーム」「学習コスト」 → 開発速度高

##### 1.2 TechnologyProposalAgent
- **責務**: RequirementAnalyzerAgent の出力を受け取り、3〜5セットの候補テック スタック（言語・フレームワーク・DB・クラウド）を提案
- **入力**: RequirementAnalyzerAgent の出力JSON
- **出力**: 構造化JSON（3セット提案）
  ```json
  {
    "candidates": [
      {
        "rank": 1,
        "set_name": "Python + Django + PostgreSQL",
        "language": "Python",
        "framework": "Django",
        "database": "PostgreSQL",
        "cloud": "AWS",
        "justification": "...",
        "pros": ["..."],
        "cons": ["..."]
      },
      {...},
      {...}
    ],
    "analysis_notes": "..."
  }
  ```
- **実装形式**: Markdown + YAML frontmatter（`.md`形式、100-150行）
- **提案ロジック**:
  - スケーラビリティ重視 → Node.js + Express + MongoDB / Java + Spring + Cassandra
  - セキュリティ重視 → Go + Gin + PostgreSQL / Rust + Actix + PostgreSQL
  - パフォーマンス重視 → Rust + Actix / Go + Gin / C++ + Boost
  - 開発速度重視 → Python + Flask / JavaScript + Next.js / Ruby on Rails
  - 複合要件 → 複数候補の中から優先度に応じて3セットを提案

#### 2. ハーネス実装（意思決定統合）

##### 2.1 スコアリング機能
- **入力**: RequirementAnalyzerAgent + TechnologyProposalAgent の出力
- **処理**: 「要件軸 × 技術候補」のマトリックスを作成し、各セルを 0-10 点で採点
- **採点基準の例**:
  
  | 軸 / 候補 | Python+Django | Node+Express | Rust+Actix | Go+Gin |
  |----------|:---:|:---:|:---:|:---:|
  | **スケーラビリティ** | 7 | 8 | 9 | 8 |
  | **セキュリティ** | 6 | 6 | 9 | 8 |
  | **パフォーマンス** | 5 | 7 | 10 | 9 |
  | **開発速度** | 9 | 8 | 5 | 7 |
  | **学習コスト** | 8 | 8 | 3 | 6 |
  | **コミュニティ** | 9 | 10 | 7 | 8 |

- **出力**: 点数表 + 重み付け合計スコア
  - 例: スケーラビリティ 0.3 × score + セキュリティ 0.2 × score + ... = 総合スコア

##### 2.2 トレードオフ分析
- **形式**: 「技術 A は◎だが××」形式での対立軸説明
- **例**:
  ```
  Python + Django:
    ◎ 開発速度: 短期間で MVP 開発可能
    ◎ チーム採用: Python 経験者が豊富
    ◎ ライブラリ: データサイエンス / 機械学習との連携が強い
    ×× パフォーマンス: C10K 問題への対応が課題
    ×× スケーラビリティ: マイクロサービス化時に複雑化

  Rust + Actix:
    ◎ パフォーマンス: メモリ効率・並行処理で最高峰
    ◎ セキュリティ: 所有権システムによるメモリ安全性
    ×× 開発速度: 学習曲線が急峻（習得に 3-6ヶ月）
    ×× チーム採用: Rust エンジニアが少なく採用困難
  ```

##### 2.3 最終推奨とその根拠
- **出力形式**:
  ```
  【最終推奨テック スタック】
  言語: Python
  フレームワーク: Django
  データベース: PostgreSQL
  クラウド: AWS

  【選定理由】
  1. 要件優先軸の適合度: 開発速度(9) × 0.4 + スケーラビリティ(7) × 0.3 + ...
  2. チーム状況への適合: Python 経験者 3名、3ヶ月で MVP リリース必須
  3. リスク評価: 将来のスケーリングに向けて事前設計は必須だが、
     初期段階では開発速度を優先
  4. 学習・運用: チームが既に習熟しており、追加学習コスト最小化
  ```

## エッジケース（事前対応）

### Case 1: 要件が矛盾する場合
**状況**: 「スケーラビリティ重視」「開発速度重視」「セキュリティ重視」が同時に最優先

**対応策**:
1. ハーネスが優先度ウェイト自動調整: スケーラビリティ 0.4, 開発速度 0.3, セキュリティ 0.3
2. トレードオフ分析で「全軸を同時に満たす技術は存在しない」と明示
3. 推奨: スコア最高のもの + 「次の優先軸は◎だが△」と注釈

**出力例**:
```
推奨: Python + Django + PostgreSQL (スケーラビリティ・開発速度に○)
注釈: セキュリティが △ のため、運用時に以下の対策を検討
  - WAF 導入
  - 定期セキュリティ監査
  - セキュリティエンジニア採用
```

---

### Case 2: レガシーシステムからのマイグレーション
**状況**: 「既存 PHP コードベース」「チーム全員 PHP 経験」「新言語への移行を検討」

**対応策**:
1. RequirementAnalyzerAgent が「マイグレーション」フラグを検出
2. TechnologyProposalAgent が学習コスト・マイグレーション工期を加算
3. スコアリングに「既存資産の活用」軸を追加（例: 6/10 = PHP との親和性）
4. トレードオフ分析で「新言語は○だが、既存資産の活用に××」と明示

**出力例**:
```
候補A: Python + Django（◎学習容易、◎スケーラビリティ、×× PHP 資産活用不可、工期 +3ヶ月）
候補B: PHP 8.x + Laravel（○既存資産活用、△スケーラビリティ、×× 長期メンテナンス）

推奨: Python + Django（長期視点では新言語への投資が回収可能、初期 3ヶ月は 2チーム並行実装で対応）
```

---

### Case 3: 新興技術（Rust・Go等）の位置付け
**状況**: 「Rust での実装を検討したい」が、チームに Rust 経験者ゼロ

**対応策**:
1. TechnologyProposalAgent が候補に Rust + Actix を含める
2. スコアリングに「採用リスク」軸を追加（例: 学習曲線の急峻さ = 3/10）
3. リスク分析に「Rust 採用の前提条件」を列挙

**出力例**:
```
Rust + Actix:
  ◎ パフォーマンス: 最高峰（10/10）
  ◎ メモリ安全性: 所有権システム（10/10）
  ×× 学習曲線: 3/10（3-6ヶ月の習得期間必須）
  ×× チーム採用: Rust エンジニアの採用困難度（8/10）

推奨条件:
  - 専任 Rust コンサルタント採用（3ヶ月）
  - または既存 Rust エンジニア 1名以上の確保
  - スケジュール: 初期遅延を許容（MVP +3ヶ月）
  - 想定コスト: 追加学習 $50K

判定: 「現時点では不推奨。ハイパフォーマンスが必須で予算・期間に余裕がある場合のみ候補」
```

### Case 4: 極端な予算制約
**状況**: 「年間インフラコスト 20万円以内」「開発予算も最小化」

**対応策**:
1. CostEstimatorAgent が価格の優先度を最高に設定
2. スコアリング表に「ランニングコスト」軸を追加（0-10点）
3. 推奨は「OSS + 自前運用」パターンを優先

**出力例**:
```
推奨: Python + Flask + SQLite + Heroku Free（月額 $0）
  ◎ コスト: 最小化（年額 ¥0 - ¥12万）
  ◎ 学習・採用: 容易
  ×× スケーラビリティ: 初期規模 < 1000ユーザー
  ×× パフォーマンス: 低リクエスト頻度向け

拡張計画:
  → ユーザー増加時に Python + Django + RDS への移行パス
  → 想定時期: 1-2年後（その時点で予算確保可能）
```

### Case 5: セキュリティ要件が極めて厳格
**状況**: 「HIPAA / GDPR / PCI-DSS 準拠必須」「単一国家管轄下でのデータ保管」

**対応策**:
1. RequirementAnalyzerAgent がセキュリティスコアを 10/10 に設定
2. TechnologyProposalAgent が「セキュリティ認定済み」技術のみ候補化
3. スコアリング表に「セキュリティフレームワーク対応」軸を追加

**出力例**:
```
推奨: Java + Spring Security + PostgreSQL + AWS GovCloud
  ◎ セキュリティ: HIPAA / GDPR / PCI-DSS 準拠フレームワーク搭載
  ◎ 監査: すべてのトランザクションをログ（HIPAA Audit Trail 要件達成）
  ◎ コンプライアンス: 年間 2 回の認定監査で合格実績
  ×× 開発速度: セキュリティ設定が複雑（学習 +1-2ヶ月）
  ×× コスト: 認定クラウド利用でランニングコスト高

必須付加機能:
  - セキュリティエンジニア常駐（運用）
  - 年間 2 回のセキュリティ監査契約
  - 暗号化キー管理 (AWS KMS / HashiCorp Vault)
```
