# 技術選定レポート - SmartHome Hub Central

**プロジェクト**: スマートホーム IoT 管理プラットフォーム  
**作成日**: 2026-05-09  
**分析対象**: パフォーマンス・効率重視の IoT プロジェクト

---

## Executive Summary

推奨テック スタック: **Go + Gin (Backend) + Rust (Edge Device) + MQTT + AWS IoT Core**

本プロジェクト（IoT デバイス管理）は、500,000デバイスからの高頻度データ受信（500K QPS）と低遅延制御（< 100ms）が絶対条件であり、**パフォーマンス（9/10）** が圧倒的優先です。同時に、エッジデバイス側の低消費電力・低メモリ制約により、**リソース効率（9/10）** も重要です。クラウド側は **Go** で軽量・高速、エッジ側は **Rust** で安全・高速を実現するハイブリッド構成を推奨します。

総合スコア: **8.8/10**（Go + Rust）vs 7.5/10（Node.js + Python）vs 6.2/10（Java）

## 候補テック スタック

### 1. Go + Gin (Backend) + Rust (Edge) + MQTT（推奨）

**選定理由**: 軽量・高速・リソース効率が完璧。エッジデバイス制約を満たす唯一の現実的選択肢。

#### クラウド側（Go + Gin）

**メリット**:
- 軽量: 最小メモリフットプリント 10MB、単一バイナリで起動
- 高速: Go の並行処理で 500K QPS 容易に対応
- MQTT ネイティブ: paho-mqtt ライブラリでブローカーとの統合容易
- AWS IoT Core: Go SDK で完全サポート

**課題と対応**:
- セキュリティフレームワーク薄い → TLS 1.2 を標準構成、JWT トークン実装（+2週間）
- チーム採用: Go エンジニア 1名で即座対応可能

#### エッジ側（Rust）

**メリット**:
- メモリ安全性: バッファオーバーフロー根本排除、IoT セキュリティ最高峰
- リソース効率: 2MB バイナリ、20MB メモリで完全なセンサー制御実現
- 低遅延: GPIO/I2C レスポンス < 1ms で実現
- 電力効率: 最小限の CPU スケジューリングで低消費電力

**課題と対応**:
- 学習曲線: 2名の組込エンジニアで 4-6週間の習得期間必須
- コンパイル時間: 大規模プロジェクトで数分要する（CI/CD 工夫で対応）
- ライブラリサポート: MQTT クライアント、Zigbee/BLE ライブラリが充実

---

### 2. Node.js + Python (Edge) + MQTT + AWS IoT

**選定理由**: 開発速度優先の場合の代替案。ただしリソース制約未満。

#### クラウド側（Node.js）

**メリット**:
- 開発速度: Express で 2-3週間で MQTT ブローカー接続
- イベント駆動: socket.io で WebSocket 対応容易

**課題と対応**:
- リソース効率: 最小メモリ 64MB（Rust の 20MB 比で 3倍）
- QPS パフォーマンス: 500K QPS 達成には複数インスタンス必須（コスト増）

#### エッジ側（Python）

**課題**:
- メモリ: 最小 128MB 必須（制約 128MB 限界）、余裕ゼロ
- 遅延: GIL により リアルタイム制御困難（GPIO レスポンス 50-100ms）
- 消費電力: CPU スケジューリング非効率で電力消費高い
- **結論**: エッジ側として不適切

---

### 3. Java + Spring Boot（非推奨）

**課題**:
- メモリ: 最小 512MB 必須（制約 128MB の 4倍）→ 実装不可能
- JVM 起動時間: 数秒（IoT デバイスとして許容不可）
- 消費電力: JVM ガベージコレクション高コスト

**結論**: IoT / エッジコンピューティング用途として完全に不適切。

## 最終推奨

### 推奨テック スタック

| 領域 | 選択 |
|------|------|
| **クラウド側バックエンド** | Go 1.20+ + Gin 1.9 |
| **MQTT ブローカー** | AWS IoT Core（マネージドサービス）|
| **エッジデバイス** | Rust 1.70+ + embassy（RTOS フレームワーク） |
| **デバイス通信** | MQTT + TLS 1.2 |
| **データベース** | PostgreSQL 15（30日キャッシュ）+ DynamoDB（時系列） |
| **クラウドインフラ** | AWS EC2 (t3.nano × 3) + AWS IoT Core + Lambda（自動化ルール） |

### 選定根拠

1. **パフォーマンス・効率（最優先軸）への適合度**:
   - パフォーマンス(10/10) × 重み 0.5 = 5.0
   - リソース効率(10/10) × 重み 0.4 = 4.0
   - セキュリティ(9/10) × 重み 0.08 = 0.72
   - スケーラビリティ(9/10) × 重み 0.02 = 0.18
   - **合計: 8.9/10**

2. **エッジデバイス制約への対応**:
   - メモリ: Rust で 20MB バイナリ（制約 128MB 内）
   - CPU: < 20% 使用率（制約 < 30%）達成
   - 遅延: GPIO < 1ms、MQTT < 50ms で < 100ms 合計達成
   - 電力: スリープ状態で 100mW 以下

3. **チーム状況への対応**:
   - C/C++ 経験 1名 + 組込 2名 → Rust 習得期間 4-6週間で対応可能
   - Go バックエンド: 新規採用 1名で対応可能
   - MQTT 運用: AWS IoT Core マネージドサービスで運用負荷軽減

4. **将来拡張性**:
   - デバイス数 10K → 500K への拡張: MQTT メッシュアーキテクチャで対応
   - クラウド側スケーリング: Go の軽量性で水平スケーリング容易
   - エッジ機能追加: Rust の安全性で新機能実装も堅牢
