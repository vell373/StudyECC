# 採点ルーブリック: Tech Selection Agent Group (Theme C)

## 概要

Theme C「技術選定エージェント群」の完成度を評価するための4つの採点軸。複数エージェントの協調による技術選定ハーネスの実装として、以下の品質を総合的に判定する。

- **要件充足度**: Must-Have要件の達成度
- **品質・構造**: エージェント設計・コード品質・責務分離の妥当性
- **完成度・動作性**: 実装の堅牢性・エッジケース対応・再現性
- **創造性・判断力**: 仕様解釈・設計判断・ユーザー体験向上への工夫

---

| **品質・構造** | 0-10 | エージェント定義が200行超過。命名規則混在。責務不明確。コード重複多数 | エージェント定義が150-200行。命名規則ほぼ統一。責務が大まかに分離。可読性は平均以下 | エージェント定義が100-150行。命名規則統一。責務明確に分離。可読性良好。若干の重複あり | エージェント定義が100行以内。命名規則完全統一。責務明確・疎結合。高い可読性。重複なし。コメント適切 | **完璧な設計**。エージェント定義100行以内で責務完全分離。「なぜこう設計したか」を説明するドキュメント。エージェント間インタフェースが明確で拡張容易。ハーネスの統合ロジックがシンプルで検証可能 |

## 採点チェックリスト

### 1. 要件充足度（10点）

**Must-Have要件（必須）**

- [ ] **エージェント実装**: 最低2体の専門エージェントが `.md` 形式で定義
  - [ ] 要件分析エージェント実装（スケーラビリティ・セキュリティ・パフォーマンス・開発速度の重要度判定ロジック存在）
  - [ ] 技術提案エージェント実装（言語・フレームワーク・DB の3-5セット候補提案ロジック存在）
  - [ ] 各エージェント定義が200行以下（簡潔性の要件）

- [ ] **ハーネス実装**: 複数エージェント結果の統合・意思決定機能
  - [ ] 複数エージェント結果の受け取り・統合ロジック実装
  - [ ] 統一スコアリング基準の実装（0-10点スケール）
  - [ ] 優位性マトリックス（要件軸 × 技術候補）の生成

- [ ] **トレードオフ分析**: 複数技術の優劣を明確に表現
  - [ ] 「技術A は◎だが××」形式での対立軸説明
  - [ ] トレードオフ分析の詳細度が適切（単なる列挙ではなく、「なぜか」を説明）

- [ ] **出力フォーマット**: 完全なMarkdownレポート生成
  - [ ] 候補テック スタック表（言語・フレームワーク・DB・クラウド毎に3案）
  - [ ] スコアリング表（要件軸 × 技術、0-10点採点）
  - [ ] トレードオフ分析セクション
  - [ ] 最終推奨テック スタック + 根拠

- [ ] **サンプル要件書**: 異なる優先軸のテストケース
  - [ ] サンプル要件書2件以上提供
  - [ ] 異なる優先軸（例：SaaS=スケーラビリティ、ヘルスケア=セキュリティ）
  - [ ] 異なる優先軸に対して、異なる推奨結果を出力

**Should-Have要件（強く推奨）**

- [ ] **3体目エージェント**: コスト見積もり or スコアリングエージェント実装
- [ ] **エージェント間協調**: 要件→提案→スコアリング→コストのパイプライン実装
- [ ] **複数業界対応**: SaaS・ヘルスケア・IoT等の判別ロジック
- [ ] **ドキュメント充実**: スコアリング評価基準・学習曲線推定・リスク分析

**Nice-to-Have要件（あると嬉しい）**

- [ ] マイグレーションパス分析実装
- [ ] OWASP等セキュリティフレームワーク連携
- [ ] 開発体験（DX）スコア実装
- [ ] AIモデル検討エージェント実装

### 2. 品質・構造（10点）

- [ ] **エージェント設計の明確性**
  - [ ] 各エージェントの責務が明確に記述されている
  - [ ] エージェント間のインタフェース（入力・出力）が明記されている
  - [ ] 責務の重複がない（疎結合）

- [ ] **ファイル構成と命名規則**
  - [ ] ファイル名が kebab-case で統一（例: `requirement-analyzer-agent.md`）
  - [ ] エージェント名が PascalCase で統一（例: `RequirementAnalyzerAgent`）
  - [ ] フォルダ構成が論理的（agents/, harness/, samples/ 等で整理）

- [ ] **コード品質**
  - [ ] エージェント定義が100-150行以内（200行超過で減点）
  - [ ] 重複コード・ロジックがない
  - [ ] コメントが「なぜ」を説明（「何を」ではなく）

- [ ] **エージェント定義の詳細性**
  - [ ] 判定ロジックが明確に記述（例：「スケール重要度は、『ユーザー数』『アクティブ同時接続数』から判定」）
  - [ ] 提案ロジックが透明性高い（「なぜこの3候補を選んだか」が説明可能）

- [ ] **ハーネスロジックの簡潔性**
  - [ ] スコアリング基準が明確・説明可能
  - [ ] トレードオフ分析ロジックが簡潔で理解しやすい
  - [ ] ハーネス内の判定が客観的（恣意的でない）

---

### 3. 完成度・動作性（10点）

- [ ] **実装の完成度**
  - [ ] 2体以上のエージェントが正常に動作
  - [ ] ハーネスが複数エージェント結果をマージできる
  - [ ] 推奨テック スタックが確実に出力される

- [ ] **エラーハンドリング**
  - [ ] 不正な入力（スペル誤字・不完全な要件書等）に対する例外処理
  - [ ] エラーメッセージが実用的（「何が問題か」「どう修正するか」を示唆）

- [ ] **エッジケース対応**
  - [ ] 要件矛盾（複数優先軸が同時に必須）→ 優先度付けで解決
  - [ ] レガシー移行（既存PHPからの移行等）→ 学習コスト考慮
  - [ ] 新興技術判定（Rust・Go等）→ リスク評価実施
  - [ ] 予算制約厳しい場合 → コスト優先の提案

- [ ] **再現性**
  - [ ] 同じ要件書を複数回実行 → 同じ推奨結果
  - [ ] 環境依存がない（APIキー等不要、またはダミーで動作可能）

- [ ] **テスト実施**
  - [ ] サンプル要件書2件以上で実行テスト完了
  - [ ] 異なる優先軸に対して、異なる推奨結果を出力確認
  - [ ] テスト結果がドキュメント化されている

---

### 4. 創造性・判断力（10点）

- [ ] **仕様の解釈・補完**
  - [ ] 仕様の曖昧な部分をどう埋めたか、説明がある
  - [ ] 「スコアリングの0-10点の定義」を独自に設定・説明
  - [ ] 「優位性マトリックス」の形式を工夫的に設計

- [ ] **スコアリングロジックの説得力**
  - [ ] 「なぜこの技術に10点か、5点か」が定量的・定性的に説明可能
  - [ ] スコアリングが業界標準（GitHub stars, npm downloads, セキュリティベストプラクティス等）に基づいている

- [ ] **トレードオフ分析の深さ**
  - [ ] 単なる「◎と××の列挙」ではなく、「どちらを選ぶべきか」の判断基準を示唆
  - [ ] 複数業界パターンでトレードオフが異なることを示現

- [ ] **ユーザー体験への工夫**
  - [ ] 「学習曲線の推定」「チームオンボーディング時間」等の付加価値情報
  - [ ] レポート形式が読みやすく、意思決定を支援する設計
  - [ ] 「この推奨に従った場合、3年後のリスクは」等の先読み分析

- [ ] **拡張性・再利用可能性**
  - [ ] 実装が特定プロジェクトに固有化していない（他の選定ケースに流用可能）
  - [ ] エージェント・ハーネスが「次のテーマに応用できそう」と感じさせる設計
  - [ ] Should-Have / Nice-to-Have要件の実装が基盤を強化している

## 採点方法

### ステップ1: 各軸の得点算出

各軸について、上表の基準に従って 0-10点を採点。

```
総合得点 = (要件充足度 + 品質・構造 + 完成度・動作性 + 創造性・判断力) / 4
```

### ステップ2: グレード判定

| 総合得点 | グレード | 評価 |
|---|---|---|
| 9-10 | S | 卓越している。そのまま本番導入可能 |
| 8-8.9 | A | 優秀。ごく微調整で完成 |
| 7-7.9 | B | 良好。要件充足だが改善余地あり |
| 6-6.9 | C | 及第点。必須要件は達成だが、品質・創造性で磨き必要 |
| 5-5.9 | D | 要改善。複数要件未充足または大きな設計問題あり |
| 0-4.9 | F | 不合格。要件充足が不十分 |

## 採点時の注意点

1. **要件充足度の厳密性**
   - "Must-Have" は達成必須。95%以上の達成が 8-9点。100%で 10点
   - Should-Have・Nice-to-Have の達成度は「創造性・判断力」に加算

2. **品質・構造の評価**
   - エージェント定義200行超過は大きな減点（設計の複雑化・複数エージェント協調の困難化を示唆）
   - 「責務の分離」が最重要。1体のエージェントで複数役割を担当していないか確認

3. **完成度・動作性の検証**
   - 「同じ要件で同じ推奨結果」を実際に実行確認することが重要
   - サンプル2件が「本当に異なる優先軸」であることを確認

4. **創造性・判断力の加点**
   - 「ここまで工夫は要求していない」という実装が多いほど高評価
   - 特にスコアリングロジックの透明性とトレードオフ分析の深さが重要
