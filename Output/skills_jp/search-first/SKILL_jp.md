# Search-First ワークフロー

コードを書く前にリサーチする。車輪の再発明を避ける。

## コアの考え方

実装を始める前に、既存のソリューションが問題の大部分（またはすべて）を解決できないかを確認する。

## 5ステップのワークフロー

### ステップ1: ニーズの分析

達成したいことを正確に定義する:
- 解決すべき問題は何か？
- 必須の要件は何か？
- 妥当なトレードオフは何か？
- 非機能要件（パフォーマンス、セキュリティ、ライセンス）は何か？

### ステップ2: 並列検索

複数の戦略で同時に検索する:

```
並列で検索する:
├─→ パッケージレジストリ（npm/PyPI/crates.io）
├─→ GitHubで類似プロジェクトを検索
├─→ 既存コードベースで関連実装を確認
└─→ ドキュメントでビルトイン機能を確認
```

**検索のポイント:**
- 完全な解決策だけでなく、部分的な解決策も探す
- 積極的にメンテナンスされているか確認する（最終コミット日）
- スター数、ダウンロード数、issueの解決率を確認する
- ライセンスの互換性を確認する

### ステップ3: 評価

各候補を評価する:

| 基準 | 確認事項 |
|------|---------|
| 機能的な適合 | 要件の何%をカバーしているか？ |
| メンテナンス状況 | 積極的にメンテナンスされているか？ |
| バンドルサイズ | 依存関係の重さ |
| ドキュメント | 品質と完全性 |
| セキュリティ | 既知の脆弱性 |
| ライセンス | プロジェクトとの互換性 |

### ステップ4: 意思決定

決定マトリクス:

```
Adopt（採用）  — 要件の80%以上をカバーする既存のソリューション
Extend（拡張）  — 要件の50-80%をカバーするが、カスタマイズが必要
Compose（組み合わせ）— 複数の小さなライブラリを組み合わせて解決
Build（自作）  — 既存のソリューションでは不十分な場合のみ
```

**Buildを選ぶ正当な理由:**
- 固有のビジネスロジックが必要
- 既存ライブラリがパフォーマンス要件を満たさない
- ライセンスの制約がある
- セキュリティ上の要件で外部依存が許可されない

### ステップ5: 実装

意思決定に基づいて実装する:
- **Adopt**: ドキュメントに従ってセットアップ
- **Extend**: ライブラリを採用し、必要な拡張を実装
- **Compose**: 各ライブラリのAPIを理解してから統合
- **Build**: 明確な仕様から始めてTDDで実装

## 実践例

### 例: JWT認証の実装

**ステップ1のニーズ分析:**
- RS256署名によるJWT生成・検証が必要
- Node.js + TypeScriptの環境
- 有効期限の処理が必要
- セキュリティが重要

**ステップ2の検索結果:**
- `jsonwebtoken` — 500Kダウンロード/週、良好なメンテナンス
- `jose` — より現代的、W3C標準準拠
- `fast-jwt` — パフォーマンス重視
- Node.js組み込みの`crypto` — 低レベルだがJWT抽象化なし

**ステップ3の評価:**
- `jsonwebtoken`: 要件の100%をカバー、実績あり ✓
- `jose`: 要件の100%をカバー、Web Crypto API対応 ✓
- 自作: セキュリティリスク高、正当な理由なし ✗

**ステップ4の意思決定:**
→ `jose` を Adopt（現代的なWeb Crypto API対応）

**ステップ5の実装:**
```typescript
import { SignJWT, jwtVerify } from 'jose';
// ...
```

## よくある間違い

- **すぐに自作する** — 既存のソリューションを確認せずに実装を始める
- **最初に見つけたものを採用する** — 比較せずに最初の結果を使う
- **メンテナンス状況を無視する** — 2年以上更新されていないライブラリを採用する
- **ライセンスを無視する** — GPL-3.0ライブラリを商用プロジェクトに組み込む
