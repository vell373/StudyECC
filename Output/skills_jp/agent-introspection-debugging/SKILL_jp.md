# エージェントの自己内省デバッグ

エージェントの実行が繰り返し失敗している、トークンを消費しても進捗がない、同じツールでループしている、または意図したタスクから逸脱しているときにこのスキルを使う。

これはワークフロースキルであり、隠されたランタイムではない。人間にエスカレーションする前に、エージェントが体系的に自己デバッグするための方法を教える。

## 起動するタイミング

- ツール呼び出し回数の上限 / ループ制限での失敗
- 前進なしに繰り返されるリトライ
- 出力品質を劣化させるコンテキスト肥大またはプロンプトドリフト
- 想定と現実の間でのファイルシステムまたは環境の状態不一致
- 診断と小さな修正アクションで回復できる可能性のあるツール障害

## スコープの境界

このスキルを起動するのは以下の場合:
- やみくもにリトライする前に障害状態を記録する
- エージェント固有の一般的な障害パターンを診断する
- 封じ込めた回復アクションを適用する
- 人間が読めるデバッグレポートを構造的に生成する

以下の目的でこのスキルを主要ソースとして使用しないこと:
- コード変更後の機能検証; `verification-loop` を使う
- より絞り込まれた ECC スキルが既に存在するフレームワーク固有のデバッグ
- 現在のハーネスが自動的に強制できないランタイムの約束

## 4フェーズループ

### フェーズ 1: 障害の記録

回復を試みる前に、障害を正確に記録する。

記録内容:
- エラーの種類、メッセージ、スタックトレース（利用可能な場合）
- 最後の意味のあるツール呼び出しシーケンス
- エージェントが実行しようとしていたこと
- 現在のコンテキスト負荷: 繰り返されるプロンプト、過大に貼り付けたログ、重複した計画、または暴走したノート
- 現在の環境の前提: cwd、ブランチ、関連するサービス状態、期待されるファイル

最小記録テンプレート:

```markdown
## 障害の記録
- セッション / タスク:
- 進行中のゴール:
- エラー:
- 最後に成功したステップ:
- 最後に失敗したツール / コマンド:
- 確認された繰り返しパターン:
- 確認すべき環境の前提:
```

### フェーズ 2: 根本原因の診断

何かを変更する前に、既知のパターンに障害を照合する。

| パターン | 考えられる原因 | 確認方法 |
| --- | --- | --- |
| ツール呼び出し回数の上限 / 同じコマンドの繰り返し | ループまたは出口のないオブザーバーパス | 最後のN回のツール呼び出しで繰り返しを確認 |
| コンテキストオーバーフロー / 推論の劣化 | 制限のないノート、繰り返されるプラン、過大なログ | 重複と低シグナルの塊について最近のコンテキストを確認 |
| `ECONNREFUSED` / タイムアウト | サービス利用不可または間違ったポート | サービスの健全性、URL、ポートの前提を確認 |
| `429` / クォータ枯渇 | リトライストームまたはバックオフの不足 | 繰り返された呼び出しを数え、リトライ間隔を確認 |
| 書き込み後にファイルが見つからない / 古い差分 | 競合状態、間違った cwd、またはブランチドリフト | パス、cwd、git status、実際のファイル存在を再確認 |
| 「修正」後もテストが失敗 | 間違った仮説 | 具体的に失敗しているテストを特定し、バグを再導出する |

診断の質問:
- これはロジック障害、状態障害、環境障害、それともポリシー障害か？
- エージェントは実際の目標を失い、間違ったサブタスクを最適化し始めているか？
- 障害は決定論的か、それとも一時的か？
- 診断を検証する最小の可逆アクションは何か？

### フェーズ 3: 封じ込めた回復

診断の表面を変える最小のアクションで回復する。

安全な回復アクション:
- 繰り返しのリトライを停止し、仮説を言い直す
- 低シグナルのコンテキストを削除し、アクティブなゴール、ブロッカー、および証拠のみを保持する
- 実際のファイルシステム / ブランチ / プロセスの状態を再確認する
- タスクを1つの失敗しているコマンド、1つのファイル、または1つのテストに絞り込む
- 推測的な推論から直接観察に切り替える
- 障害が高リスクまたは外部でブロックされている場合は人間にエスカレーションする

現在の環境で実際のツールを通じて実行していない限り、「エージェント状態のリセット」や「ハーネス設定の更新」のようなサポートされない自動修復アクションを主張しないこと。

封じ込め回復チェックリスト:

```markdown
## 回復アクション
- 選択した診断:
- 実行した最小アクション:
- これが安全な理由:
- 修正が機能したことを証明する証拠:
```

### フェーズ 4: 内省レポート

次のエージェントまたは人間が回復内容を理解できるレポートで終了する。

```markdown
## エージェント自己デバッグレポート
- セッション / タスク:
- 障害:
- 根本原因:
- 回復アクション:
- 結果: 成功 | 部分的 | ブロック
- トークン / 時間消費リスク:
- 必要なフォローアップ:
- 後でエンコードする予防的変更:
```

## 回復ヒューリスティック

以下の順序で介入を優先する:

1. 実際の目標を1文で言い直す。
2. 記憶を信頼するのではなく、世界の状態を確認する。
3. 失敗しているスコープを縮小する。
4. 1つの識別チェックを実行する。
5. その後でのみリトライする。

悪いパターン:
- 少し異なる言い回しで同じアクションを3回リトライする

良いパターン:
- 障害を記録する
- パターンを分類する
- 1つの直接チェックを実行する
- チェックが支持する場合にのみ計画を変更する

## ECC との統合

- コードが変更された場合は回復後に `verification-loop` を使用する。
- 障害パターンがインスティンクトや後のスキルにする価値がある場合は `continuous-learning-v2` を使用する。
- 問題が技術的な障害ではなく意思決定の曖昧さである場合は `council` を使用する。
- 障害が競合するローカル状態またはリポジトリドリフトから発生した場合は `workspace-surface-audit` を使用する。

## 出力標準

このスキルがアクティブな場合、「修正しました」だけで終わらないこと。

常に以下を提供する:
- 障害パターン
- 根本原因の仮説
- 回復アクション
- 状況が改善されたか、またはまだブロックされていることの証拠
