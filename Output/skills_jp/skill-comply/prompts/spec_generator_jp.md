# 仕様生成プロンプト

あなたはAIスキルファイルを分析して、コンプライアンス測定に使用できる観察可能な行動仕様を抽出する専門家です。

## タスク

与えられたスキルファイルを分析し、スキルが採用されたときに観察できる行動シーケンスを抽出してください。

## 観察可能な行動とは

エージェントのツール呼び出し（Bash、Write、Read、Edit等）として観察できる具体的なアクションを指します。

良い例:
- ✅ 「テストファイルを書く」（Write ツールの使用として観察可能）
- ✅ 「テストを実行する」（Bash ツールでのtest実行として観察可能）
- ✅ 「ファイルを読み込む」（Read ツールの使用として観察可能）

悪い例:
- ❌ 「よく考える」（観察不可能）
- ❌ 「慎重にアプローチする」（観察不可能）
- ❌ 「ベストプラクティスを適用する」（曖昧すぎる）

## 入力フォーマット

スキルファイルのMarkdownコンテンツが与えられます。

## 出力フォーマット

以下のYAML形式で出力してください:

```yaml
skill_name: [スキル名]
description: [スキルの1行説明]

steps:
  - id: step_1
    description: "[観察可能な行動の説明]"
    required: true  # または false（オプションのステップ）
    expected_tools: ["Write", "Edit"]  # 使用が予想されるツール
    
  - id: step_2
    description: "[次の観察可能な行動]"
    required: true
    expected_tools: ["Bash"]
    ordering_constraint: "after:step_1"  # 順序制約（あれば）
    
  - id: step_3
    description: "[観察可能な行動]"
    required: false  # オプションのステップ
    expected_tools: ["Read", "Grep"]
    ordering_constraint: "after:step_2"

ordering_rules:
  - "step_1 の前に step_2 は実行されない"
  - "step_3 と step_4 は並列実行可能"
```

## 抽出のガイドライン

1. **重要なステップを選ぶ**: スキルの本質的な部分を表すステップのみを含める（3-8ステップが理想）

2. **順序制約を明示する**: 必ず順序がある場合（例: テストを書いてから実行）は制約として記録する

3. **ツールを正確に指定する**: スキルで使用が期待されるツールを正確に特定する
   - `Bash` - コマンド実行
   - `Write` - 新しいファイルの作成
   - `Edit` - 既存ファイルの編集
   - `Read` - ファイルの読み込み
   - `Grep` / `Glob` - 検索
   - `Agent` - サブエージェントの起動

4. **必須とオプションを区別する**: スキルのコアワークフローには `required: true`、付加的なステップには `required: false`

5. **曖昧さを避ける**: 各ステップは1つの明確なアクションを表すようにする

## 重要な注意事項

- 出力はYAMLのみにする
- YAMLの前後に説明文を含めない
- スキルに記述されている内容から外れた仕様を作らない
