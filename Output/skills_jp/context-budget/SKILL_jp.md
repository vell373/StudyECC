# コンテキストバジェット

Claude Code セッションで読み込まれるすべてのコンポーネントのトークンオーバーヘッドを分析し、コンテキストスペースを節約するための実行可能な最適化案を提示する。

## いつ使うか

- セッションのパフォーマンスが低下したり、出力品質が落ちてきた場合
- スキル・エージェント・MCP サーバーを多数追加した後
- 実際にどれだけのコンテキスト余裕があるかを知りたい場合
- さらにコンポーネントを追加する前に余裕を確認したい場合
- `/context-budget` コマンドを実行する場合（このスキルがそれをバックアップする）

## 動作の仕組み

### フェーズ 1: インベントリ

すべてのコンポーネントディレクトリをスキャンしてトークン消費量を推計する。

**エージェント** (`agents/*.md`)
- ファイルごとの行数とトークン数（単語数 × 1.3）を数える
- フロントマターの `description` の長さを抽出する
- フラグ: 200 行超（重い）、説明 30 語超（フロントマターが肥大化）

**スキル** (`skills/*/SKILL.md`)
- SKILL.md ごとのトークン数を数える
- フラグ: 400 行超
- `.agents/skills/` の重複コピーを確認し、同一ファイルは二重カウントしない

**ルール** (`rules/**/*.md`)
- ファイルごとのトークン数を数える
- フラグ: 100 行超
- 同じ言語モジュール内のルールファイルで内容が重複していないか検出する

**MCP サーバー** (`.mcp.json` またはアクティブな MCP 設定)
- 設定済みサーバーとツール数の合計を数える
- スキーマオーバーヘッドをツール 1 件あたり約 500 トークンと推計する
- フラグ: ツール 20 件超のサーバー、シンプルな CLI コマンドをラップするサーバー（`gh`、`git`、`npm`、`supabase`、`vercel`）

**CLAUDE.md**（プロジェクトおよびユーザーレベル）
- CLAUDE.md チェーン内の各ファイルのトークン数を数える
- フラグ: 合計 300 行超

### フェーズ 2: 分類

すべてのコンポーネントをバケットに分類する。

| バケット | 基準 | アクション |
|--------|----------|--------|
| **常に必要** | CLAUDE.md で参照済み、アクティブなコマンドのバックアップ、または現在のプロジェクトタイプに合致 | 保持 |
| **時々必要** | ドメイン固有（例: 言語パターン）、CLAUDE.md で未参照 | オンデマンド起動を検討 |
| **ほぼ不要** | コマンド参照なし、コンテンツが重複、プロジェクトに合致しない | 削除またはレイジーロード |

### フェーズ 3: 問題の検出

以下の問題パターンを特定する。

- **エージェント説明の肥大化** — フロントマターの説明が 30 語超になると、すべての Task ツール起動時に読み込まれる
- **重いエージェント** — 200 行超のファイルは、spawn のたびに Task ツールコンテキストを膨らませる
- **冗長なコンポーネント** — エージェントのロジックと重複するスキル、CLAUDE.md と重複するルール
- **MCP 過剰サブスクリプション** — サーバー 10 件超、または無料で使える CLI ツールをラップするサーバー
- **CLAUDE.md の肥大化** — 冗長な説明、陳腐化したセクション、ルールに移すべき指示

### フェーズ 4: レポート

コンテキストバジェットレポートを出力する。

```
コンテキストバジェットレポート
═══════════════════════════════════════

推定オーバーヘッド合計: ~XX,XXX トークン
コンテキストモデル: Claude Sonnet（200K ウィンドウ）
有効な利用可能コンテキスト: ~XXX,XXX トークン（XX%）

コンポーネント内訳:
┌─────────────────┬────────┬───────────┐
│ コンポーネント  │ 件数   │ トークン  │
├─────────────────┼────────┼───────────┤
│ エージェント    │ N      │ ~X,XXX    │
│ スキル          │ N      │ ~X,XXX    │
│ ルール          │ N      │ ~X,XXX    │
│ MCP ツール      │ N      │ ~XX,XXX   │
│ CLAUDE.md       │ N      │ ~X,XXX    │
└─────────────────┴────────┴───────────┘

警告: 問題が見つかりました（N 件）:
[トークン節約量の順にランク付け]

上位 3 つの最適化:
1. [アクション] → 約 X,XXX トークン節約
2. [アクション] → 約 X,XXX トークン節約
3. [アクション] → 約 X,XXX トークン節約

潜在的な節約量: ~XX,XXX トークン（現在のオーバーヘッドの XX%）
```

詳細モードでは、さらに以下を出力する。ファイルごとのトークン数、最もヘビーなファイルの行ごとの内訳、重複コンポーネント間の具体的な重複行、MCP ツールリストとツールごとのスキーマサイズ推計。

## 例

**基本的な監査**
```
ユーザー: /context-budget
スキル: セットアップをスキャン → 16 エージェント（12,400 トークン）、28 スキル（6,200）、87 MCP ツール（43,500）、2 CLAUDE.md（1,200）
       フラグ: 重いエージェント 3 件、MCP サーバー 14 件（3 件は CLI で代替可能）
       節約のトップ候補: MCP サーバー 3 件削除 → -27,500 トークン（オーバーヘッドを 47% 削減）
```

**詳細モード**
```
ユーザー: /context-budget --verbose
スキル: 全レポート + ファイルごとの内訳。planner.md（213 行、1,840 トークン）、
       MCP ツールリストとツールごとのサイズ、重複ルールの行を並べて表示
```

**拡張前の確認**
```
ユーザー: MCP サーバーをあと 5 件追加したいが、余裕はあるか？
スキル: 現在のオーバーヘッド 33% → 5 件追加（約 50 ツール）で約 25,000 トークン増 → オーバーヘッドが 45% に
       推奨: まず CLI 代替可能なサーバー 2 件を削除して 40% 以内に収めること
```

## ベストプラクティス

- **トークン推計**: 散文は `単語数 × 1.3`、コードが多いファイルは `文字数 / 4`
- **MCP が最大のレバー**: ツールスキーマ 1 件で約 500 トークン。30 ツールのサーバーはスキル全体より高コスト
- **エージェントの説明は常に読み込まれる**: 一度も呼ばれないエージェントでも、その `description` フィールドはすべての Task ツールコンテキストに存在する
- **詳細モードはデバッグ用**: オーバーヘッドを引き起こしているファイルを特定するのに使う。通常の監査には不要
- **変更後は必ず監査する**: エージェント・スキル・MCP サーバーを追加するたびに実行してクリープを早期発見する
