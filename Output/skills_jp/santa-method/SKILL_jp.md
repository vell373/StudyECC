# Santaメソッド

二重の独立レビューによるマルチエージェント敵対的検証フレームワーク。

## 動作原理

```
フェーズ1: 生成
  └─→ メインエージェントが成果物を作成する

フェーズ2: 二重独立レビュー（並列）
  ├─→ レビュアーA: 批判的視点で評価
  └─→ レビュアーB: 批判的視点で評価（独立）

フェーズ3: 判定ゲート
  ├─→ 両方PASS → 完了
  └─→ いずれかFAIL → フェーズ4へ

フェーズ4: Nice修正ループ（最大MAX_ITERATIONS=3回）
  ├─→ 両レビュアーのフィードバックを統合する
  ├─→ メインエージェントが修正する
  └─→ フェーズ2に戻る
```

## 使用するタイミング

- 本番環境へのデプロイ前のコードレビュー
- セキュリティ上重要な変更の検証
- アーキテクチャの意思決定の評価
- 高リスクのリファクタリング
- テスト品質の検証
- ドキュメントの正確性確認

## 実装

### エージェントプロンプト

**レビュアーAのプロンプト:**
```
あなたは厳格なコードレビュアーです。以下の成果物を批判的に評価してください。
バグ、セキュリティ上の問題、パフォーマンスの問題、設計上の欠陥を探してください。
問題がなければPASS、問題があればFAILと判定し、具体的なフィードバックを提供してください。
```

**レビュアーBのプロンプト:**
```
あなたは独立したコード品質専門家です。以下の成果物を別の視点から評価してください。
保守性、テスト可能性、命名規則、エッジケースを重点的に確認してください。
問題がなければPASS、問題があればFAILと判定し、具体的な改善点を提供してください。
```

### ワークフロー実装

```python
MAX_ITERATIONS = 3

async def santa_method(artifact):
    for iteration in range(MAX_ITERATIONS):
        # フェーズ2: 並列レビュー
        review_a, review_b = await asyncio.gather(
            reviewer_a.evaluate(artifact),
            reviewer_b.evaluate(artifact)
        )
        
        # フェーズ3: 判定ゲート
        if review_a.verdict == "PASS" and review_b.verdict == "PASS":
            return {"status": "approved", "artifact": artifact, "iterations": iteration + 1}
        
        # フェーズ4: フィードバックを統合して修正
        combined_feedback = merge_feedback(review_a, review_b)
        artifact = await main_agent.revise(artifact, combined_feedback)
    
    # 最大イテレーション到達
    return {"status": "needs_human_review", "artifact": artifact, "feedback": combined_feedback}
```

## 判定基準

### PASSの条件
- 機能要件を満たしている
- セキュリティ上の問題がない
- 明らかなバグや論理エラーがない
- コードスタイルと規約に準拠している

### FAILの条件
- バグまたは論理エラーがある
- セキュリティの脆弱性がある
- パフォーマンスの重大な問題がある
- 要件が満たされていない
- 保守性に重大な問題がある

## 出力フォーマット

```

イテレーション: [実行回数]
最終判定: [APPROVED / NEEDS_HUMAN_REVIEW]

レビュアーA:
- 判定: PASS / FAIL
- 主なコメント: [フィードバック]

レビュアーB:
- 判定: PASS / FAIL
- 主なコメント: [フィードバック]

[各イテレーションの修正概要]
```

## ヒント

- **独立性の維持**: レビュアーAとBは互いのフィードバックを見ない（フェーズ2では）
- **フィードバックの統合**: フェーズ4では両方のフィードバックを統合して矛盾を解決する
- **エスカレーション**: 3回のイテレーション後も両方PASSしない場合は人間のレビューに回す
- **ドメイン特化**: レビュアーのプロンプトをドメイン（セキュリティ、パフォーマンス等）に特化させるとより効果的
