# プランコマンド

このコマンドはコードを書く前に、包括的な実装プランを作成します。

デフォルトではインラインで実行します。Taskツールまたはサブエージェントをデフォルトでは呼び出しません。これにより、エージェントファイルがないプラグインインストールから提供されるコマンドでも `/plan` が使用可能になります。

## このコマンドの機能

1. **要件を再度説明** - 構築が必要な内容を明確にします
2. **リスクを特定** - 潜在的な問題とブロッカーを表面化させます
3. **ステップバイステッププランを作成** - 実装をフェーズに分割します
4. **確認を待つ** - 進める前に必ずユーザー承認を受ける必要があります

## 使用時

以下の場合に `/plan` を使用してください:
- 新しい機能を開始する
- 大規模なアーキテクチャ変更を行う
- 複雑なリファクタリングに取り組む
- 複数のファイル/コンポーネントが影響を受ける
- 要件が不明確またはあいまい

## 動作方法

アシスタントは以下を実行します:

1. **リクエストを分析**し、明確な用語で要件を再度説明する
2. **フェーズに分割**し、具体的で実行可能なステップを提供する
3. **コンポーネント間の依存関係を特定**する
4. **リスクを評価**し、潜在的なブロッカーを特定する
5. **複雑性を推定**（高/中/低）
6. **プランを提示**し、明示的な確認を待つ

## 使用例

```
ユーザー: /plan マーケットが解決されるときにリアルタイム通知を送る必要があります

アシスタント:
# 実装プラン: リアルタイムマーケット解決通知

## 要件の再度説明
- ユーザーが監視しているマーケットが解決されるときに通知を送信する
- 複数の通知チャネルをサポート（アプリ内、メール、Webhook）
- 通知が確実に配信されるようにする
- マーケット成果とユーザーのポジション結果を含める

## 実装フェーズ

### フェーズ1: データベーススキーマ
- notifications テーブルを追加: id、user_id、market_id、type、status、created_at カラム付き
- user_notification_preferences テーブルをチャネルプリファレンス用に追加
- パフォーマンス向けに user_id と market_id にインデックスを作成

### フェーズ2: 通知サービス
- lib/notifications.ts に通知サービスを作成
- BullMQ/Redis を使用して通知キューを実装
- 失敗した配信に再試行ロジックを追加
- 通知テンプレートを作成

### フェーズ3: 統合ポイント
- マーケット解決ロジックにフック（ステータスが「解決済み」に変更されるとき）
- マーケットにポジションがあるすべてのユーザーをクエリ
- 各ユーザー向けに通知をキューイング

### フェーズ4: フロントエンドコンポーネント
- ヘッダーに NotificationBell コンポーネントを作成
- NotificationList モーダルを追加
- Supabase サブスクリプション経由でリアルタイム更新を実装
- 通知プリファレンスページを追加

## 依存関係
- Redis（キュー用）
- メールサービス（SendGrid/Resend）
- Supabase リアルタイムサブスクリプション

## リスク
- 高: メール配信可能性（SPF/DKIM必須）
- 中: マーケットあたり1000+ユーザーでのパフォーマンス
- 中: マーケットが頻繁に解決された場合の通知スパム
- 低: リアルタイムサブスクリプションのオーバーヘッド

## 推定複雑性: 中
- バックエンド: 4-6時間
- フロントエンド: 3-4時間
- テスト: 2-3時間
- 合計: 9-13時間

**確認待ち**: このプランで進めますか?（はい/いいえ/変更）
```

## 重要な注意事項

**重大**: このコマンドは、あなたが「はい」や「進める」などの肯定的な応答でプランを明示的に確認するまで、**コードを書きません**。

変更したい場合は、以下のように応答してください:
- "変更: [あなたの変更]"
- "別のアプローチ: [代替案]"
- "フェーズ2をスキップしてフェーズ3を最初に実行"

## 他のコマンドとの統合

計画後:
- TDD ワークフロー用に `tdd-workflow` スキルを使用
- ビルドエラーが発生した場合は `/build-fix` を使用
- 実装完了後に `/code-review` を使用

> **より深い計画が必要ですか?** PRD統合、コードベース分析、パターン抽出を備えた artifact 生成計画用に `/prp-plan` を使用してください。これらのプランを厳密な検証ループで実行するには `/prp-implement` を使用してください。

## オプション: Planner エージェント

ECC はエージェントファイルを含む手動インストール用の `planner` エージェントも提供します。ローカルランタイムがそのサブエージェントをすでに公開していて、ユーザーが明示的に計画を委譲するよう求める場合にのみ使用してください。

`planner` サブエージェントが利用不可の場合、「Agent type 'planner' not found」エラーを出すのではなく、代わりにインラインで計画を続けてください。

手動インストールでは、ソースファイルは以下の場所にあります:
`agents/planner.md`
