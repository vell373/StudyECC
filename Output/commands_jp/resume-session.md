# セッション再開コマンド

最後に保存したセッション状態を読み込み、作業開始前に完全に把握します。
このコマンドは `/save-session` のカウンターパートです。

## 使用時

- 前の日から続行するために新しいセッションを開始する
- コンテキスト制限のため新しいセッションを開始した後
- 別のソースからセッションファイルをハンドオフするときに（ファイルパスを指定）
- セッションファイルがあり、進める前に Claude に完全に吸収させたいときはいつでも

## 使用方法

```
/resume-session                                                      # ~/.claude/session-data/ の最新ファイルを読み込む
/resume-session 2024-01-15                                           # その日の最新セッションを読み込む
/resume-session ~/.claude/session-data/2024-01-15-abc123de-session.tmp  # 特定の短ID セッションファイルを読み込む
/resume-session ~/.claude/sessions/2024-01-15-session.tmp               # 特定のレガシー形式ファイルを読み込む
```

## プロセス

### ステップ1: セッションファイルを検出

引数が提供されない場合:

1. `~/.claude/session-data/` をチェック
2. 最も最近修正された `*-session.tmp` ファイルを選択
3. フォルダが存在しないか、マッチするファイルがない場合、ユーザーに通知:
   ```
   ~/.claude/session-data/ にセッションファイルが見つかりません
   セッションの終了時に /save-session を実行してセッションファイルを作成してください。
   ```
   その後停止。

引数が提供された場合:

- 日付のように見える場合（`YYYY-MM-DD`）、`~/.claude/session-data/` をまず検索し、その後 legacy `~/.claude/sessions/` を検索して、その日付にマッチする `YYYY-MM-DD-session.tmp`（レガシー形式）または `YYYY-MM-DD-<shortid>-session.tmp`（現在の形式）を検出し、その日付の最も最近修正されたバリアントを読み込む
- ファイルパスのように見える場合、そのファイルを直接読み込む
- 見つからない場合、明確に報告して停止

### ステップ2: セッションファイル全体を読み込む

ファイル全体を読み込む。まだサマリーしない。

### ステップ3: 理解を確認

この正確なフォーマットで構造化されたブリーフィングで応答:

```
セッション読み込み: [ファイルの実際の解決されたパス]
════════════════════════════════════════════════

プロジェクト: [プロジェクト名 / ファイルのトピック]

構築中のもの:
[2-3 文のサマリーを自分の言葉で]

現在の状態:
成功: 動作確認済み: [カウント] 項目
 進行中: [進行中のファイルのリスト]
 未開始: [計画済みだが未着手]

再試行しないこと:
[失敗したアプローチとその理由をすべてリストアップ — これは重要]

未解決の質問・ブロッカー:
[ブロッカーまたは未回答の質問をリスト]

次のステップ:
[ファイルで定義されている場合は正確な次のステップ]
[定義されていない場合: 「次のステップが定義されていません — 開始前に『未試行のアプローチ』セクションを一緒に確認することをお勧めします」]

════════════════════════════════════════════════
続行する準備ができています。何をしたいですか?
```

### ステップ4: ユーザーを待つ

自動的に作業を開始しない。ファイルに触れない。ユーザーが次に何をするか言うのを待つ。

セッションファイルで次のステップが明確に定義されており、ユーザーが「継続」「はい」などと同様に言う場合 — その正確な次のステップで進める。

次のステップが定義されていない場合 — ユーザーに開始場所を尋ね、オプションで「未試行のアプローチ」セクションからのアプローチを提案する。

## エッジケース

**同じ日付の複数セッション**（`2024-01-15-session.tmp`, `2024-01-15-abc123de-session.tmp`）:
レガシー ID なし形式と現在の短ID形式を問わず、その日付にマッチする最も最近修正されたファイルを読み込む。

**セッションファイルがもはや存在しないファイルを参照する:**
ブリーフィング中にこれを注記 — 「警告: `path/to/file.ts` はセッションで参照されていますが、ディスク上に見つかりません。」

**セッションファイルが 7日以上前の場合:**
ギャップを注記 — 「警告: このセッションは N 日前のもの（閾値: 7日）。状況が変わっているかもしれません。」 — その後、通常通り進める。

**ユーザーがファイルパスを直接提供する（例：チームメイトから転送）:**
それを読み込み、同じブリーフィングプロセスに従う — ソースに関わらずフォーマットは同じ。

**セッションファイルが空か不正な形式:**
レポート: 「セッションファイルが見つかりましたが、空またはしてください。新しいものを /save-session で作成する必要があるかもしれません。」

## 注記

- セッションファイルを読み込むときに修正しない — これは読み取り専用の履歴記録
- ブリーフィングフォーマットは固定 — セクションが空でも省略しない
- 「再試行しないこと」は常に表示（「なし」と言っても） — これは見落とすには重要すぎる
- 再開後、ユーザーは新しいセッションの終了時に `/save-session` を再度実行して新しい日付付きファイルを作成したいかもしれない
