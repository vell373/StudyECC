# ビルドと修正

ビルドと型エラーを最小限のセーフな変更で段階的に修正。

## ステップ1：ビルドシステムを検出

プロジェクトのビルドツールを特定して、ビルドを実行:

| 指標 | ビルドコマンド |
|-----------|---------------|
| `package.json`に`build`スクリプト | `npm run build`または`pnpm build` |
| `tsconfig.json`（TypeScriptのみ） | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build 2>&1` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall -q .`または`mypy .` |

## ステップ2：エラーを解析してグループ化

1. ビルドコマンドを実行してstderrをキャプチャ
2. ファイルパス別にエラーをグループ化
3. 依存順に並べ替え（インポート/型エラーをロジックエラーの前に修正）
4. 進捗追跡用に総エラー数をカウント

## ステップ3：修正ループ（一度に1つのエラー）

各エラーについて:

1. **ファイルを読む** — Read toolを使用してエラーコンテキストを確認（エラー周辺10行）
2. **診断する** — 根本原因を識別（欠落したインポート、間違った型、構文エラー）
3. **最小限に修正** — Edit toolを使用してエラーを解決する最小の変更
4. **ビルドを再実行** — エラーが消え、新しいエラーが導入されていないことを確認
5. **次へ移動** — 残りのエラーで続行

## ステップ4：ガードレール

以下の場合は停止してユーザーに聞く:
- 修正が**修正より多くのエラーをもたらす**
- **同じエラーが3試行後でも続く**（より深い問題の可能性）
- 修正が**アーキテクチャの変更**を必要とする（ビルド修正だけではない）
- ビルドエラーが**欠落した依存関係**に由来する（`npm install`、`cargo add`等が必要）

## ステップ5：サマリー

結果を表示:
- 修正したエラー（ファイルパス付き）
- 残りのエラー（ある場合）
- 導入された新しいエラー（ゼロ であるべき）
- 未解決の問題に対する提案された次のステップ

## リカバリ戦略

| 状況 | アクション |
|-----------|--------|
| 欠落したモジュール/インポート | パッケージがインストールされているかチェック; インストールコマンドを提案 |
| 型の不一致 | 両方の型定義を読む; より狭い型を修正 |
| 循環依存 | インポートグラフで循環を特定; 抽出を提案 |
| バージョン競合 | `package.json`/`Cargo.toml`のバージョン制約をチェック |
| ビルドツールの誤設定 | 設定ファイルを読む; 動作中のデフォルトと比較 |

安全のため、一度に1つのエラーを修正。リファクタリングよりも最小diffを優先。
