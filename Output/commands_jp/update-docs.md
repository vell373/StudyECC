# ドキュメントを更新

コードベースと同期してドキュメントを更新し、信頼できるソースファイルから生成します。

## ステップ 1: 信頼できるソースを特定

| ソース | 生成対象 |
|--------|----------|
| `package.json` scripts | 利用可能なコマンドリファレンス |
| `.env.example` | 環境変数ドキュメント |
| `openapi.yaml` / ルートファイル | API エンドポイントリファレンス |
| ソースコードエクスポート | パブリック API ドキュメント |
| `Dockerfile` / `docker-compose.yml` | インフラストラクチャセットアップドキュメント |

## ステップ 2: スクリプトリファレンスを生成

1. `package.json`（または `Makefile`、`Cargo.toml`、`pyproject.toml`）を読み込む
2. すべてのスクリプト/コマンドと説明を抽出
3. リファレンステーブルを生成:

```markdown
| コマンド | 説明 |
|---------|------|
| `npm run dev` | ホットリロード付き開発サーバーを起動 |
| `npm run build` | 型チェック付き本番ビルド |
| `npm test` | カバレッジ付きテストスイートを実行 |
```

## ステップ 3: 環境変数ドキュメントを生成

1. `.env.example`（または `.env.template`、`.env.sample`）を読み込む
2. 目的と共にすべての変数を抽出
3. 必須オプションとして分類
4. 期待される形式と有効な値をドキュメント化

```markdown
| 変数 | 必須 | 説明 | 例 |
|------|------|------|-----|
| `DATABASE_URL` | はい | PostgreSQL 接続文字列 | `postgres://user:pass@host:5432/db` |
| `LOG_LEVEL` | いいえ | ログレベル（デフォルト: info） | `debug`、`info`、`warn`、`error` |
```

## ステップ 4: Contribute ガイドを更新

`docs/CONTRIBUTING.md` を生成または更新:
- 開発環境セットアップ（前提条件、インストール手順）
- 利用可能なスクリプトと目的
- テスト手順（実行方法、新規テストの書き方）
- コードスタイル強制（リンター、フォーマッター、プリコミットフック）
- PR 提出チェックリスト

## ステップ 5: Runbook を更新

`docs/RUNBOOK.md` を生成または更新:
- デプロイ手順（ステップバイステップ）
- ヘルスチェックエンドポイントと監視
- 一般的な問題と修正方法
- ロールバック手順
- アラートとエスカレーションパス

## ステップ 6: 古さチェック

1. 90 日以上修正されていないドキュメントファイルを検出
2. 最近のソースコード変更と相互参照
3. 潜在的に古い可能性のあるドキュメントに手動確認用フラグを付ける

## ステップ 7: サマリーを表示

```
ドキュメント更新
──────────────────────────────
更新: docs/CONTRIBUTING.md（スクリプトテーブル）
更新: docs/ENV.md（3 つの新しい変数）
フラグ: docs/DEPLOY.md（142 日古い）
スキップ: docs/API.md（変更なし検出）
──────────────────────────────
```

## ルール

- **単一の信頼できるソース**: 常にコードから生成し、生成セクションは手動編集しない
- **手動セクションを保持**: 生成セクションのみ更新し、手書きの散文はそのままにする
- **生成コンテンツにマーク**: `<!-- AUTO-GENERATED -->` マーカーで生成セクションを囲む
- **未要請のドキュメントを作成しない**: コマンドが明示的に要求した場合のみ新しいドキュメントファイルを作成
