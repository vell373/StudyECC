# Aside コマンド

タスク実行中に質問し、即座で焦点を絞った答えを得る。その後、中断したところから続行します。現在のタスク、ファイル、コンテキストは決して変更されません。

## いつ使うか

- Claude が作業中に何かを質問したいが、勢いを失いたくない
- 現在編集しているコードについて簡潔な説明が必要
- 決定について2番目の意見や説明が必要だがタスクを脇道に逸らしたくない
- エラー、概念、パターンを理解してから Claude が進む前に理解したい
- 現在のタスクとは無関係なことを尋ねたいが、新しいセッションを開始したくない

## 使用方法

```
/aside <your question>
/aside this function は実際に何を返していますか？
/aside このパターンはスレッドセーフですか？
/aside ここで X ではなく Y を使っているのはなぜですか？
/aside foo() と bar() の違いは何ですか？
/aside 追加したばかりの N+1 クエリを心配する必要がありますか？
```

## プロセス

### ステップ1：現在のタスク状態をフリーズする

回答する前に、以下の点に注意してください：
- 現在アクティブなタスクは何ですか？（どのファイル、機能、または問題に取り組んでいたのか）
- `/aside` が呼び出された時点で、どのステップが進行中でしたか？
- 次に何が起こるはずでしたか？

aside 中にファイルを変更、編集、作成、または削除しないでください。

### ステップ2：質問に直接答える

回答は、完全で有用でありながら、最も簡潔な形式で答えてください。

- 推論ではなく、答えで始める
- 簡潔に保つ — 完全な説明が必要な場合は、タスク後に詳しく説明することを提案
- 質問が現在のファイルまたは編集中のコードに関するものである場合は、正確に参照してください（関連がある場合はファイルパスと行番号）
- 回答にファイルを読むことが必要な場合は、読むこと — ただし読み取り専用で、決して書き込みしない

レスポンスを以下のようにフォーマットします：

```
ASIDE: [質問を簡潔に言い換える]

[ここにあなたの答え]

— タスクに戻る: [何をしていたかの1行説明]
```

### ステップ3：メインタスクを再開する

答えを提供した後、すぐに、aside を開始した正確なポイントからアクティブなタスクを続けます。aside の答えがブロッカーを明らかにするか、現在のアプローチを再検討する理由がない限り、再開の許可を求めないでください（「エッジケース」を参照）。

## エッジケース

**質問が提供されていない（`/aside` の後に何もない）：**
返信してください：
```
ASIDE: 質問が提供されていません

何を知りたいですか？（質問をして、現在のタスク コンテキストを失わずに答えます）

— タスクに戻る: [何をしていたかの1行説明]
```

**質問が現在のタスクで潜在的な問題を明らかにする：**
再開する前に明確にフラグを立てます：
```
ASIDE: [答え]

警告：この答えは、現在のアプローチに関する [問題] を示唆しています。続行する前にこれに対処したいですか、それとも計画どおりに進めますか？
```
再開する前にユーザーの決定を待ちます。

**質問が実は タスク リダイレクト（単なる副次的な質問ではない）：**
質問が構築対象を変更することを示唆している場合（例：`/aside 実際に、代わりに Redis を使用しましょう`）、明確にしてください：
```
ASIDE: それは情報だけの副次的な質問ではなく、方向転換のようです。
あなたはこれを以下のいずれかにしたいですか？
  (a) これを情報としてのみ回答し、現在の計画を維持する
  (b) 現在のタスクを一時停止し、アプローチを変更する
```
ユーザーの答えを待つ — 推測しないでください。

**質問が現在開いているファイルまたはコードに関するもの：**
ライブ コンテキストから答えてください。セッションの前にファイルを読めば、直接参照してください。読んでいない場合は、今すぐ読んで（読み取り専用）、file:line 参照で答えてください。

**アクティブなタスクがない（`/aside` が呼び出されたときに何も進行中でない）：**
レスポンス形状が一貫したままになるように、常に標準ラッパーを使用してください：
```
ASIDE: [質問を簡潔に言い換える]

[ここにあなたの答え]

— タスクに戻る: 再開するアクティブなタスクがありません
```

**質問に長い答えが必要な場合：**
本質的な答えを簡潔に与えてから、以下を提案してください：
```
これは短いバージョンです。[現在のタスク] を完了した後、より詳しい説明が必要ですか？
```

**複数の `/aside` 質問が連続している：**
各質問に順番に答えてください。最後の答えの後、メインタスクを再開してください。aside チェーン全体でタスク状態を失わないでください。

**Aside の答えがコード変更が必要であることを示唆している：**
必要な変更に注意してくださいが、aside 中に変更しないでください：
```
ASIDE: [答え]

 修正する価値がある: [何を変更するべきか]。現在のタスク完了後にこれにフラグを立てます。変更を今すぐアドレスしたい場合を除き。
```

**質問があいまいすぎる：**
答えに必要な情報を得るための最短の質問 — 1つの明確な質問をしてください。複数の質問をしないでください。

## 出力例

```
User: /aside fetchWithRetry() は実際に何をしていますか？

ASIDE: fetchWithRetry() は何をしていますか？

fetchWithRetry()（src/api/retry.ts:12）は、指数バックオフ（250ms → 500ms → 1s）で最大3回リクエストを試します。5xx レスポンスとネットワークエラーのみで再試行されます — 4xx エラーは最終と見なされ、再試行されません。

— タスクに戻る: src/middleware/auth.ts の認証ミドルウェアをリファクタリング
```

```
User: /aside 現在のアプローチはスレッドセーフですか？

ASIDE: 現在のアプローチはスレッドセーフですか？

いいえ — src/cache/store.ts:34 の共有キャッシュ オブジェクトはロックなしで変更されます。
同時リクエストでは、これはレース条件です。シングルプロセスの Node.js サーバーではリスクが低いですが、ワーカースレッドまたはクラスタリングでは本当の問題になります。

警告：この機能に影響する可能性があります。今すぐアドレスしますか、それともそれを後のフォローアップで修正しますか？
```

## 注記

- aside 中にファイルを決して変更しないでください — 読み取り専用アクセスのみ
- aside は会話の一時停止であり、新しいタスクではありません — オリジナル タスクは常に再開する必要があります
- 回答に焦点を当てます：目的は、ユーザーをすぐにブロック解除することであり、講義を提供することではありません
- aside が大きな議論を引き出す場合、現在のタスクを完了してから。aside がブロッカーを明らかにしない限り
- Aside はセッション ファイルに保存されません。タスク結果に明示的に関連がある場合を除き
