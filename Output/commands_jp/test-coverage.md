# テストカバレッジ

テストカバレッジを分析し、ギャップを特定し、80% 以上のカバレッジに到達するための欠落テストを生成します。

## ステップ 1: テストフレームワークを検出

| 指標 | カバレッジコマンド |
|-----|-------------------|
| `jest.config.*` または `package.json` jest | `npx jest --coverage --coverageReporters=json-summary` |
| `vitest.config.*` | `npx vitest run --coverage` |
| `pytest.ini` / `pyproject.toml` pytest | `pytest --cov=src --cov-report=json` |
| `Cargo.toml` | `cargo llvm-cov --json` |
| `pom.xml` JaCoCo 付き | `mvn test jacoco:report` |
| `go.mod` | `go test -coverprofile=coverage.out ./...` |

## ステップ 2: カバレッジレポートを分析

1. カバレッジコマンドを実行
2. 出力を解析（JSON サマリーまたはターミナル出力）
3. **80% 以下のファイル** を一覧表示（最悪優先）
4. カバレッジ不足の各ファイルについて、以下を特定:
   - テストされていない関数またはメソッド
   - 欠落しているブランチカバレッジ（if/else、switch、エラーパス）
   - デッドコードが分母をインフレーションさせているかどうか

## ステップ 3: 欠落テストを生成

カバレッジ不足の各ファイルについて、以下の優先順位に従ってテストを生成:

1. **ハッピーパス** — 正常な入力を用いたコア機能
2. **エラーハンドリング** — 無効な入力、欠落データ、ネットワーク障害
3. **エッジケース** — 空配列、null/undefined、境界値（0、-1、MAX_INT）
4. **ブランチカバレッジ** — 各 if/else、switch ケース、三項演算子

### テスト生成ルール

- テストをソースの隣に配置: `foo.ts` → `foo.test.ts`（またはプロジェクト慣例）
- プロジェクトの既存のテストパターンを使用（インポートスタイル、アサーションライブラリ、モッキングアプローチ）
- 外部依存関係（データベース、API、ファイルシステム）をモック
- 各テストは独立 — テスト間で共有可変状態なし
- テストを説明的に命名: `test_create_user_with_duplicate_email_returns_409`

## ステップ 4: 検証

1. フルテストスイートを実行 — すべてのテストがパスする必要がある
2. カバレッジを再実行 — 改善を検証
3. まだ 80% 以下の場合、残りのギャップについてステップ 3 を繰り返す

## ステップ 5: レポート

修正前後の比較を表示:

```
カバレッジレポート
──────────────────────────────
ファイル                   修正前  修正後
src/services/auth.ts       45%    88%
src/utils/validation.ts    32%    82%
──────────────────────────────
全体:                       67%    84%  成功
```

## 重点領域

- 複雑な分岐を持つ関数（高いサイクロマティック複雑度）
- エラーハンドラーと catch ブロック
- コードベース全体で使用されるユーティリティ関数
- API エンドポイントハンドラー（リクエスト → レスポンスフロー）
- エッジケース: null、undefined、空文字列、空配列、ゼロ、負の数
