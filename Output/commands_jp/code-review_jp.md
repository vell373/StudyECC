# コードレビュー

> PRレビューモードはWirasmによるPRPs-agentic-engから適応。PRPワークフロースリーズの一部。

**入力**: $ARGUMENTS

---

## モード選択

`$ARGUMENTS`にPR番号、PR URL、または`--pr`が含まれる場合:
→ **PRレビューモード**へジャンプ。

その他の場合:
→ **ローカルレビューモード**を使用。

---

## ローカルレビューモード

コミットされていない変更の包括的なセキュリティと品質レビュー。

### フェーズ1 — 収集

```bash
git diff --name-only HEAD
```

変更されたファイルがない場合は停止: 「レビューするものはありません。」

### フェーズ2 — レビュー

各変更されたファイルを完全に読む。以下を確認:

**セキュリティ問題（重大）:**
- ハードコーディングされた認証情報、APIキー、トークン
- SQLインジェクション脆弱性
- XSS脆弱性
- 欠落したインプット検証
- 不安全な依存関係
- パストラバーサルリスク

**コード品質（高）:**
- 50行以上の関数
- 800行以上のファイル
- ネスティング深度 4レベル以上
- 欠落したエラーハンドリング
- console.logステートメント
- TODO/FIXMEコメント
- パブリックAPI用の欠落したJSDoc

**ベストプラクティス（中）:**
- ミューテーションパターン（代わりに不変を使用）
- コード/コメント内のemoji使用
- 新しいコード用の欠落したテスト
- アクセシビリティ問題（a11y）

### フェーズ3 — レポート

以下を含むレポートを生成:
- 深刻度: 重大、高、中、低
- ファイルの場所と行番号
- 問題の説明
- 提案された修正

重大または高の問題が見つかった場合、コミットをブロック。
セキュリティ脆弱性でコードを承認しない。

## PRレビューモード

包括的なGitHub PRレビュー — diffをフェッチし、ファイル全体を読み、バリデーションを実行、レビューを投稿。

### フェーズ1 — フェッチ

入力を解析してPRを決定:

| 入力 | アクション |
|---|---|
| 番号（例：`42`） | PR番号として使用 |
| URL（`github.com/.../pull/42`） | PR番号を抽出 |
| ブランチ名 | `gh pr list --head <branch>`経由でPRを検索 |

```bash
gh pr view <NUMBER> --json number,title,body,author,baseRefName,headRefName,changedFiles,additions,deletions
gh pr diff <NUMBER>
```

PRが見つからない場合は、エラーで停止。後のフェーズ用のPRメタデータを保存。

### フェーズ2 — コンテキスト

レビューコンテキストを構築:

1. **プロジェクトルール** — `CLAUDE.md`、`.claude/docs/`、および任意の貢献ガイドラインを読む
2. **PRPアーティファクト** — このPRに関連する実装コンテキストの`.claude/PRPs/reports/`と`.claude/PRPs/plans/`をチェック
3. **PR意図** — ゴール、リンクされた問題、テスト計画のPR説明を解析
4. **変更されたファイル** — 修正されたすべてのファイルをリストして、型（ソース、テスト、設定、ドキュメント）別にカテゴリ化

### フェーズ3 — レビュー

各変更されたファイルを**完全に**読む（diffハンクだけではなく — 周囲のコンテキストが必要）。

PRレビューの場合、PR頭リビジョンでフルファイルコンテンツをフェッチ:
```bash
gh pr diff <NUMBER> --name-only | while IFS= read -r file; do
  gh api "repos/{owner}/{repo}/contents/$file?ref=<head-branch>" --jq '.content' | base64 -d
done
```

7つのカテゴリにわたってレビューチェックリストを適用:

| カテゴリ | 何を確認するか |
|---|---|
| **正確性** | ロジックエラー、オフバイワン、nullハンドリング、エッジケース、レース条件 |
| **型安全性** | 型の不一致、不安全なキャスト、`any`使用、欠落したジェネリック |
| **パターンコンプライアンス** | プロジェクト規約に一致（命名、ファイル構造、エラーハンドリング、インポート） |
| **セキュリティ** | インジェクション、認証ギャップ、秘密露出、SSRF、パストラバーサル、XSS |
| **パフォーマンス** | N+1クエリ、欠落したインデックス、無界ループ、メモリリーク、大きなペイロード |
| **完成度** | 欠落したテスト、欠落したエラーハンドリング、不完全なマイグレーション、欠落したドキュメント |
| **保守性** | デッドコード、マジックナンバー、深いネスティング、不明確な命名、欠落した型 |

各発見に深刻度を割り当てる:

| 深刻度 | 意味 | アクション |
|---|---|---|
| **重大** | セキュリティ脆弱性またはデータ損失リスク | マージ前に修正必須 |
| **高** | マージ前に修正を推奨するバグまたはロジックエラー | マージ前に修正が推奨 |
| **中** | コード品質の問題またはベストプラクティスの欠落 | 修正をお勧め |
| **低** | スタイルのニットまたは軽微な提案 | オプション |

### フェーズ4 — バリデーション

利用可能なバリデーションコマンドを実行:

設定ファイル（`package.json`、`Cargo.toml`、`go.mod`、`pyproject.toml`等）からプロジェクトタイプを検出してから、適切なコマンドを実行:

**Node.js / TypeScript** （`package.json`がある）:
```bash
npm run typecheck 2>/dev/null || npx tsc --noEmit 2>/dev/null  # 型チェック
npm run lint                                                    # リント
npm test                                                        # テスト
npm run build                                                   # ビルド
```

**Rust** （`Cargo.toml`がある）:
```bash
cargo clippy -- -D warnings  # リント
cargo test                   # テスト
cargo build                  # ビルド
```

**Go** （`go.mod`がある）:
```bash
go vet ./...    # リント
go test ./...   # テスト
go build ./...  # ビルド
```

**Python** （`pyproject.toml`/`setup.py`がある）:
```bash
pytest  # テスト
```

検出されたプロジェクトタイプに適用されるコマンドのみを実行。各コマンドのパス/失敗を記録。

### フェーズ5 — 決定

発見に基づいて推奨を形成:

| 条件 | 決定 |
|---|---|
| ゼロの重大/高の問題、バリデーション合格 | **承認** |
| 中/低の問題のみ、バリデーション合格 | コメント付き**承認** |
| 高の問題またはバリデーション失敗 | **変更をリクエスト** |
| 重大な問題 | **ブロック** — マージ前に修正必須 |

特殊ケース:
- ドラフトPR → 常に**コメント**を使用（承認/ブロックではない）
- ドキュメント/設定変更のみ → より軽いレビュー、正確性にフォーカス
- 明示的な`--approve`または`--request-changes`フラグ → 決定をオーバーライド（しかし依然としてすべての発見を報告）

### フェーズ6 — レポート

`.claude/PRPs/reviews/pr-<NUMBER>-review.md`でレビューアーティファクトを作成:

```markdown
# PRレビュー: #<NUMBER> — <TITLE>

**レビュー日時**: <日付>
**著者**: <著者>
**ブランチ**: <ヘッド> → <ベース>
**決定**: 承認 | 変更をリクエスト | ブロック

## サマリー
<全体的な評価の1-2文>

## 発見

### 重大
<発見または「なし」>

### 高
<発見または「なし」>

### 中
<発見または「なし」>

### 低
<発見または「なし」>

## バリデーション結果

| チェック | 結果 |
|---|---|
| 型チェック | 合格 / 不合格 / スキップ |
| リント | 合格 / 不合格 / スキップ |
| テスト | 合格 / 不合格 / スキップ |
| ビルド | 合格 / 不合格 / スキップ |

## レビューされたファイル
<変更タイプ付きのファイルリスト: 追加/修正/削除>
```

### フェーズ7 — 公開

GitHubにレビューを投稿:

```bash
# 承認の場合
gh pr review <NUMBER> --approve --body "<レビューのサマリー>"

# 変更をリクエストする場合
gh pr review <NUMBER> --request-changes --body "<必要な修正のサマリー>"

# コメントのみ（ドラフトPRまたは情報向け）
gh pr review <NUMBER> --comment --body "<サマリー>"
```

特定の行へのインラインコメントについては、GitHubレビューコメント API を使用:
```bash
gh api "repos/{owner}/{repo}/pulls/<NUMBER>/comments" \
  -f body="<コメント>" \
  -f path="<ファイル>" \
  -F line=<行番号> \
  -f side="RIGHT" \
  -f commit_id="$(gh pr view <NUMBER> --json headRefOid --jq .headRefOid)"
```

あるいは、複数のインラインコメント付きの単一レビューを一度にポスト:
```bash
gh api "repos/{owner}/{repo}/pulls/<NUMBER>/reviews" \
  -f event="COMMENT" \
  -f body="<全体的なサマリー>" \
  --input comments.json  # [{"path": "ファイル", "line": N, "body": "コメント"}, ...]
```

### フェーズ8 — 出力

ユーザーに報告:

```
PR #<NUMBER>: <TITLE>
決定: <承認|変更をリクエスト|ブロック>

問題: <重大カウント>重大、<高カウント>高、<中カウント>中、<低カウント>低
バリデーション: <合格カウント>/<総カウント>チェック合格

アーティファクト:
  レビュー: .claude/PRPs/reviews/pr-<NUMBER>-review.md
  GitHub: <PR URL>

次のステップ:
  - <決定に基づく文脈的な提案>
```

---

## エッジケース

- **`gh` CLIなし**: ローカルのみレビューにフォールバック（diffを読む、GitHubパブリッシュをスキップ）。ユーザーに警告。
- **分岐が分散**: `git fetch origin && git rebase origin/<base>`をレビュー前に提案。
- **大きなPR（>50ファイル）**: レビュースコープについて警告。ソース変更をまず焦点を合わせてから、テストを、設定/ドキュメントを。
