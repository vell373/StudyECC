# Asideコマンド

タスクの途中で質問し、すぐにフォーカスされた回答を得て、その後正確に続きから作業を再開する。現在のタスク、ファイル、コンテキストは決して修正されない。

## 使用時期

- Claudeが作業をしている途中に何かが気になり、勢いを失いたくない場合
- 現在編集中のコードに関する簡潔な説明が必要な場合
- 第二意見やタスクを脱線させずに決定について確認したい場合
- 進める前にエラー、概念、またはパターンを理解する必要がある場合
- 新しいセッションを開始せずに現在のタスクに関連しない何かを聞きたい場合

## 使用方法

```
/aside <あなたの質問>
/aside この関数は実際に何を返すのか？
/aside このパターンはスレッドセーフか？
/aside なぜここでXではなくYを使用しているのか？
/aside foo()とbar()の違いは何か？
/aside 追加したばかりのN+1クエリについて心配する必要があるか？
```

## プロセス

### ステップ1：現在のタスク状態を固定化

何かに答える前に、心の中で注記する:
- アクティブなタスクは何か？（何ファイル、機能、またはどの問題が進行中か）
- `/aside`が呼び出された時点でどのステップが進行中だったか？
- 次は何が起きようとしていたか？

aside中に、ファイルを触った、編集、作成、削除をしない。

### ステップ2：質問に直接答える

質問に最も簡潔な形で答える。ただし完全で有用な形で。

- 理由ではなく答えリード
- 短く保つ — 完全な説明が必要な場合は、タスク後にさらに深掘りすることを提案
- 質問が現在のファイルまたは進行中のコードに関する場合は、正確に参照（関連がある場合はファイルパスと行番号）
- 回答がファイルの読み取りを必要とする場合は、読み取る — ただし読み取るだけ、決して書き込みはしない

レスポンスを次のようにフォーマットする:

```
ASIDE: [質問を簡潔に改述]

[あなたの回答はここ]

— タスクに戻る: [進行中の内容の1行説明]
```

### ステップ3：メインタスクを再開

回答を提供した後、一時停止したポイントから正確にアクティブなタスクをすぐに続ける。aside回答がブロッカーを明らかにしたか、現在のアプローチを再検討する理由があった場合を除き、許可を求めない（エッジケースを参照）。

## エッジケース

**質問なし（`/aside`の後に何もない）:**
レスポンス:
```
ASIDE: 質問が提供されていない

何を知りたいですか?（質問して、現在のタスクコンテキストを失わずに答えます）

— タスクに戻る: [進行中の内容の1行説明]
```

**質問が現在のタスクの潜在的な問題を明らかにする:**
再開する前に明確にフラグを立てる:
```
ASIDE: [答え]

警告: このこの答えは現在のアプローチの[問題]を示唆している。続行する前にこれに対処したいですか、または計画通り続行しますか?
```
再開する前にユーザーの決定を待つ。

**質問は実はタスクのリダイレクト（サイドの質問ではない）:**
質問が構築しているものを変更することを暗に示す場合（例：`/aside 実際には、代わりにRedisを使用しましょう`）、明確にする:
```
ASIDE: これはサイドの質問ではなく、方向の変更のように聞こえます。
あなたは以下のいずれかをしたいですか:
  (a) これを情報としてのみ答え、現在の計画を保つ
  (b) 現在のタスクを一時停止して方法を変更
```
ユーザーの回答を待つ — 推測しない。

**質問が現在開いているファイルまたはコードについて:**
ライブコンテキストから答える。ファイルがセッションで早期に読み取られた場合は、直接参照する。そうでない場合は、今すぐ読み取る（読み取り専用）して、ファイル：行参照で答える。

**アクティブなタスクなし（`/aside`が呼び出された時点で進行中のタスクがない）:**
それでも標準的なラッパーを使用してレスポンス形状を一貫性を保つ:
```
ASIDE: [質問を簡潔に改述]

[あなたの回答はここ]

— タスクに戻る: 再開するアクティブなタスクはない
```

**質問が長い回答を必要とする:**
本質的な答えを簡潔に与えてから、次のように提案:
```
短いバージョンはこちら。[現在のタスク]を完了したら、さらに深い説明がほしいですか?
```

**複数の`/aside`質問が連続:**
各質問を順番に答える。最後の回答の後、メインタスクを再開。一連のaside全体でタスク状態を失わない。

**Aside回答はコード変更が必要であることを暗に示す:**
変更が必要なことを注記するが、asideの間は変更しない:
```
ASIDE: [答え]

修正する価値: [何を変更すべきか]。現在のタスク後にこれをフラグ立てます。今すぐ対処したい場合を除き。
```

**質問が曖昧または曖昧すぎる:**
1つの明確化質問を聞く — 回答が必要な情報を得るための最短質問。複数の質問を聞かない。

## 出力例

```
ユーザー: /aside fetchWithRetry()は実際に何をするのか?

ASIDE: fetchWithRetry()は何をするのか?

fetchWithRetry()（src/api/retry.ts:12）はリクエストを最大3回試行する
指数バックオフ（250ms → 500ms → 1s）。 5xxレスポンスと
ネットワークエラーのみ再試行する — 4xxエラーは最終として扱い、再試行されない。

— タスクに戻る: src/middleware/auth.tsの認証ミドルウェアのリファクタリング
```

```
ユーザー: /aside 採用しているアプローチはスレッドセーフか?

ASIDE: 現在のアプローチはスレッドセーフか?

いいえ — src/cache/store.ts:34の共有キャッシュオブジェクトはロックなしで変更される。
並行リクエストの下ではこれはレース条件。単一プロセスでは低リスク
Node.jsサーバーだが、ワーカースレッドまたはクラスタリングでは本物の問題になるだろう。

警告: これは構築している機能に影響を与える可能性があります。今すぐこれに対処したいか、それとも続行してフォローアップで修正しますか?
```

---

## 注記

- aside中にファイルを変更しない — 読み取り専用アクセスのみ
- asideは会話の一時停止、新しいタスクではない — 元のタスクは常に再開する必要がある
- 回答をフォーカスを保つ: 目標はユーザーを迅速にブロック解除することで、講義を提供することではない
- asideが大きな議論につながる場合は、asideがブロッカーを明らかにするまで現在のタスクを完了するまで待つ
- Asideはタスク結果に明示的に関連がない限り、セッションファイルに保存されない
