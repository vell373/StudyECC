# コードマップを更新

コードベース構造を分析し、トークン削減型のアーキテクチャドキュメントを生成します。

## ステップ 1: プロジェクト構造をスキャン

1. プロジェクトタイプを特定（モノレポ、シングルアプリ、ライブラリ、マイクロサービス）
2. すべてのソースディレクトリを検出（src/、lib/、app/、packages/）
3. エントリーポイントをマッピング（main.ts、index.ts、app.py、main.go など）

## ステップ 2: コードマップを生成

`docs/CODEMAPS/`（または `.reports/codemaps/`）にコードマップを作成または更新:

| ファイル | 内容 |
|---------|------|
| `architecture.md` | 高レベルのシステム図、サービス境界、データフロー |
| `backend.md` | API ルート、ミドルウェアチェーン、サービス → リポジトリマッピング |
| `frontend.md` | ページツリー、コンポーネント階層、状態管理フロー |
| `data.md` | データベーステーブル、リレーションシップ、マイグレーション履歴 |
| `dependencies.md` | 外部サービス、サードパーティ統合、共有ライブラリ |

### コードマップ形式

各コードマップはトークン削減型 — AI コンテキスト消費を最適化:

```markdown
# バックエンドアーキテクチャ

## ルート
POST /api/users → UserController.create → UserService.create → UserRepo.insert
GET  /api/users/:id → UserController.get → UserService.findById → UserRepo.findById

## キーファイル
src/services/user.ts（ビジネスロジック、120 行）
src/repos/user.ts（データベースアクセス、80 行）

## 依存関係
- PostgreSQL（プライマリデータストア）
- Redis（セッションキャッシュ、レート制限）
- Stripe（支払い処理）
```

## ステップ 3: Diff 検出

1. 以前のコードマップが存在する場合、diff 率を計算
2. 変更が 30% を超える場合、diff を表示し、上書き前にユーザーの承認を要求
3. 変更が 30% 以下の場合、その場で更新

## ステップ 4: メタデータを追加

各コードマップに鮮度ヘッダーを追加:

```markdown
<!-- 生成: 2026-02-11 | スキャンファイル: 142 | トークン推定: 約 800 -->
```

## ステップ 5: 分析レポートを保存

`.reports/codemap-diff.txt` にサマリーを書き込む:
- 最後のスキャン以降に追加・削除・修正されたファイル
- 新たに検出された依存関係
- アーキテクチャの変更（新しいルート、新しいサービスなど）
- 90 日以上更新されていないドキュメントの古さ警告

## ヒント

- **高レベルの構造** に焦点を当てる、実装の詳細ではなく
- **ファイルパスと関数シグネチャ** を優先、フルコードブロックではなく
- 各コードマップを **1000 トークン以下** に保つ（効率的なコンテキスト読み込みのため）
- 冗長な説明ではなく、データフロー用に ASCII 図を使用
- 大規模な機能追加またはリファクタリングセッション後に実行
