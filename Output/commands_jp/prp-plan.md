# PRPプラン作成コマンド

このコマンドはコードベースの詳細解析、プロジェクト構造の理解、既存パターンの抽出を含む包括的なプランを作成します。

デフォルトではインラインで実行します。

## このコマンドの機能

1. **コードベースを自動解析** — ファイル構造、既存の設計パターン、規約を検出
2. **依存関係グラフを構築** — コンポーネント間の関係を理解
3. **既存パターンを抽出** — 類似実装を参照して一貫性を確保
4. **段階的プランを生成** — PRD統合、リスク評価を含む実装ロードマップ

## 使用時

以下の場合に `/prp-plan` を使用してください:
- `/plan` より深い計画が必要
- 大規模プロジェクトで一貫性が重要
- 既存コードパターンに準拠する必要がある
- チーム内で詳細ドキュメントが必要

## 動作方法

### フェーズ1: 検出（DETECT）
- プロジェクトトポロジーを特定
- フレームワーク、言語、ツールを検出
- 既存の設計パターン（MVC, Flux, DDD等）を識別

### フェーズ2: 解析（PARSE）
- ファイル構造を走査
- ディレクトリ構成から責務分離パターンを抽出
- 重要なファイル（app.js, main.py, main.go等）を特定

### フェーズ3: 探索（EXPLORE）
- ドメインモデルを分析
- ドメイン層、アプリケーション層、プレゼンテーション層の分離を検査
- 既存のエンティティ、インターフェース、型定義を列挙

| コンポーネント | 検出方法 | 例 |
|-------------|--------|-----|
| エンティティ | ドメイン層ファイル | User, Order, Product |
| サービス | application/services | UserService, OrderService |
| リポジトリ | data/db/persistence | UserRepository |
| コントローラー | routes/handlers/controllers | UserController |
| ミドルウェア | middleware/interceptors | AuthMiddleware |

### フェーズ4: リサーチ（RESEARCH）
- 関連する既存実装を読む（同じドメインの似たコンポーネント）
- テスト構造とテストカバレッジを把握
- 既存の設定ファイル、DI コンテナ、ロギングシステムを確認

### フェーズ5: 設計（DESIGN）
ユーザーの要件に基づいて、以下を設計:
- **データモデル**: 新しいエンティティ、既存エンティティへの変更
- **ビジネスロジック**: 新しいサービス、既存サービスの拡張
- **I/O**: 新しいリポジトリ、API エンドポイント
- **UI**: 新しいコンポーネント、ページ

既存パターンを参照して一貫性を確保する。

### フェーズ6: 生成（GENERATE）
包括的なプランを生成:

```markdown
# 実装プラン: [フィーチャー名]

## 要件の再度説明
[ユーザー入力から解析した要件]

## コードベース分析

### 現在の構造
- フレームワーク: [例: Next.js App Router]
- パターン: [例: Server Components + Actions]
- 関連コンポーネント: [既存コンポーネントのリスト]

### 既存パターン（参照）
- [類似コンポーネントの実装パターン]
- [既存の規約、命名則]

## 実装フェーズ

### フェーズ1: [スコープ]
- [ファイル/コンポーネント]: [説明]
- [テスト]: [テストケース]

### フェーズ2: [スコープ]
...

## 依存関係グラフ
```
フェーズ1
  ├─ フェーズ2 (フェーズ1に依存)
  └─ フェーズ3 (フェーズ1に依存)
フェーズ2
  └─ フェーズ4
```

## リスク評価
- [リスク]: [対策]

## 推定工数
- [フェーズ]: [見積もり]

**確認待ち**: このプランで進めますか?
```

## 推奨読書リスト

プラン生成後、自動的に参照すべきコードセクションを提示:

```
## 参照実装

以下のファイルを読んでから実装を始めてください:

1. `src/domain/User.ts` — ユーザーエンティティのパターン
2. `src/application/UserService.ts` — サービス層の実装パターン
3. `src/presentation/user/UserForm.tsx` — フォームコンポーネントのパターン
4. `tests/application/UserService.test.ts` — テスト構造
```

## 使用例

```
ユーザー: /prp-plan 注文が複数の支払い方法で分割決済できる機能

アシスタント:

# 実装プラン: 分割決済機能

## 要件の再度説明
- 1つの注文を複数の支払い方法で分割決済
- 各支払い方法の決済ステータスを独立で追跡
- 部分決済の場合は注文ステータスを「部分支払い」に

## コードベース分析

### 現在の構造
- Next.js App Router
- Prisma ORM
- Server Actions

### 既存パターン
- 決済: Order → Payment → Transaction (1:N)
- ステータス追跡: status カラム + updated_at

## 実装フェーズ

### フェーズ1: スキーマ拡張
- prisma/schema.prisma: Payment に paymentMethods テーブル追加
- migration: add_payment_methods

### フェーズ2: 支払いサービス
- PaymentSplitService: 分割ロジック実装

### フェーズ3: 注文ステータス管理
- OrderService: 部分支払いステータス対応

### フェーズ4: UI更新
- OrderDetail: 支払い方法セレクター追加

## 参照実装
1. `src/domain/Order.ts` — 注文エンティティ
2. `src/domain/Payment.ts` — 決済パターン
3. `src/application/PaymentService.ts` — 決済ロジック
4. `app/orders/[id]/page.tsx` — 注文詳細ページ

**確認待ち**: このプランで進めますか?
```

## 重要な注意事項

**参照実装を必ず読む**: 生成されたプランには参照すべきコードセクションが含まれます。実装前に必ずそれらを読んでください。

**パターン一貫性**: 新しいコードは既存パターンに準拠する必要があります。一貫性を破く理由がある場合はユーザー確認を取ってください。

## 他のコマンドとの統合

プラン作成後:
- `/prp-implement` → プランを実行
- `/plan` → より単純な計画が必要な場合

参照コードの確認中:
- `/code-review` → 参照コードをレビュー
