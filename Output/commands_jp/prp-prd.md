# PRP PRD生成コマンド

このコマンドは対話的にPRD（Product Requirements Document）を生成します。問題から始まり、段階的に要件を掘り下げて、最終的なドキュメントを生成します。

## このコマンドの機能

1. **問題優先型** — 「何を作るか」ではなく「何の問題を解くか」から始める
2. **対話的な質問フロー** — ユーザーの回答に基づいて深堀質問
3. **段階的なグラウンディング** — 仮説を検証しながら要件を確定
4. **包括的なPRD生成** — 完成度の高いドキュメントを自動生成

## 使用時

```bash
/prp-prd                         # 対話的にPRDを生成
/prp-prd --quick                 # 簡潔版（6フェーズ）
/prp-prd --template standard     # 標準テンプレートを使用
```

## 動作方法

### フェーズ1: 開始（INITIATE）

ユーザーに問題・課題を自由形式で入力させる:

```
# PRD生成ウィザード

**問題・課題について教えてください。**

例:
- 「ユーザーが購入後に在庫切れ通知をスパム受信している」
- 「チーム内の知識が個人に依存していて属人化が激しい」
- 「新規ユーザーのオンボーディングが複雑で離脱率が高い」

[ユーザー入力を待つ]
```

### フェーズ2: 基礎（FOUNDATION）

入力された問題から、以下を抽出・確認:

| 要素 | 質問 | 目的 |
|------|------|------|
| **ステークホルダー** | 「誰がこの問題に直面していますか?」 | ターゲットユーザーを特定 |
| **影響度** | 「このために失っているもの（時間・コスト・ユーザー）は?」 | 優先度を評価 |
| **現状対応** | 「現在はどう対応していますか?」 | 既存ワークアラウンドを理解 |
| **理想状態** | 「理想的な解決策は?」 | ビジョンを確認 |

例:

```
ユーザー: 「ユーザーが購入後に在庫切れ通知をスパム受信している」

アシスタント: 
## 基礎確認

1. **誰が困っていますか?**
   - 購入ユーザー全体ですか、それとも特定セグメント?
   - 通知を受け取る側？送る側？

2. **どの程度の影響ですか?**
   - 1日あたり何件の苦情ですか?
   - 購入完了後どのタイミングで通知が来ますか?

3. **現在はどう対応していますか?**
   - メッセージを手動で削除?
   - 通知設定で無効化?

4. **理想的な解決策は?**
   - 在庫復活時のみ通知を送る?
   - ユーザーが頻度を制御できる?
```

### フェーズ3: グラウンディング（GROUNDING）第1回

仮説を立てて、ユーザーに確認:

```
## 仮説のグラウンディング（第1回）

**問題の仮説:**
- 在庫チェック実行のたびに通知が送信される（バッチ処理が複数回実行？）
- 通知設定でフィルタリングがない

**確認質問:**
- これはシステム設計の問題ですか、それとも運用の問題ですか?
- 通知は「購入ユーザー全員」ですか、それとも「在庫復活時に購入ユーザーをフィルタ」ですか?
```

### フェーズ4: 深堀（DEEP DIVE）

確認に基づいて、詳細な要件を掘り下げる:

```
## 詳細要件

**1. スコープ**
- 対象商品: [食品 / 全品 / プレミアム商品のみ]?
- 対象ユーザー: [ウィッシュリスト登録者 / 購入ユーザー / 特定セグメント]?

**2. 通知ルール**
- 在庫復活時のみ通知 vs. 定期ステータス報告?
- 通知頻度: 最大いくつ/日 or /週?

**3. ユーザー制御**
- 通知をON/OFFできますか?
- 通知頻度をカスタマイズできますか?

**4. 技術制約**
- 現在のシステムで実現可能ですか?
- DBやメール送信システムに制限がありますか?
```

### フェーズ5: グラウンディング（GROUNDING）第2回

詳細要件を検証:

```
## 検証と決定

**決定事項:**
- 在庫復活時のみ通知（1回限り）
- ユーザーがメールプリファレンスで通知をOFF可能
- 初期状態: ON（ユーザーがOFFに変更）

**確認:**
これで要件を確定しますか?（はい/変更あり）
```

### フェーズ6: 決定（DECISIONS）

最終決定の確認:

```
## 最終決定

**機能概要:**
在庫復活時に購入ユーザーへ1回のみ通知を送信し、
ユーザーが通知プリファレンスで制御可能

**実装スコープ:**
- DB: inventory_alerts テーブル（user_id, product_id, sent_at）
- API: POST /notifications/preferences（通知制御）
- Email: template「在庫復活通知」

**成功指標:**
- 通知苦情が50%削減
- 購入後通知開封率 > 30%

**制約:**
- メール配信遅延 < 5分
- ユーザー設定反映 < 1分

これで進めますか?（はい/修正）
```

### フェーズ7: 生成（GENERATE）

最終PRDを生成:

```markdown
# PRD: 在庫復活通知の最適化

## エグゼクティブサマリー
購入ユーザーが在庫切れ商品の復活時に重複通知を受け取る問題を解決。
在庫復活時に1回のみ通知を送信し、ユーザーが頻度を制御可能にする。

## 問題ステートメント
- 現状: 在庫チェックのたびに通知が送信される
- 影響: ユーザーが通知をスパムと認識→開封率低下
- 原因: バッチ処理が複数回実行される、フィルタリングなし

## ターゲットユーザー
- 購入後に在庫復活通知を受け取りたいユーザー
- 通知頻度を制御したいユーザー

## ビジネス目標
- 通知苦情を50%削減
- 通知開封率を30%以上に維持

## 機能仕様

### 1. 在庫復活アラート
- 在庫復活時に購入ユーザーへ1回のみメール送信
- メール送信後 `inventory_alerts.sent_at` に記録
- 同じユーザー・商品への重複送信を防止

### 2. 通知プリファレンス管理
- ユーザーが「在庫復活通知」をON/OFF
- 初期状態: ON
- ユーザー設定ページで変更可能

### 3. オペレーター制御
- オペレーターが通知テンプレートを編集可能
- 送信タイミング、対象セグメントを制御

## データモデル

```
inventory_alerts:
  id, user_id, product_id, notification_type, sent_at, created_at

user_notification_preferences:
  id, user_id, preference_type (inventory_revival, ...), enabled, updated_at
```

## API仕様

### GET /notifications/preferences
ユーザーの通知設定を取得

### POST /notifications/preferences
ユーザーの通知設定を更新

```json
{
  "preference_type": "inventory_revival",
  "enabled": true
}
```

## 成功指標（KPI）
- 通知苦情: < 50件/月（現: 100件/月）
- 開封率: > 30%（現: 15%）
- プリファレンス設定率: > 20%（新規ユーザー）

## 制約・制限
- メール送信遅延: < 5分
- ユーザー設定反映: < 1分
- 対応言語: 日本語, 英語

## 外部依存
- Sendgrid（メール配信）
- 在庫管理システムのAPI

## リスク・ミティゲーション
- リスク: メール配信遅延
  対策: キューイング + 再試行ロジック

## 実装スケジュール
- Phase1: DB + API（2日）
- Phase2: メールテンプレート（1日）
- Phase3: UI（2日）
- Phase4: テスト + デプロイ（2日）

## 参考資料
- 既存の在庫管理フロー
- メール通知システム設計
```

### フェーズ8: 出力（OUTPUT）

生成されたPRDをユーザーに提示:

```
## PRDが生成されました

**ドキュメント:**
```markdown
[完全なPRDをMarkdown形式で表示]
```

**ダウンロード:**
- PRD.md をプロジェクトに保存しますか?

**次のステップ:**
- `/prp-plan` → 実装プランを作成
- `/code-review` → 要件をレビュー
- GitHub Issues → 要件に基づいてイシューを作成
```

## 使用例

```
ユーザー: /prp-prd
問題: 新規ユーザーのオンボーディングが複雑で30%が離脱している

[対話的質問フロー]

アシスタント:
1. ステークホルダー確認: 「新規ユーザー」
2. 影響度確認: 「月100人のうち30人が離脱」
3. 仮説: 「設定ステップが多い?」
4. 深堀: 「簡略化 vs. ガイド表示」
5. 決定: 「3ステップに簡略化 + インタラクティブガイド」

## PRDが生成されました
[完全なPRD]
```

## 重要な注意事項

**問題から始める**: 「何を作るか」ではなく「何の問題を解くか」を最初に確認してください。

**グラウンディングは2回**: 第1回は仮説、第2回は詳細要件を検証します。

**ユーザー確認を何度も取る**: 各フェーズで明示的に確認を取り、進める前にユーザー同意を得ます。

## 他のコマンドとの統合

PRD生成後:
- `/prp-plan` → 実装プランを作成
- `/prp-implement` → プランを実行
- GitHub Issues → 要件に基づいてイシュー作成
