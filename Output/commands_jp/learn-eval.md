# /learn-eval - 抽出、評価、保存

品質ゲート、保存場所の決定、スキルファイルを書き込む前のナレッジプレースメント認識を使用して `/learn` を拡張します。

## 抽出する内容

以下を探します：

1. **エラー解決パターン** — 根本原因 + 修正 + 再利用可能性
2. **デバッグテクニック** — 非自明なステップ、ツール組み合わせ
3. **回避策** — ライブラリの癖、API制限、バージョン固有の修正
4. **プロジェクト固有のパターン** — 規約、アーキテクチャ決定、統合パターン

## プロセス

1. セッションから抽出可能なパターンを確認
2. 最も価値がある/再利用可能なインサイトを特定

3. **保存場所を決定:**
   - 質問: 「このパターンは別のプロジェクトで有用ですか?」
   - **グローバル** (`~/.claude/skills/learned/`): 2つ以上のプロジェクトで使用可能なジェネリックパターン（bash互換性、LLM APIの動作、デバッグテクニックなど）
   - **プロジェクト** (現在のプロジェクト内の `.claude/skills/learned/`): プロジェクト固有のナレッジ（特定の構成ファイルの癖、プロジェクト固有のアーキテクチャ決定など）
   - 迷った場合は グローバル を選択（グローバル → プロジェクト の移動はその逆より簡単）

4. このフォーマットを使用してスキルファイルをドラフト：

```markdown
---
name: pattern-name
description: "130文字未満"
user-invocable: false
origin: auto-extracted
---

# [記述的パターン名]

**抽出日:** [日付]
**コンテキスト:** [このパターンが適用される時についての簡単な説明]

## 問題
[この問題を解決するもの - 具体的に]

## 解決策
[パターン/テクニック/回避策 - コード例付き]

## いつ使うか
[トリガー条件]
```

5. **品質ゲート — チェックリスト + 総合的な判定**

   ### 5a. 必須チェックリスト（ファイルを実際に読んで検証）

   評価する前に以下を**すべて**実行：

   - [ ] `~/.claude/skills/` とプロジェクトの `.claude/skills/` ファイルをキーワードでグリップして内容の重複を確認
   - [ ] MEMORY.md（プロジェクトとグローバル両方）で重複を確認
   - [ ] 既存スキルに追加するだけで十分かどうかを検討
   - [ ] これが再利用可能なパターンであり、一回限りの修正でないことを確認

   ### 5b. 総合的な判定

   チェックリスト結果とドラフト品質を統合して、以下の**1つ**を選択：

   | 判定 | 意味 | 次のアクション |
   |------|------|-------------|
   | **保存** | ユニーク、具体的、適切な範囲 | ステップ6に進む |
   | **改善して保存** | 価値があるが改善が必要 | 改善事項をリスト → 修正 → 再評価（1回） |
   | **[X]に吸収** | 既存スキルに追加すべき | 対象スキルと追加を表示 → ステップ6 |
   | **削除** | 自明、冗長、または抽象的すぎる | 推論を説明して停止 |

**ガイドライン次元**（判定に情報を提供、スコア化されない）：

- **具体性 & 実用性**: すぐに使用可能なコード例またはコマンドを含む
- **スコープ適合性**: 名前、トリガー条件、内容が一致し、単一パターンに焦点
- **ユニーク性**: 既存スキルでカバーされていない価値を提供（チェックリスト結果に基づく）
- **再利用性**: 将来のセッションで現実的なトリガーシナリオが存在

6. **判定固有の確認フロー**

- **改善して保存**: 必須改善 + 修正済みドラフト + 更新チェックリスト/判定を提示; 修正後の判定が **保存** であれば、ユーザー確認後に保存、そうでなければ新しい判定に従う
- **保存**: 保存パス + チェックリスト結果 + 1行の判定根拠 + フル ドラフトを提示 → ユーザー確認後に保存
- **[X]に吸収**: 対象パス + 追加（diff形式）+ チェックリスト結果 + 判定根拠を提示 → ユーザー確認後に追加
- **削除**: チェックリスト結果 + 推論のみを表示（確認不要）

7. 決定した場所に保存/吸収

## ステップ5の出力形式

```
### チェックリスト
- [x] skills/ grep: 重複なし（または: 重複が見つかった → 詳細）
- [x] MEMORY.md: 重複なし（または: 重複が見つかった → 詳細）
- [x] 既存スキル追加: 新規ファイルが適切（または: [X]に追加すべき）
- [x] 再利用性: 確認（または: 一回限り → 削除）

### 判定: 保存 / 改善して保存 / [X]に吸収 / 削除

**根拠:** (判定を説明する1~2文)
```

## 設計根拠

このバージョンは、以前の5次元数値スコアリングルブリック（具体性、実用性、スコープ適合性、非冗長性、カバレッジをそれぞれ1-5でスコア化）をチェックリストベースの総合的判定システムに置き換えます。最新のフロンティアモデル（Opus 4.6以上）は強力なコンテキスト判定を持っています — リッチな質的シグナルを数値スコアに強制すると、ニュアンスが失われ、誤解を招く合計が生じます。総合的アプローチにより、すべての要因が自然に重み付けされて、より正確な保存/削除決定が生じると同時に、明示的なチェックリストで重大なチェックが省略されないようにします。

## ノート

- 自明な修正（タイプミス、単純な構文エラー）を抽出しない
- 一度限りの問題（特定のAPI停止など）を抽出しない
- 将来のセッションで時間を節約するパターンに焦点
- スキルを焦点を絞ったものに保つ — 1スキルあたり1パターン
- 判定が「吸収」の場合、新規ファイルを作成するのではなく既存スキルに追加
