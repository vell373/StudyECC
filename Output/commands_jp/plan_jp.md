# Planコマンド

このコマンドはコードを書く前に包括的な実装計画を作成します。

デフォルトではインラインで実行。Taskツールまたはサブエージェントはデフォルトでは呼び出さない。これにより、エージェントファイルなしでコマンドを出荷するプラグインインストールから`/plan`を使用可能にします。

## このコマンドがすること

1. **要件を改めて述べる** - 何を構築する必要があるかを明確にする
2. **リスクを特定する** - 潜在的な問題とブロッカーを表面化させる
3. **ステップ計画を作成** - 実装をフェーズに分解する
4. **確認を待つ** - コードに触れる前にユーザー承認を受ける必須

## 使用時期

以下の場合に`/plan`を使用:
- 新機能の開始
- 重要なアーキテクチャ変更の実施
- 複雑なリファクタリングの取り組み
- 複数のファイル/コンポーネントが影響を受ける
- 要件が不明確または曖昧

## 動作方法

アシスタントは:

1. **リクエストを分析**して、明確な用語で要件を改めて述べる
2. **フェーズに分解**し、具体的で実行可能なステップを提供
3. **コンポーネント間の依存関係を特定**
4. **リスクを評価**し、潜在的なブロッカーを確認
5. **複雑性を推定**（High/Medium/Low）
6. **計画を提示**し、明示的な確認をあなたから待つ

## 使用例

```
ユーザー: /plan マーケット解決時のリアルタイム通知機能が必要

アシスタント:
# 実装計画: リアルタイムマーケット解決通知

## 要件の改めての説明
- マーケットが解決したときに、ウォッチしているユーザーに通知を送信
- 複数の通知チャネルをサポート（アプリ内、メール、webhook）
- 通知確実配信の確保
- マーケット結果と、ユーザーのポジション結果を含める

## 実装フェーズ

### フェーズ1: データベーススキーマ
- notificationsテーブル（カラム: id、user_id、market_id、type、status、created_at）を追加
- user_notification_preferencesテーブル（チャネル設定用）を追加
- パフォーマンス用にuser_idとmarket_idのインデックスを作成

### フェーズ2: 通知サービス
- lib/notifications.tsに通知サービスを作成
- BullMQ/Redisを使用した通知キューを実装
- 配信失敗用の再試行ロジックを追加
- 通知テンプレートを作成

### フェーズ3: 統合ポイント
- マーケット解決ロジック（ステータスが「resolved」に変更されるときにフック）
- マーケットにポジションを持つすべてのユーザーをクエリ
- 各ユーザーの通知をエンキュー

### フェーズ4: フロントエンドコンポーネント
- ヘッダーに NotificationBellコンポーネントを作成
- NotificationListモーダルを追加
- Supabase subscriptionsでリアルタイム更新を実装
- 通知設定ページを追加

## 依存関係
- Redis（キュー用）
- メールサービス（SendGrid/Resend）
- Supabaseリアルタイム購読

## リスク
- HIGH: メール配信可能性（SPF/DKIM必須）
- MEDIUM: マーケットあたり1000+ユーザーでのパフォーマンス
- MEDIUM: マーケットが頻繁に解決する場合の通知スパム
- LOW: リアルタイム購読オーバーヘッド

## 推定複雑性: MEDIUM
- バックエンド: 4～6時間
- フロントエンド: 3～4時間
- テスト: 2～3時間
- 合計: 9～13時間

**確認待機中**: この計画で進めますか? (yes/no/modify)
```

## 重要な注記

**重大**: このコマンドは、計画を「yes」または「proceed」またはそれに類する肯定的な応答で明示的に確認するまで、コードを書きません。

変更が必要な場合は、以下のように応答:
- "modify: [あなたの変更]"
- "different approach: [代替案]"
- "skip phase 2 and do phase 3 first"

## その他のコマンドとの統合

計画策定後:
- `tdd-workflow`スキルを使用してテスト駆動開発で実装
- ビルドエラーが発生した場合は`/build-fix`を使用
- 完成した実装をレビューする場合は`/code-review`を使用

> **さらに深い計画が必要ですか?** `/prp-plan`を使用してPRD統合、コードベース分析、パターン抽出を備えたアーティファクト生成計画を実施。`/prp-implement`を使用してそれらの計画を厳密な検証ループで実行。

## オプションのPlanner Agent

ECCはエージェントファイルを含むマニュアルインストール用に`planner`エージェントも提供します。ローカルランタイムがそのサブエージェントを既に公開している場合、およびユーザーが明示的に計画委譲を要求する場合にのみそれを使用してください。

`planner`サブエージェントが利用できない場合は、「Agent type 'planner' not found」エラーを表示する代わりにインラインで計画を続行してください。

マニュアルインストールの場合、ソースファイルは以下に存在します:
`agents/planner.md`
