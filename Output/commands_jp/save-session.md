# セッション保存コマンド

進捗中のセッション状態を構造化されたマークダウンファイルとして保存し、別のセッションで続行できます。

このコマンドは `/resume-session` のカウンターパートです。

## 使用時

- セッション終了時に状態を保存する
- コンテキスト制限により新しいセッションを開始する必要がある場合
- 同じプロジェクトで複数の日にまたがって作業する
- 別のチームメイトにセッションを引き継ぐ

## 使用方法

```
/save-session                      # ~/.claude/session-data/ に現在のセッションを保存
/save-session --summary            # サマリーのみを表示（保存しない）
/save-session --export ~/path/file # 特定のパスにエクスポート
```

## セッションファイル形式

セッションファイルは以下の構成を持つマークダウン形式で保存されます:

```markdown
# セッション情報
- 日付: YYYY-MM-DD
- 時間: HH:MM:SS
- プロジェクト: プロジェクト名
- ブランチ: 現在のブランチ名

## 概要
構築中のもの・目標などの 2-3 文での説明

## 達成したもの（動作確認済み）
- ✓ ファイルやタスク1
- ✓ ファイルやタスク2

## 進行中の作業
- 進行中のファイル1（状態）
- 進行中のファイル2（状態）

## 未開始の計画
- 計画済みだが未着手のタスク1
- 計画済みだが未着手のタスク2

## 失敗したアプローチ
- 「試し」と理由
- 「失敗したこと」とその原因

## 未解決の問題・ブロッカー
- 未回答の質問
- 技術的ブロッカー
- 確認が必要な決定

## 次のステップ
現在のセッションで取り組むべき具体的な次のステップ（段落形式）

## コンテキスト（参考資料）
- 関連ファイル: `path/to/file.ts`
- 関連するコマンド: `/command-name`
- 外部参照: URL など

## メモ
その他の雑多なメモ・考察
```

## 使用例

### 例 1: JWT 認証実装セッションの保存

```bash
/save-session
```

生成されるファイル: `~/.claude/session-data/2024-01-15-abc123de-session.tmp`

内容:
```markdown
# セッション情報
- 日付: 2024-01-15
- 時間: 14:32:45
- プロジェクト: my-app
- ブランチ: feat/jwt-auth

## 概要
httpOnly クッキーに保存された JWT トークンを使用したユーザー認証。
登録・ログインエンドポイントはほぼ完成。
ミドルウェア経由のルート保護はまだ開始していません。

## 達成したもの（動作確認済み）
- ✓ ユーザー登録エンドポイント（app/api/auth/register/route.ts）
- ✓ JWT トークン生成と署名
- ✓ bcrypt を用いたパスワードハッシング
- ✓ ユーザーモデルとスキーマの定義

## 進行中の作業
- app/api/auth/login/route.ts
  状態: トークン生成は動作、クッキー設定は未テスト
  
## 未開始の計画
- middleware.ts - 認証トークンの検証とルート保護
- app/login/page.tsx - ログインフォーム UI
- /dashboard ルート保護テスト

## 失敗したアプローチ
- Next-Auth ライブラリ
  理由: カスタム Prisma アダプターと競合、すべてのリクエストで 500 エラー
  
- localStorage に JWT 保存
  理由: SSR ハイドレーション不一致、Next.js App Router と互換性なし

## 未解決の問題・ブロッカー
- cookies().set() は Route Handler 内で動作するのか、それとも Server Actions でのみ動作するのか?
  → Postman でテスト予定

## 次のステップ
app/api/auth/login/route.ts において、cookies().set('token', jwt, { httpOnly: true, secure: true, sameSite: 'strict' }) を使用して JWT を httpOnly クッキーとして設定し、
次に Postman でレスポンスの Set-Cookie ヘッダーをテストしてください。
テスト成功後、middleware.ts を作成してトークン検証を実装してください。

## コンテキスト（参考資料）
- 関連ファイル: `app/api/auth/register/route.ts`, `lib/db.ts`
- 参考: [Next.js Cookies documentation](https://nextjs.org/docs/app/api-reference/functions/cookies)
- テストツール: Postman (localhost:3000/api/auth/login)

## メモ
- `process.env.JWT_SECRET` は .env.local に設定済み
- bcrypt ライブラリはインストール済み（bcryptjs ではなく bcrypt）
- クッキーの Secure フラグは開発では false にする必要があるかもしれない
```

## セッションファイルの構成

### セッション情報セクション
```markdown
# セッション情報
- 日付: YYYY-MM-DD
- 時間: HH:MM:SS
- プロジェクト: プロジェクト名
- ブランチ: git ブランチ名
```

### 概要セクション
プロジェクトの目標と進捗を 2-3 文で簡潔に記述。

### 達成したものセクション
動作確認済みの項目を箇条書きで記述。各行は `- ✓ タイトル` 形式。

### 進行中の作業セクション
各項目は以下の形式:
```
- ファイル名またはタスク名
  状態: 詳細な状態説明（何が完了し、何が残っているか）
```

### 未開始の計画セクション
計画済みだが未着手のタスク。実施順序があれば注記。

### 失敗したアプローチセクション
失敗した試みを記録。形式:
```
- 試みたライブラリ/アプローチ
  理由: 失敗した具体的な理由
```

このセクションは `/resume-session` で「再試行しないこと」として表示されるため、最も重要です。

### 未解決の問題・ブロッカーセクション
- 不明な技術的質問
- API ドキュメント未参照の点
- 外部の承認待ち項目

### 次のステップセクション
具体的で実行可能な次のステップを段落形式で記述。

```
[ファイル名またはタスク]: [具体的な実装内容・テスト内容]
[その後]: [検証方法]
```

## ファイルシステム保存場所

```
~/.claude/session-data/
├── YYYY-MM-DD-session.tmp           # レガシー形式（1日1ファイル）
├── YYYY-MM-DD-abc123de-session.tmp  # 現在の形式（1日複数ファイル可）
└── ...
```

ファイル名の短 ID（`abc123de`）はランダムに生成され、同じ日付に複数セッションが保存されることを防ぎます。

## セッションファイルの有効期限

- **7日以内**: 完全に有効。`/resume-session` で通常の復元可能
- **7日超過**: 警告付きで復元可能。プロジェクト状態が変わっているかもしれません
- **30日超過**: 参照用のみ。新しいセッションを開始する推奨

## セッションファイルの確認

保存前にサマリーを確認:

```bash
/save-session --summary
```

出力:
```
セッションサマリー
════════════════════════════════
プロジェクト: my-app
達成: 4 項目
進行中: 1 項目
未開始: 2 項目
失敗したアプローチ: 2 件
ブロッカー: 1 件
次のステップ: [定義済み]

保存先: ~/.claude/session-data/2024-01-15-abc123de-session.tmp

続行しますか? (y/n)
```

## プロジェクト別セッション管理

複数プロジェクトで作業する場合、各セッションファイルのヘッダーでプロジェクト名を明確に記述してください。

`/resume-session` で特定プロジェクトのセッションを検索する際に参考になります:

```bash
/resume-session --project my-app
```

## セッション引き継ぎ

セッションファイルをエクスポートしてチームメイトに共有:

```bash
/save-session --export ~/Downloads/session-handoff.md
```

受け取った側:
```bash
/resume-session ~/Downloads/session-handoff.md
```
