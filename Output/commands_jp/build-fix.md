# ビルドと修正

最小限で安全な変更でビルドと型エラーを段階的に修正します。

## ステップ1：ビルドシステムを検出

プロジェクトのビルドツールを特定して、ビルドを実行します：

| インジケーター | ビルドコマンド |
|-----------|---------------|
| `package.json` の `build` スクリプト | `npm run build` または `pnpm build` |
| `tsconfig.json`（TypeScript のみ） | `npx tsc --noEmit` |
| `Cargo.toml` | `cargo build 2>&1` |
| `pom.xml` | `mvn compile` |
| `build.gradle` | `./gradlew compileJava` |
| `go.mod` | `go build ./...` |
| `pyproject.toml` | `python -m compileall -q .` または `mypy .` |

## ステップ2：エラーを解析してグループ化

1. ビルドコマンドを実行して stderr をキャプチャ
2. ファイルパスでエラーをグループ化
3. 依存関係の順序でソート（ロジックエラーの前にインポート/型を修正）
4. 進捗追跡のため総エラー数をカウント

## ステップ3：修正ループ（一度に1つのエラー）

各エラーについて：

1. **ファイルを読む** — Read ツールを使用して、エラー周辺のコンテキスト（エラーの周囲10行）を確認
2. **診断** — 根本原因を特定（インポート不足、型の誤り、構文エラー）
3. **最小限に修正** — Edit ツールを使用して、エラーを解決する最小限の変更
4. **ビルドを再実行** — エラーが解消され、新しいエラーが導入されていないことを確認
5. **次へ移動** — 残りのエラーに進む

## ステップ4：ガードレール

以下の場合は停止してユーザーに確認：
- 修正が**解決するより多くのエラーを導入**する
- **同じエラーが3回の試行後も残存**（より深い問題の可能性）
- 修正が**アーキテクチャの変更を必要**（ビルド修正ではなく）
- ビルドエラーが**依存関係不足から発生**（`npm install`、`cargo add` などが必要）

## ステップ5：サマリー

結果を表示：
- 修正されたエラー（ファイルパス付き）
- 残っているエラー（存在する場合）
- 導入された新しいエラー（ゼロであるべき）
- 未解決の問題への次のステップの提案

## リカバリー戦略

| 状況 | アクション |
|-----------|--------|
| モジュール/インポート不足 | パッケージがインストールされているか確認。インストールコマンドを提案 |
| 型の不一致 | 両方の型定義を読み込み。より狭い型を修正 |
| 循環依存 | インポートグラフで循環を特定。抽出を提案 |
| バージョン競合 | `package.json` / `Cargo.toml` のバージョン制約を確認 |
| ビルドツールの設定ミス | 設定ファイルを読み込み。動作する既定値と比較 |

安全性のため一度に1つのエラーを修正。リファクタリングよりも最小限の差分を優先。
