# コードレビュー

> PR レビューモードは、Wirasm による PRPs-agentic-eng から適応。PRP ワークフローシリーズの一部。

**入力**: $ARGUMENTS

---

## モード選択

`$ARGUMENTS` が PR 番号、PR URL、または `--pr` を含む場合：
→ 以下の **PR レビューモード** にジャンプします。

その他の場合：
→ **ローカルレビューモード** を使用します。

---

## ローカルレビューモード

未コミット変更の包括的なセキュリティと品質レビュー。

### フェーズ1 — 収集

```bash
git diff --name-only HEAD
```

変更されたファイルがない場合は停止：「レビューするものがありません。」

### フェーズ2 — レビュー

変更された各ファイルを完全に読み込みます。以下をチェック：

**セキュリティ問題（重大）:**
- ハードコードされた認証情報、API キー、トークン
- SQL インジェクション脆弱性
- XSS 脆弱性
- 入力検証不足
- 不安全な依存関係
- パストラバーサルリスク

**コード品質（高）:**
- 関数 > 50 行
- ファイル > 800 行
- ネスト深度 > 4 レベル
- エラーハンドリング不足
- console.log ステートメント
- TODO/FIXME コメント
- パブリック API 用の JSDoc 不足

**ベストプラクティス（中）:**
- ミューテーションパターン（代わりに不変性を使用）
- コード/コメント内の絵文字使用
- 新しいコード用のテスト不足
- アクセシビリティ問題（a11y）

### フェーズ3 — 報告

報告を生成：
- 重大度: 重大、高、中、低
- ファイルの場所と行番号
- 問題の説明
- 提案される修正

重大または高い問題が見つかった場合、コミットをブロック。
セキュリティ脆弱性があるコードは決して承認しないでください。

## PR レビューモード

包括的な GitHub PR レビュー — diff を取得し、完全なファイルを読み込み、検証を実行し、レビューを投稿します。

### フェーズ1 — 取得

入力を解析して PR を決定：

| 入力 | アクション |
|---|---|
| 数値（例：`42`） | PR 番号として使用 |
| URL（`github.com/.../pull/42`） | PR 番号を抽出 |
| ブランチ名 | `gh pr list --head <branch>` 経由で PR を検索 |

```bash
gh pr view <NUMBER> --json number,title,body,author,baseRefName,headRefName,changedFiles,additions,deletions
gh pr diff <NUMBER>
```

PR が見つからない場合は、エラーで停止。後のフェーズのために PR メタデータを保存。

### フェーズ2 — コンテキスト

レビュー コンテキストを構築：

1. **プロジェクトルール** — `CLAUDE.md`、`.claude/docs/`、およびコントリビューションガイドラインを読み込み
2. **PRP アーティファクト** — この PR に関連する実装コンテキストのために `.claude/PRPs/reports/` と `.claude/PRPs/plans/` をチェック
3. **PR の意図** — PR の説明から目標、リンクされた問題、テスト計画を解析
4. **変更されたファイル** — すべての変更されたファイルをリストし、タイプ別（ソース、テスト、設定、ドキュメント）に分類

### フェーズ3 — レビュー

各変更されたファイルを**完全に**読み込み（diff ハンクだけでなく — 周囲のコンテキストが必要）。

PR レビュー用に、PR ヘッド リビジョンの完全なファイル内容を取得：
```bash
gh pr diff <NUMBER> --name-only | while IFS= read -r file; do
  gh api "repos/{owner}/{repo}/contents/$file?ref=<head-branch>" --jq '.content' | base64 -d
done
```

7 つのカテゴリ全体でレビューチェックリストを適用：

| カテゴリ | チェック対象 |
|---|---|
| **正確性** | ロジックエラー、off-by-one、null ハンドリング、エッジケース、レース条件 |
| **型の安全性** | 型の不一致、安全でない キャスト、`any` 使用、ジェネリック不足 |
| **パターン準拠** | プロジェクト規約に準拠（命名、ファイル構造、エラーハンドリング、インポート） |
| **セキュリティ** | インジェクション、認証ギャップ、シークレット露出、SSRF、パストラバーサル、XSS |
| **パフォーマンス** | N+1 クエリ、インデックス不足、無制限ループ、メモリリーク、大きなペイロード |
| **完全性** | テスト不足、エラーハンドリング不足、不完全なマイグレーション、ドキュメント不足 |
| **保守性** | デッドコード、マジックナンバー、深いネスト、不明瞭な命名、型不足 |

各検索に重大度を割り当て：

| 重大度 | 意味 | アクション |
|---|---|---|
| **重大** | セキュリティ脆弱性またはデータ損失リスク | マージ前に修正必須 |
| **高** | バグまたはロジックエラーが問題を引き起こす可能性 | マージ前に修正推奨 |
| **中** | コード品質の問題またはベストプラクティス不足 | 修正をお勧め |
| **低** | スタイル指摘または軽微な提案 | オプション |

### フェーズ4 — 検証

利用可能な検証コマンドを実行：

設定ファイル（`package.json`、`Cargo.toml`、`go.mod`、`pyproject.toml` など）からプロジェクトタイプを検出し、適切なコマンドを実行：

**Node.js / TypeScript**（`package.json` あり）:
```bash
npm run typecheck 2>/dev/null || npx tsc --noEmit 2>/dev/null  # 型チェック
npm run lint                                                    # Lint
npm test                                                        # テスト
npm run build                                                   # ビルド
```

**Rust**（`Cargo.toml` あり）:
```bash
cargo clippy -- -D warnings  # Lint
cargo test                   # テスト
cargo build                  # ビルド
```

**Go**（`go.mod` あり）:
```bash
go vet ./...    # Lint
go test ./...   # テスト
go build ./...  # ビルド
```

**Python**（`pyproject.toml` / `setup.py` あり）:
```bash
pytest  # テスト
```

検出されたプロジェクトタイプに適用されるコマンドのみを実行。各コマンドについて成功/失敗を記録。

### フェーズ5 — 決定

検索結果に基づいて推奨を形成：

| 条件 | 決定 |
|---|---|
| ゼロの重大/高い問題、検証成功 | **承認** |
| のみ中/低い問題、検証成功 | **承認** とコメント |
| 任意の高い問題または検証失敗 | **変更をリクエスト** |
| 任意の重大問題 | **ブロック** — マージ前に修正が必須 |

特殊なケース：
- ドラフト PR → 常に **コメント** を使用（承認/ブロックではなく）
- ドキュメント/設定変更のみ → より軽いレビュー、正確性に焦点
- 明示的な `--approve` または `--request-changes` フラグ → 決定をオーバーライド（ただし、すべての検出をレポート）

### フェーズ6 — 報告

`.claude/PRPs/reviews/pr-<NUMBER>-review.md` でレビュー アーティファクトを作成：

```markdown
# PR レビュー: #<NUMBER> — <TITLE>

**レビュー日**: <日付>
**著者**: <著者>
**ブランチ**: <ヘッド> → <ベース>
**決定**: 承認 | 変更をリクエスト | ブロック

## 概要
<全体的な評価の1～2文>

## 検査結果

### 重大
<検査結果または「なし」>

### 高
<検査結果または「なし」>

### 中
<検査結果または「なし」>

### 低
<検査結果または「なし」>

## 検証結果

| チェック | 結果 |
|---|---|
| 型チェック | 成功 / 失敗 / スキップ |
| Lint | 成功 / 失敗 / スキップ |
| テスト | 成功 / 失敗 / スキップ |
| ビルド | 成功 / 失敗 / スキップ |

## レビュー済みファイル
<変更タイプ付きのファイルのリスト: 追加/変更/削除>
```

### フェーズ7 — 公開

GitHub にレビューを投稿：

```bash
# 承認の場合
gh pr review <NUMBER> --approve --body "<レビューの概要>"

# 変更をリクエストする場合
gh pr review <NUMBER> --request-changes --body "<必要な修正を含む概要>"

# コメントのみ（ドラフト PR または情報提供）
gh pr review <NUMBER> --comment --body "<概要>"
```

特定の行への行内コメント用に、GitHub レビューコメント API を使用：
```bash
gh api "repos/{owner}/{repo}/pulls/<NUMBER>/comments" \
  -f body="<コメント>" \
  -f path="<ファイル>" \
  -F line=<行番号> \
  -f side="RIGHT" \
  -f commit_id="$(gh pr view <NUMBER> --json headRefOid --jq .headRefOid)"
```

別の方法として、複数の行内コメントを含む単一のレビューを一度に投稿：
```bash
gh api "repos/{owner}/{repo}/pulls/<NUMBER>/reviews" \
  -f event="COMMENT" \
  -f body="<全体的な概要>" \
  --input comments.json  # [{"path": "file", "line": N, "body": "comment"}, ...]
```

### フェーズ8 — 出力

ユーザーに報告：

```
PR #<NUMBER>: <TITLE>
決定: <承認|変更をリクエスト|ブロック>

問題: <重大数> 重大, <高数> 高, <中数> 中, <低数> 低
検証: <成功数>/<総数> チェック成功

アーティファクト:
  レビュー: .claude/PRPs/reviews/pr-<NUMBER>-review.md
  GitHub: <PR URL>

次のステップ:
  - <決定に基づく文脈に応じた提案>
```

## エッジケース

- **`gh` CLI なし**: ローカルのみのレビューにフォールバック（diff を読み込み、GitHub 公開をスキップ）。ユーザーに警告。
- **発散したブランチ**: レビュー前に `git fetch origin && git rebase origin/<base>` を提案。
- **大型PR（>50ファイル）**: レビュー範囲について警告。最初にソース変更、次にテスト、その後設定/ドキュメントに焦点。
