# リファクタークリーン

デッドコードを安全に特定して削除し、各ステップでテスト検証を実施します。

## ステップ1: デッドコードを検出

プロジェクトタイプに基づいて解析ツールを実行:

| ツール | 検出内容 | コマンド |
|--------|--------|---------|
| knip | 未使用エクスポート、ファイル、依存関係 | `npx knip` |
| depcheck | 未使用 npm 依存関係 | `npx depcheck` |
| ts-prune | 未使用 TypeScript エクスポート | `npx ts-prune` |
| vulture | 未使用 Python コード | `vulture src/` |
| deadcode | 未使用 Go コード | `deadcode ./...` |
| cargo-udeps | 未使用 Rust 依存関係 | `cargo +nightly udeps` |

利用可能なツールがない場合、Grep を使用してインポートがないエクスポートを検出:

```
# エクスポートを検出し、どこでインポートされているか確認
```

## ステップ2: 検出結果を分類

検出結果を安全性レベルで分類:

| レベル | 例 | アクション |
|--------|-----|---------|
| **安全** | 未使用ユーティリティ、テストヘルパー、内部関数 | 安心して削除 |
| **注意** | コンポーネント、API ルート、ミドルウェア | 動的インポート・外部コンシューマーを検証 |
| **危険** | コンフィグファイル、エントリーポイント、型定義 | 削除する前に調査 |

## ステップ3: 安全な削除ループ

各「安全」な項目について:

1. **フルテストスイートを実行** — ベースラインを確立（すべて緑）
2. **デッドコードを削除** — Edit ツールで surgical 削除を実行
3. **テストスイートを再実行** — 何も破損していないか検証
4. **テスト失敗時** — すぐに `git checkout -- <file>` で revert し、この項目をスキップ
5. **テスト成功時** — 次の項目に進む

## ステップ4: 注意が必要な項目を処理

「注意」レベルの項目を削除する前に:
- 動的インポートを検索: `import()`, `require()`, `__import__`
- 文字列参照を検索: ルート名、コンフィグ内のコンポーネント名
- 公開パッケージ API からエクスポートされているか確認
- 外部コンシューマーがないか検証（公開されている場合は依存関係を確認）

## ステップ5: 重複を統合

デッドコードを削除した後、以下を確認:
- 近似重複関数（80%以上類似） — 1つに統合
- 冗長な型定義 — 統合
- 付加価値のないラッパー関数 — インラインに
- 目的のない再エクスポート — 削除して間接参照を削除

## ステップ6: サマリー

結果をレポート:

```
デッドコードクリーンアップ
──────────────────────────────
削除: 12 個の未使用関数
      3 個の未使用ファイル
      5 個の未使用依存関係
スキップ: 2 項目（テスト失敗）
削除行数: 約 450 行削除
──────────────────────────────
すべてのテストが成功: PASS
```

## ルール

- **テスト実行なしで削除しない**
- **一度に1つの削除** — アトミックな変更により簡単に rollback 可能
- **不確実な場合はスキップ** — デッドコードを残す方が本番環境の障害より良い
- **クリーンアップ中にリファクタリングしない** — 関心を分離（クリーンアップ先、リファクタリング後）
