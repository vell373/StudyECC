# /learn-eval - 抽出、評価、保存

品質ゲート、保存場所の決定、スキルファイルを書き込む前の知識配置認識により `/learn` を拡張します。

## 抽出対象

以下を探します:

1. **エラー解決パターン** — 根本原因 + 修正 + 再利用性
2. **デバッグ技術** — 明白ではないステップ、ツール組み合わせ
3. **回避策** — ライブラリの癖、API 制限、バージョン固有の修正
4. **プロジェクト固有パターン** — 規約、アーキテクチャ決定、統合パターン

## プロセス

1. セッションを抽出可能なパターンについてレビュー
2. 最も価値のある/再利用可能な洞察を特定

3. **保存場所を決定:**
   - 質問: 「このパターンは異なるプロジェクトで有用だろうか?」
   - **グローバル** (`~/.claude/skills/learned/`): 2+ プロジェクトで再利用可能な汎用パターン（bash 互換性、LLM API の動作、デバッグ技術など）
   - **プロジェクト** (現在のプロジェクトの `.claude/skills/learned/`): プロジェクト固有の知識（特定の設定ファイルの癖、プロジェクト固有のアーキテクチャ決定など）
   - 不確定な場合はグローバルを選択（グローバル → プロジェクトの移動は逆よりも簡単）

4. このフォーマットを使用してスキルファイルの下書き:

```markdown
---
name: pattern-name
description: "130 文字以内"
user-invocable: false
origin: auto-extracted
---

# [説明的なパターン名]

**抽出:** [日付]
**コンテキスト:** [これが適用される場合の簡単な説明]

## 問題
[このパターンが解決する問題 - 具体的に]

## 解決策
[パターン/技術/回避策 - コード例を含む]

## 使用時機
[トリガー条件]
```

5. **品質ゲート — チェックリスト + 全体的な判定**

   ### 5a. 必須チェックリスト（ファイルを実際に読んで検証）

   評価の下書きの前に、以下を**すべて**実行:

   - [ ] キーワード別に `~/.claude/skills/` と関連するプロジェクト `.claude/skills/` ファイルを grep してコンテンツオーバーラップをチェック
   - [ ] MEMORY.md（プロジェクトとグローバル両方）でオーバーラップをチェック
   - [ ] 既存スキルへの追加で十分かどうかを検討
   - [ ] これが再利用可能なパターンであって、一回限りの修正ではないことを確認

   ### 5b. 全体的な判定

   チェックリスト結果と下書き品質を統合し、以下のいずれか**1つ**を選択:

   | 判定 | 意味 | 次のアクション |
   |------|------|-------------|
   | **保存** | ユニークで具体的、良いスコープ | ステップ 6 に進行 |
   | **改善して保存** | 価値があるが改善が必要 | 改善リストを表示 → 修正 → 再評価（1回） |
   | **[X]に吸収** | 既存スキルに追加すべき | ターゲットスキルと追加を表示 → ステップ 6 |
   | **ドロップ** | 些細、冗長、または非常に抽象的 | 推理を説明して停止 |

**ガイドライン次元**（判定を知らせる、スコア付けされていない）:

- **具体性と実行可能性**: すぐに使用可能なコード例またはコマンドを含む
- **スコープ適合**: 名前、トリガー条件、コンテンツが整列し、単一パターンに焦点
- **ユニーク性**: チェックリスト結果によって知らされた既存スキルではカバーされていない価値を提供
- **再利用性**: 将来のセッションで現実的なトリガーシナリオが存在

6. **判定固有の確認フロー**

- **改善して保存**: 必要な改善 + 修正下書き + 更新されたチェックリスト/判定を提示; 修正判定が **保存** の場合はユーザー確認後保存、そうでなければ新しい判定に従う
- **保存**: 保存パス + チェックリスト結果 + 1 行の判定根拠 + 完全下書きを提示 → ユーザー確認後保存
- **[X]に吸収**: ターゲットパス + 追加（diff 形式） + チェックリスト結果 + 判定根拠を提示 → ユーザー確認後追加
- **ドロップ**: チェックリスト結果 + 推理のみを表示（確認不要）

7. 決定された場所に保存/吸収

## ステップ 5 の出力フォーマット

```
### チェックリスト
- [x] skills/ grep: オーバーラップなし（または: オーバーラップ見つかる → 詳細）
- [x] MEMORY.md: オーバーラップなし（または: オーバーラップ見つかる → 詳細）
- [x] 既存スキル追加: 新規ファイルが適切（または: [X] に追加すべき）
- [x] 再利用性: 確認済み（または: 一回限り → ドロップ）

### 判定: 保存 / 改善して保存 / [X]に吸収 / ドロップ

**根拠:** (判定を説明する 1-2 文)
```

## デザイン根拠

このバージョンは前の 5 次元の数値スコアリングルーブリック（具体性、実行可能性、スコープ適合、非冗長性、カバレッジを 1-5 でスコア）をチェックリストベースの全体的な判定システムに置き換えます。モダンフロンティアモデル（Opus 4.6+）は強いコンテキスト判断を持ち — 豊かな定性信号を数値スコアに強制するとニュアンスが失われ、誤解を招く合計を生成できます。全体的なアプローチはモデルがすべての要因を自然に計量することを可能にし、より正確な保存/ドロップ決定を生成しながら、明示的なチェックリストは重大なチェックがスキップされないようにします。

## 注記

- 些細な修正（タイプミス、単純な構文エラー）を抽出しない
- 一時的な問題（特定の API 停止など）を抽出しない
- 将来のセッションで時間を節約するパターンに焦点
- スキルを焦点を絞る — スキルごとに 1 つのパターン
- 判定が吸収の場合、新規ファイルを作成するのではなく既存スキルに追加
