# エージェント間のコンテキスト分離と設計思想

**日付**: 2026-05-03
**関連解析**: → [01_analysis/agents_code-reviewer.md], [01_analysis/agents_architect.md], [01_analysis/agents_planner.md]
**関連ECC**: ECC/agents/, ECC/AGENTS.md

## 今日気づいたこと
エージェント（Planner, Architect, Coder, Reviewerなど）は、同じチャットセッションの全履歴を共有するのではなく、物理的・論理的にコンテキストが分離されている。
- 各エージェントは呼び出された際に「自らコンテキストを収集（Gather Context）」する手順を持っている。
- 情報の受け渡しは「セッションの記憶」ではなく、「ファイル（Markdown, ADR, 設計書）」を介して行われる。
- これにより、人間組織の部署間連携のような、ドキュメントベースのハンドオフが実現されている。

## 自分の解釈
この設計は、単なる機能分割ではなく「客観性の担保」と「リソース最適化」の両立だと理解した。
- **客観性**: レビューアーが実装者の苦労話を知らないことで、成果物であるコードそのものを厳格に評価できる。
- **リソース最適化**: 全履歴を全エージェントに持たせると、LLMのコンテキスト窓がすぐに溢れ、推論精度が落ちる（Context Rot）。必要な情報だけをその都度読み込ませることで、常に高いパフォーマンスを維持できる。
- **自律性**: 「言われたことをやる」のではなく、「今の状況から必要な情報を自分で探して（Read/Grep）判断する」という、より自律性の高いエージェント像が描かれている。

## 個人開発・仕事への応用
AIを「何でも知っている万能な一人」として扱うのではなく、「特定の情報だけを与えた専門家の集団」として構成することで、大規模な開発でも破綻しないワークフローが作れそう。
- 複雑な指示を一つのプロンプトに詰め込まず、出力結果をファイルに書き出し、それを次の専門エージェントが読み取る「ドキュメント駆動プロンプティング」を取り入れたい。

## 次に調べること
- 実際に複数のエージェントが並列で動く際の競合制御や、ファイルベースのハンドオフで情報が欠落した際のリカバリー策。
- `relationships.md` などの「長期記憶」が、時間の経過とともにどのように整理・集約されていくのか。
