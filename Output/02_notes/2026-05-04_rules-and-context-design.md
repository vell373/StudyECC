# ルールとコンテキストの設計思想の違い

**日付**: 2026-05-04
**関連解析**: → [../01_analysis/cursor/cursor_overview.md]
**関連Question**: → [../06_questions/open/where-to-store-code-review-rules.md]

## やり取りした内容（事実のみ）
- **ユーザーの考察と問題意識**:
  - ルール（rules）にはある程度不変・固定的なものを入れるべきだと考えている。
  - しかし、ファイルが `rules` にあるか `context` にあるか自体の差はそこまで明確ではないと感じている。鍵となるのは「エージェントやスキルが、それをどのように呼び出し、どれくらい重要度を置くか」という点である。
  - コンテキストに置いても必要な時に確実に読み込めるなら問題ないし、逆にルールに置いても必要な時だけ参照される仕組みなら成立する。
  - したがって、「どちらでもいい」という状態で配置していくと、ルール的な情報が各所に分散して管理が破綻してしまう。
  - だからこそ、rules と context の特性の違いや呼び出され方のメカニズムを理解したうえで、どういう設計思想でディレクトリを分割・配置していくべきかを知る必要がある。

- **アーキテクチャのメカニズムに基づく回答・整理**:
  - `rules` と `context(skills)` の最大の違いは、呼び出し方が **受動的（強制注入）** か **能動的（オンデマンド取得）** かという点にある。
  - **Rules**: Cursor等の機能により、特定ファイル編集時などに自動的・強制的にAIのプロンプトに結合される。探す手間がない反面、詰め込みすぎると注意力が低下しコンテキストを圧迫する。
  - **Context / Skills**: ツール等を使ってAI（エージェント）が能動的にファイルを取得する。コンテキストを圧迫しないが、適切なタイミングで取得するようトリガーを設計する必要がある。
  - **設計の結論**: 「常に守るべき不変のルール」は `rules` に置き、「特定のタスク時（コードレビュー時など）にのみ参照すべき手順や重厚なドメイン知識」は `context` に配置することで、管理の破綻を防ぎ、AIのパフォーマンスも維持できる。
