# エージェントの動的プロビジョニング（スターターセット方式）

**日付**: 2026-05-04
**関連解析**: → [02_notes/2026-05-04_ecc-mvp-migration-list.md]
**関連ECC**: -

## 今日気づいたこと
- 新規プロジェクトを立ち上げる際、いきなり全てのエージェント（PlannerやArchitect等）を移植するのではなく、**「要求確認（壁打ち）」**と**「必要なエージェントの選定・移植」**を行うための「スターターセット」を最初に用意するアプローチが非常に有効である。
- プロジェクト立ち上げの理想的なフロー：
  1. 母体（ECCのような全エージェントが揃った人材プール）を用意する。
  2. 新規プロジェクトには「Consultant（壁打ち役）」と「Assigner（人事役）」、および最低限のルールのみを配置する。
  3. Consultantと対話し、要件定義書（`PRODUCT-BRIEF.md`）を作成する。
  4. Assignerが要件定義書を読み、母体から必要なエージェント（例：TypeScript特化、DB特化など）を見繕い、新規プロジェクトに移植（コピー）する。
  5. 最適化されたチーム編成が完了した段階で、Planner → Architect → 実装へと進む。

## 自分の解釈
- この「動的アサイン型（エージェント・ファクトリー）」方式をとることで、プロジェクトが不要なエージェントで肥大化せず、AIのコンテキスト（文脈）を軽量かつ高精度に保つことができる。
- 単一のプロジェクト内で全要件を満たそうとするのではなく、エージェントを「必要な時に必要な人材だけを現場に呼ぶ」という運用モデルが、これからのAgentic Developmentのスタンダードになる。

## 個人開発・仕事への応用
- 新しいプロジェクト（拡張機能開発など）を始める際、まずはこの「スターターセット（Consultant + Assigner）」のプロンプトを作成し、テンプレート化する。
- 開発途中でスコープの拡大があった場合でも、再度Assignerに頼んで必要な専門エージェントを母体から追加派遣してもらう仕組みにする。

## 次に調べること
- Consultant（壁打ち役）エージェントの具体的なプロンプト設計（ECCの `product-lens` スキルなどを参考にする）。
- Assigner（人事役）が他のディレクトリ（母体）からファイルを正しく選定し、現在のディレクトリにコピーするためのプロンプト指示とツール権限の持たせ方。
