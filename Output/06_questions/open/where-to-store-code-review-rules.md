# コード規約やレビュー観点は rules と skills（あるいは別フォルダ）どちらに置くべきか？

**起票日**: 2026-05-04
**状態**: 未解決
**関連**: ECC/.cursor/rules/, ECC/.cursor/skills/

## 疑問の内容
コードレビューの確認事項やコーディング規約のようなドキュメントを作成する場合、ECCの設計思想では `rules/` に置くのが正しいのか、それとも `context/` などの別フォルダを作って「コードレビュー観点」としてまとめるのが良いのか、どちらが適切か。

## なぜ疑問に思ったか
`rules/` には言語ごとのスタイルなどが定義されている一方で、実際のレビューという「タスク・フロー」の観点になると、それが「常に守るべき静的なルール」なのか「作業時に参照すべきコンテキスト・手順」なのか境界が曖昧に感じられたため。

## 現時点の仮説
- 「ルール（rules）」にはある程度不変・固定的なものを入れるべきだと考えている。
- しかし、ファイルが `rules` にあるか `context` にあるか自体の明確な差は現状あまり感じられていない。重要なのは「エージェントやスキルが、その情報をどのように呼び出し、どれくらい重要度を置くか（重み付け）」という点にある。
- コンテキストに置いても必要な時に確実に読み込めるなら問題ないし、逆にルールに置いても毎回全読み込みされるのではなく必要な時だけ参照される仕組みならどちらでも成立する。
- したがって、「どちらでもいい」という状態で配置していくと、ルール的な情報が各所に分散して管理が破綻してしまう。
- だからこそ、rules と context の特性の違いや呼び出され方のメカニズムを理解したうえで、どういう設計思想でディレクトリを分割・配置していくべきかを知る必要がある。
