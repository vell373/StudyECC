# ECC プロジェクト概要・思想・設計原則

ECC (Everything Claude Code) は、「AIが人間と同等、あるいはそれ以上の品質でコードを書くためには何が必要か」という問いへの答えとして設計されました。

---

## 🌟 5つの核心原則 (Core Principles)

ECCのすべての挙動は、以下の5つの原則に基づいています。

### 1. Agent-First (エージェント優先)
- **思想**: 全知全能の1つのAIではなく、特定のタスク（計画、レビュー、各言語の解決）に特化した「専門家エージェント」に早期に委譲する。
- **目的**: コンテキストを絞り込み、特定のドメインにおけるAIの推論精度を最大化する。
- **実装**: スラッシュコマンド（例: `/go-review`）がそれぞれの専門エージェントを即座に起動する。

### 2. Test-Driven (テスト駆動)
- **思想**: 「動くコード」ではなく「検証されたコード」のみを正義とする。
- **目的**: AIによるデグレード（先祖返り）やバグの混入を物理的に防ぐ。
- **実装**: **カバレッジ80%以上**を必須とし、コードを書く前にテストを書く（TDD）フローを `tdd-guide` エージェントが強制する。

### 3. Security-First (セキュリティ優先)
- **思想**: 脆弱性や秘密情報は「コミットされる前」に検知されなければならない。
- **目的**: AIによる事故（APIキーの公開、インジェクション脆弱性の混入）を防ぐ。
- **実装**: `security-reviewer` による常時スキャンと、Gitコミット前に自動発火する秘密情報検知フック。

### 4. Immutability (イミュータビリティ)
- **思想**: 既存のオブジェクトやコードを直接変更するのではなく、新しい状態を生成して置き換える。
- **目的**: 変更の影響範囲を予測可能にし、デバッグを容易にする。
- **実装**: スプレッド演算子による非破壊的更新の推奨、小さな関数・小さなファイルへの分割。

### 5. Plan Before Execute (実行前の計画)
- **思想**: AIがいきなりコードを書き始めるのは「迷走」の始まりである。
- **目的**: アーキテクチャの整合性を保ち、無駄な試行錯誤を減らす。
- **実装**: `/plan` コマンドによる段階的な実装計画の作成と、ユーザーによる「承認」がなければコードを1行も書かないプロンプト定義。

## 🛠 ワークフロー・サーフェス・ポリシー

ECCでは、AIが提供する「機能」を以下の順序で優先します。

1.  **Skills (スキルの正統性)**:
    - ワークフロー、再利用可能なパターン、ドメイン知識は `skills/` に集約されます。
    - 他のプロジェクトに「移植」可能な形でナレッジが蓄積されます。
2.  **Commands (インターフェース)**:
    - スラッシュコマンドは、ユーザーがスキルやエージェントを簡単に呼び出すための「窓口」です。
    - 内部的にはエージェントやスクリプトを連鎖（オーケストレーション）させます。
3.  **Hooks (ガードレール)**:
    - ユーザーが意識しなくても、ツール使用時に裏側で品質チェックが走り、基準を満たさない場合はツール実行を「ブロック」します。

## 📝 コーディング・スタンダード

ECCが自律的にコードを書く際に守るべき「聖典」です。

- **ファイルサイズ**: 1ファイル 200〜400行を推奨。最大800行。
- **関数サイズ**: 1関数 50行以内。
- **ディレクトリ構造**: 機能（ドメイン）ベースのフォルダ分け。
- **依存性注入**: 疎結合なコンポーネント設計。
- **ドキュメント**: 公開APIには必ずJSDoc（あるいは言語別の標準ドキュメント）を付与。
