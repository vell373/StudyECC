# コードレビュー基準

## 目的

コードレビューは、マージ前にコード品質、セキュリティ、保守性を確保します。このルールはコードレビューを実施するタイミングと方法を定義しています。

## レビュー対象

**必須レビュートリガー:**

- コード記述または修正後
- 共有ブランチへのコミット前
- セキュリティに関連するコード変更時（認証、支払い、ユーザーデータ）
- アーキテクチャ変更時
- プルリクエストマージ前

**レビュー前の必須条件:**

レビューをリクエストする前に以下を確認:

- すべての自動チェック（CI/CD）が合格している
- マージコンフリクトが解決されている
- ブランチが対象ブランチより最新である

## レビューチェックリスト

コード完了前に確認:

- [ ] コードが読みやすく、良く命名されている
- [ ] 関数が焦点を絞っている（50行未満）
- [ ] ファイルが凝集度の高い（800行未満）
- [ ] ネストが深くない（4段階未満）
- [ ] エラーが明示的に処理されている
- [ ] ハードコード化されたシークレット認証情報がない
- [ ] console.log またはデバッグステートメントがない
- [ ] 新機能に対するテストが存在している
- [ ] テストカバレッジが80%以上である

## セキュリティレビュートリガー

**以下の場合、security-reviewer エージェントを使用:**

- 認証または認可コード
- ユーザー入力処理
- データベースクエリ
- ファイルシステム操作
- 外部API呼び出し
- 暗号化操作
- 支払いまたは財務コード

## レビュー重大度レベル

| レベル | 意味 | アクション |
|-------|---------|--------|
| CRITICAL | セキュリティ脆弱性またはデータ損失リスク | **ブロック** - マージ前に修正が必須 |
| HIGH | バグまたは重大な品質問題 | **警告** - マージ前に修正を推奨 |
| MEDIUM | 保守性の懸念 | **情報** - 修正を検討 |
| LOW | スタイルまたは軽微な提案 | **注記** - オプション |

## エージェント利用

コードレビューに以下のエージェントを使用:

| エージェント | 目的 |
|-------|---------|
| **code-reviewer** | 一般的なコード品質、パターン、ベストプラクティス |
| **security-reviewer** | セキュリティ脆弱性、OWASP Top 10 |
| **typescript-reviewer** | TypeScript/JavaScript固有の問題 |
| **python-reviewer** | Python固有の問題 |
| **go-reviewer** | Go固有の問題 |
| **rust-reviewer** | Rust固有の問題 |

## レビューワークフロー

```
1. git diffを実行して変更を理解する
2. セキュリティチェックリストを最初にチェック
3. コード品質チェックリストをレビュー
4. 関連テストを実行
5. カバレッジ >= 80%を確認
6. 詳細なレビューに適切なエージェントを使用
```

## 捕捉する一般的な問題

### セキュリティ

- ハードコード化された認証情報（APIキー、パスワード、トークン）
- SQLインジェクション（クエリの文字列連結）
- XSS脆弱性（エスケープされていないユーザー入力）
- パストラバーサル（サニタイズされていないファイルパス）
- CSRF保護の欠落
- 認証バイパス

### コード品質

- 大きな関数（50行超） - 小さく分割
- 大きなファイル（800行超） - モジュール抽出
- 深いネスト（4段階超） - 早期リターン使用
- エラーハンドリングの欠落 - 明示的に処理
- ミューテーションパターン - 不変操作を優先
- テストの欠落 - テストカバレッジを追加

### パフォーマンス

- N+1クエリ - JOIN またはバッチ処理を使用
- ページネーション欠落 - LIMIT をクエリに追加
- 無制限クエリ - 制約を追加
- キャッシング欠落 - 高コスト操作をキャッシュ

## 承認基準

- **承認**: CRITICAL または HIGH 問題なし
- **警告**: HIGH 問題のみ（注意してマージ）
- **ブロック**: CRITICAL 問題が発見された
