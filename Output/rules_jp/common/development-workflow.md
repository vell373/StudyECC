# 開発ワークフロー

> このファイルは [common/git-workflow.md](./git-workflow.md) を拡張し、Git操作の前に発生する完全な機能開発プロセスを説明しています。

機能実装ワークフローは開発パイプラインを説明しています：リサーチ、計画、TDD、コードレビュー、その後 Git へのコミット。

## 機能実装ワークフロー

0. **リサーチ＆再利用** _（新しい実装の前に必須）_
   - **GitHub コード検索を最初に:** `gh search repos` と `gh search code` を実行して、新しいものを書く前に既存実装、テンプレート、パターンを探します。
   - **ライブラリドキュメントを次に:** Context7 または主要ベンダードキュメントを使用して、実装前に API 動作、パッケージ使用法、バージョン固有の詳細を確認します。
   - **最初の2つが不十分な場合のみ Exa を使用:** GitHub 検索と主要ドキュメント後、より広いウェブリサーチまたは発見に Exa を使用します。
   - **パッケージレジストリを確認:** npm、PyPI、crates.io、その他のレジストリを検索してから、ユーティリティコードを書きます。手作りのソリューションより戦闘テスト済みライブラリを優先します。
   - **適応可能な実装を検索:** 問題の80%以上を解決でき、フォーク、ポート、またはラップできるオープンソースプロジェクトを探します。
   - 要件を満たす場合、ネットワーク新規コードを書くより、証明済みのアプローチの採用またはポートを優先します。

1. **最初に計画を立てる**
   - **planner** エージェントを使用して実装計画を作成
   - コーディング前に計画ドキュメントを生成：PRD、アーキテクチャ、システム設計、テックドキュ、タスク一覧
   - 依存関係とリスクを特定
   - フェーズに分割

2. **TDD アプローチ**
   - **tdd-guide** エージェントを使用
   - テストを最初に書く（RED）
   - テストに合格する実装（GREEN）
   - リファクタリング（IMPROVE）
   - カバレッジ80%以上を確認

3. **コードレビュー**
   - コード作成直後に **code-reviewer** エージェントを使用
   - CRITICAL と HIGH 問題に対応
   - 可能な場合 MEDIUM 問題を修正

4. **コミット＆プッシュ**
   - 詳細なコミットメッセージ
   - 従来のコミット形式に従う
   - コミットメッセージの形式と PR プロセスについては [git-workflow.md](./git-workflow.md) を参照

5. **レビュー前のチェック**
   - すべての自動チェック（CI/CD）が成功していることを確認
   - マージコンフリクトを解決
   - ブランチが対象ブランチより最新であることを確認
   - これらのチェックが成功した後のみレビューをリクエスト
