# コーディングスタイル

## イミュータビリティ（不変性）（重大）

新しいオブジェクトを作成し、既存のオブジェクトを変更しない：

```
// 疑似コード
悪い例:  modify(original, field, value) → 元のオブジェクトをインプレースで変更
正しい例: update(original, field, value) → 変更を加えた新しいコピーを返す
```

理由：イミュータブルデータは隠れた副作用を防ぎ、デバッグを容易にし、安全な並行処理を可能にします。

## コア原則

### KISS（シンプルに保つ）

- 実際に機能する最もシンプルなソリューションを優先する
- 早すぎる最適化を避ける
- 賢さより明確さを最適化

### DRY（繰り返さない）

- 繰り返されるロジックを共有関数またはユーティリティに抽出
- コピー&ペースト実装ドリフトを避ける
- 繰り返しが実際である場合のみ抽象化を導入（推測ではなく）

### YAGNI（必要ないことはやらない）

- 必要になるまで機能または抽象化を構築しない
- 推測的な一般性を避ける
- シンプルに始め、圧力が実在したときにリファクタリング

## ファイル構成

多くの小さいファイル > 少数の大きいファイル:
- 高い凝集度、低い結合度
- 典型的には200-400行、最大800行
- 大規模モジュールからユーティリティを抽出
- タイプではなく、機能/ドメインで整理

## エラーハンドリング

常にエラーを包括的に処理する：
- すべてのレベルでエラーを明示的に処理
- UI向けコードでユーザーフレンドリーなエラーメッセージを提供
- サーバー側で詳細なエラーコンテキストをログ
- エラーを黙って飲み込まない

## 入力検証

常にシステム境界で検証する：
- 処理前にすべてのユーザー入力を検証
- 利用可能な場合はスキーマベースの検証を使用
- 明確なエラーメッセージでファストフェイル
- 外部データ（APIレスポンス、ユーザー入力、ファイルコンテンツ）を信頼しない

## 命名規則

- 変数と関数: `camelCase` で説明的な名前
- ブール値: `is`, `has`, `should`, または `can` プレフィックスを優先
- インターフェース、タイプ、コンポーネント: `PascalCase`
- 定数: `UPPER_SNAKE_CASE`
- カスタムフック: `use` プレフィックス付き `camelCase`

## 避けるべきコード臭

### 深いネスト

ロジックが積み重なり始めたら、ネストした条件文ではなく、早期リターンを優先。

### マジックナンバー

意味のある閾値、遅延、制限に名前付き定数を使用。

### 長い関数

大きな関数を責務が明確な小さなピースに分割。

## コード品質チェックリスト

作業完了前に確認:
- [ ] コードが読みやすく、良く命名されている
- [ ] 関数が小さい（50行未満）
- [ ] ファイルが焦点を絞っている（800行未満）
- [ ] 深いネスト（4段階超）がない
- [ ] 適切なエラーハンドリング
- [ ] ハードコード化された値がない（定数または設定を使用）
- [ ] ミューテーション（不変パターンを使用）がない
