## 概要

ECC の rules ディレクトリには **87 個のルールファイル** が、**common（共通）+ 14 言語固有ディレクトリ** の二層構造で格納されている。ルールは `.claude/rules/` にインストールされると、セッション開始時に自動的にシステムプロンプトに注入される。LLM が「忘れる」ことができないため、プロンプトの指示より信頼性が高い。

### 優先順位の仕組み

```
言語固有ルール > 共通ルール
（CSS の詳細度や .gitignore の優先順位と同じパターン）
```

言語固有ルールが共通ルールと競合する場合、**言語固有ルールが優先**される。各言語ファイルの冒頭には `> This file extends [common/xxx.md](../common/xxx.md)` という参照が含まれる。

## 設計上の特徴

1. **レイヤード設計**: 共通 → 言語固有の二層構造により、DRY（Don't Repeat Yourself）を実現。共通原則は一箇所で管理し、言語固有の差分のみを各ディレクトリに記述
2. **システム注入型の信頼性**: ルールはセッション開始時に自動注入されるため、プロンプトの指示と異なり「LLM が忘れる」ことができない。chief-of-staff エージェントの設計原則にも「Rules are system-injected」と明記
3. **ルール vs スキルの明確な役割分担**:
   - **ルール**: 「何をすべきか」の基準・チェックリスト（例: テストカバレッジ 80%）
   - **スキル**: 「どうやるか」の詳細な手順・リファレンス（例: pytest の使い方）
4. **インストール方式**: `install.sh` スクリプトによる自動インストールまたは手動コピー。**ディレクトリ単位でコピー**が必須（フラット化すると同名ファイルが上書きされる）
5. **開発ワークフローの統合**: `development-workflow.md` がワークフロー全体を定義し、各ステップで使用するエージェントを指定。Research → Plan(planner) → TDD(tdd-guide) → Review(code-reviewer) → Commit
